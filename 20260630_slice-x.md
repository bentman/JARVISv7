# 20260630_slice-x.md
## Slice X — Local LLM Tier Activation and Model Lifecycle

Status: planned. This document is an implementation contract for a coding agent. It is not completion evidence.

Baseline commit: `cb23d5a8f1eeba90d7fba091478e5ed4238ef7b7`.

---

## Purpose

Activate the existing Slice V model-selection mechanism with real, defined local LLM model tiers while preserving the current catalog -> selection -> serve-profile -> sidecar -> OpenAI-compatible runtime flow.

The slice must improve end-to-end repo function, not merely rename policies. A new clone should be able to bootstrap the selected local LLM path. An existing repo should be transformable from the legacy development model to the new defined model ladder. Legacy local model artifacts should be removable from normal operation without breaking supported default behavior.

---

## Baseline Alignment at `cb23d5a8f1eeba90d7fba091478e5ed4238ef7b7`

Observed baseline:

- `config/models/llm.yaml` already has `llm_selection.default_policy: auto`, Operator env keys `LLM_MODEL_POLICY` and `LLM_MODEL_ID`, and roles `portable`, `balanced`, `quality`, `vision_preview`, and `diagnostic`.
- All roles currently resolve to `assistant-small-q4`; `vision_preview` is explicitly a placeholder with no vision runtime.
- Existing hardware selectors include:
  - `windows_amd64_cpu`
  - `windows_arm64_cpu`
  - `windows_amd64_gpu_nvidia_cuda`
  - `windows_amd64_gpu_amd`
  - `windows_amd64_gpu_intel`
  - `windows_arm64_gpu_qualcomm_adreno_opencl`
  - `windows_arm64_npu_qualcomm_base`
  - `windows_arm64_npu_qualcomm_qnn`
- Existing serve profiles use llama.cpp sidecars under `runtimes/llama.cpp/...`.
- AMD64 CUDA has repo-owned runtime artifact acquisition and live sidecar precedent.
- AMD64 CPU and ARM64 CPU local llama.cpp are live-proven.
- ARM64 Adreno OpenCL is staged/end-user-buildable or degraded depending on host state.
- AMD64 AMD/Intel Vulkan-style profiles are declared degraded/pending pinned artifact.
- ARM64 Qualcomm NPU/QNN LLM profiles are declared degraded/pending viability and must not be treated as proven LLM acceleration.

Conclusion: Slice X should use the existing model-selection mechanism and add real model entries/role mappings. It must not introduce a parallel selector, downloader, runtime launcher, desktop-only model manager, or new setup workflow.

---

## Outcome Target

At closeout, the repository should satisfy these functional outcomes:

1. New clone setup works through the existing setup path.
   - `scripts/bootstrap.py` and `scripts/ensure_models.py` must be able to fetch or verify the selected LLM model and current-host runtime sidecar.
   - The default path must not require manual model path editing.

2. Existing repos can be transformed safely.
   - Existing `.env` files with blank or default model policy values should move to the new auto-selected model path.
   - Existing explicit `LLM_MODEL_ID=assistant-small-q4` overrides must fail visibly or remain compatible; they must not silently select a different model.
   - Operators must have a clear way to clear a legacy override and return to policy selection.

3. Legacy local model artifacts can be removed from normal operation.
   - No default policy role should require `assistant-small-q4` after Slice X.
   - Removing `models/llm/assistant-small-q4/` must not break the default Auto, Portable, Balanced, or Quality policy paths.
   - If `assistant-small-q4` remains in the catalog, it must be labeled legacy compatibility or replaced by a new diagnostic model.

4. Operator experience remains acceptable.
   - Operator should choose a policy, not a raw model path.
   - Readiness/status must show selected policy, role, concrete model id, serve profile, accelerator/degraded state, and selection reason.
   - Raw model id/path/binary overrides remain advanced escape hatches.

5. End-to-end local assistant function improves without destabilizing the repo.
   - The default local LLM must be more capable than the legacy development model where hardware allows.
   - Unsupported accelerators must degrade cleanly to a safe selected model/runtime.
   - No unsupported vision or NPU LLM capability may be claimed.

## Model Ladder

Use Qwen-family GGUF models for the first activation unless direct implementation evidence proves another family is safer. Gemma may remain a future alternate, especially for a later vision or long-context slice, but Slice X should avoid adding multimodal runtime scope.

Recommended model roles:

| Role | Candidate model id | Purpose | Expected artifact posture |
|---|---|---|---|
| `diagnostic` | `assistant-qwen3-0p6b-q8-diagnostic` or `assistant-qwen3-1p7b-q8-diagnostic` | Fast smoke tests and low-cost fallback | Small GGUF, repo-fetchable |
| `portable` | `assistant-qwen3-4b-q4-portable` | Default cross-host local assistant model | GGUF Q4-class, repo-fetchable |
| `balanced` | `assistant-qwen3-8b-q4-balanced` or `assistant-qwen3-8b-q5-balanced` | Better local reasoning on proven acceleration | GGUF, selected only where appropriate |
| `quality` | `assistant-qwen3-14b-q4-quality` | Explicit operator quality trial on capable AMD64 CUDA | GGUF, opt-in or gated |
| `vision_preview` | unchanged placeholder | No activation in this slice | Do not claim vision behavior |

Exact Hugging Face repo ids, filenames, expected sizes, quantization names, and hashes/provenance should be filled from current source-of-truth model sources during implementation. Do not invent artifact metadata if it cannot be verified.

## Hardware Profile Selection Targets

These are best-effort target mappings based on the existing hardware profiles in `config/models/llm.yaml`. They are selection estimates until proven by validation.

| Hardware selector | Current baseline state | Auto target | Notes |
|---|---|---|---|
| `windows_amd64_cpu` | CPU sidecar path exists; CPU local llama.cpp has prior live proof | `portable` | Use Qwen3 4B-class only if CPU latency is acceptable; otherwise default to `diagnostic` and make `portable` operator-selectable. |
| `windows_arm64_cpu` | CPU sidecar path exists; CPU local llama.cpp has prior live proof | `portable` or `diagnostic` if 4B latency is poor | Must remain safe on Snapdragon without requiring Adreno build. |
| `windows_amd64_gpu_nvidia_cuda` | CUDA sidecar acquisition/live precedent exists | `balanced` | Primary high-ROI activation target. RTX 3060-class host should validate `balanced`; `quality` remains explicit/gated until live proof. |
| `windows_amd64_gpu_amd` | Declared degraded, pending pinned Vulkan/HIP artifact | `portable` | Do not auto-select a larger accelerated model until runtime artifact is pinned and validated. |
| `windows_amd64_gpu_intel` | Declared degraded, pending pinned Vulkan/OpenVINO/SYCL artifact | `portable` | Do not auto-select a larger accelerated model until runtime artifact is pinned and validated. |
| `windows_arm64_gpu_qualcomm_adreno_opencl` | Build-required/degraded; prior staged/end-user-buildable evidence exists | `portable` | Prefer portable model; do not require OpenCL sidecar for new clone success. If OpenCL is present, readiness may report degraded/accelerated state truthfully. |
| `windows_arm64_npu_qualcomm_base` | Pending viability, no proven LLM sidecar | `portable` via CPU/available sidecar | Do not claim Hexagon/NPU LLM support. |
| `windows_arm64_npu_qualcomm_qnn` | Pending viability, no proven LLM sidecar | `portable` via CPU/available sidecar | Do not claim QNN LLM support. QNN STT evidence does not prove QNN LLM sidecar viability. |

Policy behavior:

```text
Auto:
- AMD64 CUDA -> balanced
- AMD64 CPU -> portable unless latency validation forces diagnostic
- ARM64 CPU -> portable unless latency validation forces diagnostic
- AMD64 AMD/Intel degraded GPU -> portable
- ARM64 Adreno OpenCL -> portable
- ARM64 Qualcomm NPU/QNN -> portable via non-NPU LLM runtime

Portable:
- all hardware -> portable

Balanced:
- AMD64 CUDA -> balanced
- all other hardware -> portable or fail closed with clear unsupported-policy reason

Quality:
- AMD64 CUDA only after proof -> quality
- all other hardware -> fail closed or degrade explicitly according to existing policy style

Diagnostic:
- all hardware -> diagnostic

Vision Preview:
- remains placeholder/no-vision-runtime unless a separate multimodal runtime slice is approved
```
