# 20260615_slice-q.md
## Slice Q — General Housekeeping Items

**Date:** 2026-06-15
**Status:** Proposed
**Depends on:** Group P correction recorded in `slices.md`; current runtime surfaces inspected against `repo_tree.md`.

---

## Purpose

Recent architecture slices left the product path broadly coherent, but the repository now carries compatibility ingress paths, thin pass-through service layers, stale slice-era helpers, and duplicated status ownership. Slice Q is housekeeping, not redesign.

The goal is to reduce duplicate ownership and stale compatibility surfaces before adding streaming voice, enabled agents, semantic memory, local LLM activation, or new model/runtime behavior. Each housekeeping item is intentionally isolated as its own sub-slice so cleanup remains reviewable and adjacent behavior can be validated after every change.

---

## Repo-Tree Alignment

`repo_tree.md` is the cleanup standard for Slice Q:

- `routing/runtime_selector.py` is the single escalation-policy owner for LLM and search.
- `services/` is the API-facing layer; it orchestrates runtimes, cognition, and memory through public interfaces and should not accumulate domain logic.
- Routes and UI are thin adapters.
- Cognition and runtime-consuming paths go through `routing/runtime_selector.py`.

Slice Q does not treat `repo_tree.md` as proof that the implementation already conforms. It uses the file as the target boundary while inspecting actual code before each sub-slice.

---

## Current Repository Facts

- Voice ingress has two backend paths: resident/realtime PTT through `/session/ptt` and `ResidentVoiceInvocationService`, and direct WAV upload through `backend/app/api/routes/voice.py` at `/task/voice`.
- The desktop Rust command surface still has a `submit_voice` path while the current desktop UI uses `invoke_resident_ptt` from `desktop/src/main.js`.
- `backend/app/services/task_service.py` is a thin wrapper over `backend/app/services/turn_service.py`, and `turn_service.py` is itself a thin wrapper over `TurnEngine`.
- `backend/app/api/app.py::build_startup_state()` selects `OllamaLLM()` directly while `backend/app/routing/runtime_selector.py` owns LLM selection.
- `backend/app/api/routes/readiness.py` hardcodes family runtime labels in `_RUNTIME_NAMES` while separately reporting `active_llm_runtime=state.llm.runtime_name()`.
- `TurnEngine.enter_stub_state()` remains in product code and current unit tests still reference it.
- `RealtimeConversationSession`, `SessionService`, `WakeMonitorService`, and desktop polling each own part of visible realtime/wake status.
- `desktop/src/main.js` owns startup, backend lifecycle calls, session polling, wake polling, personality switching, settings restart flow, resident voice rendering, conversation log rendering, PTT state, and error display.

## Execution Rules

- Each housekeeping item is a separate sub-slice.
- Each sub-slice must inspect actual references before deleting or relabeling a compatibility surface.
- Each sub-slice must search for redundant tests created only to preserve stale compatibility.
- Redundant tests are removed or adjusted only when the behavior they cover is intentionally removed, renamed, or moved.
- Adjacent stacks are re-validated after each change.
- `backend/.venv/Scripts/python scripts/validate_backend.py regression` and `git diff --check` are required closeout evidence for every implemented sub-slice.
- `CHANGE_LOG.md` is updated only after validation evidence exists.
- `SYSTEM_INVENTORY.md` changes only when observable capability truth changes; most Slice Q items should not need inventory updates.

## Execution Order

```
Q.1 -> Q.2 -> Q.3 -> Q.4 -> Q.5 -> Q.6 -> Q.7 -> Q.8 -> Q.9
```

Lowest-risk cleanup order:

1. Normalize or relabel legacy direct voice ingress.

3. Route startup LLM selection through `runtime_selector`.
4. Replace hardcoded readiness runtime labels with selected runtime metadata.
5. Remove or isolate stale turn-engine stub helpers.

7. Extract wake state only if the boundary is low-risk and behavior-preserving.
