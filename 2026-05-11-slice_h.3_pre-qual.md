I simplified the pre-qual around the repo’s actual path: **ARM64 repo provisioning + `onnxruntime-qnn` plugin discovery + Qualcomm precompiled Whisper-Base Snapdragon X Elite download**. This removes the x64 export path as primary work, removes `QAIRT_SDK_PATH` as a required setup item, and treats `transformers` as an unresolved probe dependency rather than silently mutating the repo venv. The repo QNN extra already declares `onnxruntime`, `onnxruntime-qnn`, and `onnx` for Windows ARM64; the current QNN provider uses `onnxruntime_qnn.get_library_path()` and `get_qnn_htp_path()`; Qualcomm publishes the Snapdragon X Elite `PRECOMPILED_QNN_ONNX` asset with QAIRT/ORT versions in `release_assets.json`. ## Purpose

H.3 failed previously because the artifact/runtime contract was not proven before runtime wiring began.

This pre-qualification proves the simplest viable path:

1. The ARM64 repo environment can load the QNN provider through the current `onnxruntime-qnn` plugin path.
2. Qualcomm’s precompiled Snapdragon X Elite Whisper-Base QNN ONNX artifact can be downloaded.
3. The encoder and decoder artifacts can be opened with `QNNExecutionProvider` on ARM64.
4. The encoder/decoder tensor contract can be captured before runtime implementation begins.
5. A mechanical one-step decode probe can be attempted using the repo’s known-audio fixture.

Final result is one of:

- `PROCEED`
- `DEFER - <named stop condition>`
