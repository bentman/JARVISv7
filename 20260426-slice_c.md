## Slice C — Canonical Turn Engine

**Date:** 2026-04-26
**Status:** Proposed
**Depends on:** Slice B closed and verified (SYSTEM_INVENTORY.md 2026-04-26 19:45)

---

## Purpose

Convert the proven runtime families from Slice B into one canonical turn lifecycle shared by voice and text. Group C establishes the state machine, the session/artifact boundary, the personality input contract, and the interruption path. It does not build the durable shell; it makes the turn engine correct and provable first.

Group C exists here because:
- STT, TTS, LLM, and Wake are callable through hardware-evidence-driven selectors.
- No turn engine exists yet; text and voice have no shared execution path.
- Artifact and session schema must be fixed before multi-turn and interruption work builds on top of them.
- The proving host (`scripts/run_jarvis.py`) must exist before D (desktop shell) to catch regressions during the proving period.

Runtime selectors never called directly by the turn engine — they are called once at startup and their outputs (`STTBase`, `TTSBase`, `LLMBase`, `WakeBase`) are injected as dependencies.

## Out of Scope

Slice C introduces no Group D/E/F/G/H/I implementation scope. Specific exclusions:

- No durable desktop shell (`desktop/`) — `scripts/run_jarvis.py` is the proving host only (C.6)
- No backend HTTP API routes or services beyond a minimal `turn_service.py` boundary used internally
- No Redis, SearXNG, cache layer (Group E)
- No tool registry, executor ACTING state, or tool invocations (Group F)
- No episodic memory retrieval across sessions (Group G); bounded working memory only
- No llama.cpp activation (H.1), QNN inference (H.2), DirectML uplift (H.3)
- No agent framework (Group I)
- No full personality implementation — `personality/adapter.py` is introduced as an explicit input boundary only; personality profiles are loaded and injected but not deeply tuned in this slice
- No production wake/listen loop — wake integration into a resident loop is Group D scope; C-group uses push-to-talk or direct audio input for proving

## Pre-Implementation Gates

**Gate C-G1 — Runtime selector outputs are injectable.**
Confirm `select_stt_runtime()`, `select_tts_runtime()`, `select_llm()`, and `wake_runtime.select_wake_runtime()` (or equivalent) return instances that can be injected into the turn engine at startup. Verify the call signature is stable before writing `conversation/engine.py`. No code change expected; this is a read-verify step.

**Gate C-G2 — `asyncio` boundary decision.**
The turn engine must choose an async model. `sounddevice` callbacks are synchronous-ish; `onnxruntime` inference is blocking. Decision: the turn engine runs synchronously by default in C-group (blocking STT → blocking LLM → blocking TTS synthesis → blocking playback). Async wrappers are deferred to D-group where the shell demands non-blocking I/O. Confirm this is acceptable before Agent begins C.1. If not, document the async boundary explicitly before starting.

**Gate C-G3 — Personality input shape.**
`prompt_assembler.py` must accept a personality input. The minimal shape required: a `PersonalityProfile` dataclass (or equivalent) with at least `display_name: str`, `tone: str`, `brevity: str`, and an optional `system_prompt_addendum: str`. Confirm no existing file in the repo defines a conflicting schema before Agent creates `backend/app/personality/schema.py`.

**Gate C-G4 — `data/turns/` and `data/sessions/` writable.**
Artifact persistence writes to `data/turns/<session_id>/<turn_id>.json` and `data/sessions/<session_id>/session.json`. Confirm `.gitignore` rules allow `.gitkeep` but exclude `*.json` writes at runtime. No code change; verify before C.3.

**Gate C-G5 — No new pip dependencies required.**
The turn engine requires only packages already in base: `pydantic`, `pyyaml`, `numpy`, `sounddevice`, `soundfile`, `structlog`. Confirm before beginning. If `asyncio` gate (C-G2) changes the model, reconsider at that point.

## Sub-Slice C.5 — Interruption / Barge-In

### Objective

User interrupts during SPEAKING; speech stops; session continues coherently. `SPEAKING → INTERRUPTED → RECOVERING → IDLE` transition is deterministic and recorded in the turn artifact.

### Design decisions

- `BargeInDetector` in `barge_in.py` (currently always returns `False`) is promoted to a real detector: it monitors a concurrent audio stream for voice energy above a threshold while `playback.play()` is running. Implementation may use a simple VAD (voice activity detection) energy threshold rather than a full wakeword pass — YAGNI.
- `playback.stop()` is the interruption action. It is already present in `backend/app/runtimes/tts/playback.py`.
- The turn engine monitors the barge-in detector on a polling interval during SPEAKING. When `detect()` returns `True`, the engine calls `playback.stop()`, records an interruption event in the `TurnArtifact`, and transitions to `INTERRUPTED → RECOVERING → IDLE`.
- Perfect acoustic barge-in (mid-word overlap) is not required for C.5 closeout. The first acceptable behavior is: VAD fires → speech stops → clean transition → artifact records event.
- Feedback loop risk (assistant speech triggering its own barge-in detector): suppress the detector during the first 0.5 s of playback and during any period where `is_playing()` has been `True` for less than a configurable guard time. Document this in `barge_in.py` comments.

### Files to Create / Modify

**`backend/app/runtimes/stt/barge_in.py`** — promote from stub:
- `BargeInDetector.__init__(energy_threshold: float = 0.02, guard_time_s: float = 0.5)`
- `detect(audio_chunk: np.ndarray) -> bool` — computes RMS energy; returns `True` if above threshold and guard time has elapsed since playback start.
- `reset()` — resets guard timer; called when playback starts.

**`backend/app/conversation/engine.py`** — extend `run_voice_turn`:
- During SPEAKING, poll `barge_in_detector.detect()` on each audio chunk from a concurrent capture stream.
- On `True`: call `playback.stop()`, append an interruption event `{type: "barge_in", timestamp: ..., recovery_state: "RECOVERING"}` to the turn context, transition `SPEAKING → INTERRUPTED → RECOVERING → IDLE`.
- `TurnResult` extended: `interrupted: bool`, `interruption_events: list[dict]`.

**`backend/tests/unit/conversation/test_engine.py`** — extend:
- `test_interruption_stops_playback_and_transitions_to_interrupted`

## Sub-Slice C.6 — Developer Usability Surface (Proving Host)

### Objective

Every failure mode in the proving host prints a clear reason. A new developer can clone the repo, provision, and understand exactly what happened on a turn. `scripts/run_jarvis.py` is introduced as an explicit diagnostic/developer tool — **not the durable application surface**.

### Files to Create

**`scripts/run_jarvis.py`** — proving host. Follows shared script CLI convention (AGENTS.md §7):
- Emits host-fingerprint as first stdout line.
- Accepts `--verbose`, `--dry-run`, `--trace-to <dir>`.
- Additional flags: `--turns N` (default 1), `--voice-only`, `--text-only`, `--profile` (print readiness and exit), `--policy-override <yaml_path>` (override policy for testing).
- On startup: runs profiler → preflight → readiness summary → prints which runtime family + device + model is selected per voice subsystem, and why (reason string from readiness deriver).
- Named explicit failure states: `STT_UNAVAILABLE`, `TTS_UNAVAILABLE`, `MODEL_MISSING`, `LLM_UNAVAILABLE`, `AUDIO_DEVICE_ERROR` — each printed with the evidence token or error that caused it.
- `--verbose`: streams `STATE → STATE` transitions to stdout for each turn.
- `--trace-to <dir>`: writes trace artifacts per turn to `<dir>/<timestamp>/`.
- `--dry-run`: prints the startup plan (which runtimes would be selected) and exits 0 without executing turns.
- Imports from `backend/app/**`; never duplicates application logic.
