# Slice U2 Implementation Guide: Active QNN Acceleration

## Purpose

Enable and prove active Windows ARM64 Qualcomm QNN acceleration after Slice U dependency/provisioning alignment. This guide is for Codex Agent running on Windows ARM64. 

## Current Evidence

Slice U validated the new split-package dependency and provisioning shape, not active QNN execution.

`CHANGE_LOG.md` entry `2026-06-27 06:54` records:

- ARM64 QNN provisioning uses paired `onnxruntime` and `onnxruntime-qnn`.
- `scripts/provision.py dry-run` produced:

```text
python.exe -m pip install -e .[hw-cpu-base,hw-arm64-base,hw-npu-qualcomm-qnn,dev]
```

- `scripts/provision.py verify` passed with both `onnxruntime` and `onnxruntime_qnn` present.
- `scripts/validate_backend.py profile` found packaged `QnnHtp.dll` under:

```text
backend\.venv\Lib\site-packages\onnxruntime_qnn\QnnHtp.dll
```

- The same profile evidence still reported:

```text
ep:QNNExecutionProvider:MISSING
```

Direct conclusion: the repo now installs and discovers the split package shape correctly, but the ARM64 host did not expose `QNNExecutionProvider` during profile validation. Active QNN acceleration is therefore not proven by Slice U.

## Direct Code Findings

### Safe Readiness Fallback Already Exists

`backend/app/hardware/readiness.py` selects STT `qnn` only when all three tokens are present:

```text
import:onnxruntime-qnn
ep:QNNExecutionProvider
dll:QnnHtp
```

If those tokens are not all present on a Qualcomm NPU profile, STT readiness returns CPU:

```text
("cpu", True, "selecting cpu")
```

This validates the architect feedback: missing `QNNExecutionProvider` is currently a safe fallback state, not an unsafe partial acceleration state.

### Preflight Discovers DLLs But Does Not Activate Provider

`backend/app/hardware/preflight.py` currently:

- imports `onnxruntime` for the base provider probe.
- imports `onnxruntime_qnn` inside `_discover_packaged_qnn_htp_path()`.
- discovers packaged `QnnHtp.dll` under either `onnxruntime` or `onnxruntime_qnn`.
- appends `ep:QNNExecutionProvider:MISSING` when `onnxruntime.get_available_providers()` did not already return `QNNExecutionProvider`.

Current gap: discovery of `onnxruntime_qnn` and `QnnHtp.dll` does not by itself make `onnxruntime.get_available_providers()` expose `QNNExecutionProvider`.

### Provider Helper Uses Provider Options Only

`backend/app/hardware/qnn_provider.py` currently:

- searches `onnxruntime` and `onnxruntime_qnn` package roots for `QnnHtp.dll`.
- falls back to `QAIRT_SDK_PATH`.
- creates sessions with:

```python
providers=["QNNExecutionProvider"]
provider_options=[{"backend_path": "...QnnHtp.dll"}]
```

It does not register a QNN provider library before provider probing or session creation.

### Runtime Selection Is Already Token-Gated

`backend/app/runtimes/stt/stt_runtime.py` returns `QnnWhisperRuntime` only when readiness selected device `qnn`.

`backend/app/runtimes/stt/onnx_whisper_runtime.py`:

- keeps `OnnxWhisperRuntime(device="qnn")` explicitly unsupported.
- uses `QnnWhisperRuntime` for QNN.
- initializes encoder/decoder with `create_qnn_session(..., disable_cpu_fallback=True)`.
- raises if either session reports a provider other than `QNNExecutionProvider` first.

This means active acceleration proof must make provider availability real; it should not weaken runtime checks.

### Live Tests Are Already Gated On QNN Provider Readiness

`backend/tests/runtime/hardware/test_qnn_gate_live.py` includes live ARM64 QNN checks for:

- `QnnHtp.dll` discovery through `get_qnn_provider_options()`
- `QNNExecutionProvider` presence in `onnxruntime.get_available_providers()`

`backend/tests/runtime/voice/test_stt_live.py` includes the live QNN STT transcript test:

```text
test_stt_qnn_live_transcript_correct
```

Both are skipped unless `SKIP_UNLESS_QNN` is false, which requires the profile token:

```text
ep:QNNExecutionProvider
```

## Scope For U2

U2 should be limited to active QNN provider activation and proof on Windows ARM64.

In scope:

- Determine why `QNNExecutionProvider` is missing after Slice U package alignment.
- Add the smallest repo-owned activation path needed for `onnxruntime.get_available_providers()` or an equivalent readiness probe to prove QNN provider availability.
- Keep `QnnHtp.dll` discovery through `onnxruntime_qnn` and `QAIRT_SDK_PATH`.
- Preserve CPU fallback when QNN provider activation is not proven.
- Prove QNN STT live execution through existing runtime tests.
- Rename the stale test:

```text
test_qnn_expected_requirement_specs_pin_ort_family
```

to:

```text
test_qnn_expected_requirement_specs_include_paired_ort_family
```

Out of scope:

- AMD64 selection, provisioning, CUDA, DirectML, or runtime behavior.
- Reverting to `onnxruntime-qnn==1.24.3`.
- Reintroducing ARM64 uninstall / force reinstall / `--no-deps`.
- Changing dependency policy from `>=` back to pins.
- TTS QNN acceleration. `derive_tts_device_readiness()` explicitly reports provider override missing for Kokoro/QNN.
- LLM QNN/Hexagon acceleration. `config/models/llm.yaml` keeps this separate from STT QNN.
- NumPy/sounddevice warnings.
- Desktop or resident voice behavior unless QNN STT live turn validation exposes a direct integration failure.

## Implementation Direction

Prefer a small helper in `backend/app/hardware/qnn_provider.py` if provider activation requires library registration.

Any activation helper should be:

- ARM64/QNN-specific.
- idempotent in one process.
- deterministic.
- callable from preflight before provider probing, if needed.
- observable through tokens and probe errors.
- safe to fail closed with `ep:QNNExecutionProvider:MISSING`.

Do not weaken this runtime invariant:

```text
QnnWhisperRuntime must only proceed when QNNExecutionProvider is primary.
```

Do not mark QNN ready merely because `QnnHtp.dll` is present.
