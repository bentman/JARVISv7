The next anchoring layer should be **Realtime Conversation Session Boundary**. It should not replace the current turn engine, session service, voice service, tools, memory, personality, or runtime selectors. It should coordinate them as a live interaction system.

The reason is already in `ProjectVision.md`: JARVISv7 is defined as ŌĆ£a real-time conversation engine with text as a fallback surface,ŌĆØ not a chatbot with voice added later. The target loop includes natural speech, interruption, turn continuity, explicit personality, and visible degradation.  The same document already defines a session as the bounded conversational context for continuity, memory policy, and interruption recovery, and says a session is not a single request, utterance, or UI screen. 

## Architectural decision

Add a **Realtime Conversation Session Boundary** between channel input and the existing `TurnEngine`.

Current simplified path:

```text
wake/PTT ŌåÆ fixed audio capture ŌåÆ STT ŌåÆ TurnEngine ŌåÆ LLM/tools/memory/personality ŌåÆ TTS/playback ŌåÆ idle
```

Target path:

```text
channel event ŌåÆ realtime conversation session ŌåÆ turn-taking / interruption policy
ŌåÆ committed turn ŌåÆ existing TurnEngine
ŌåÆ response orchestration / playback events
ŌåÆ session ledger / desktop state
```

The current `TurnEngine` should remain the owner of committed turn reasoning: STT transcription, memory retrieval, tool execution, prompt envelope, LLM generation, personality style guard, TTS synthesis/playback, and artifact recording. It already has those responsibilities wired through `run_text_turn()`, `run_voice_turn()`, `_run_reasoning_path()`, prompt envelope assembly, tool context, and artifact recording.   

The new layer should own **live conversational timing**: when a voice session starts, when audio is accepted, when a user turn is committed, when speech is interrupted, when playback is canceled, when state is visible, and how events are recorded.

## Why this matches production systems

Production voice agents separate model reasoning from conversation mechanics.

LiveKitŌĆÖs voice-agent model treats turn detection and interruption handling as first-class session behavior. Its docs describe VAD, STT endpointing, context-aware turn detection, realtime-model turn detection, manual turn control, endpointing delay, adaptive interruption handling, false-interruption handling, and session events as separate pieces of the agent session. ([LiveKit Docs][1])

TwilioŌĆÖs ConversationRelay exposes interruption and caller-input handling as explicit channel controls: `interruptible` decides whether caller input stops TTS playback, while `reportInputDuringAgentSpeech` decides whether the app receives speech/DTMF while the agent is speaking. ([Twilio][2]) ([Twilio][2])

Dialogflow CX models transactional conversations as state machines: a session has a current page, each turn either stays on the page or transitions, and pages collect information relevant to the current conversational state. ([Google Cloud][3]) Dialogflow fulfillment also has response queues and partial responses so the agent can say something quickly while a longer operation continues. ([Google Cloud][4])

That maps cleanly to JARVIS if the next layer is a **session/event coordinator**, not a bolted-on ŌĆ£agent.ŌĆØ

## Repo-state alignment

What already exists and should be consumed:

`SessionService` already tracks resident session status, active state, turn count, last transcript, last response, failure reason, invocation source, TTS output device, and wake status. It also controls voice state transitions such as `LISTENING`, `TRANSCRIBING`, `REASONING`, `RESPONDING`, `SPEAKING`, `FAILED`, and `IDLE`.  

`ResidentVoiceInvocationService` already coordinates wake/PTT enqueueing, audio capture, session state transitions, `TurnEngine.run_voice_turn()`, and failure normalization for no-speech wake invocations.  

`TurnEngine` already has preliminary interruption data surfaces: `interrupted`, `interruption_events`, `BargeInDetector`, interruption audio chunks, playback start/stop, `INTERRUPTED`, `RECOVERING`, and persisted interruption artifact data.  

`ProjectVision.md` already requires canonical states including `LISTENING`, `TRANSCRIBING`, `REASONING`, `ACTING`, `RESPONDING`, `SPEAKING`, `INTERRUPTED`, `RECOVERING`, and `FAILED`, and states that text and voice enter the same lifecycle. 

So the next work should integrate into those surfaces, not create parallel state machines.

## What to add

Add a new boundary under the conversation layer, for example:

```text
backend/app/conversation/realtime/
  session.py
  events.py
  policy.py
  turn_taking.py
  interruption.py
  response_queue.py
  ledger.py
```

Recommended ownership:

`events.py`
Defines typed event records such as `SESSION_STARTED`, `AUDIO_CAPTURE_STARTED`, `USER_SPEECH_STARTED`, `USER_TURN_COMMITTED`, `TRANSCRIPT_FINALIZED`, `ASSISTANT_RESPONSE_STARTED`, `ASSISTANT_SPEECH_STARTED`, `USER_INTERRUPTED`, `ASSISTANT_SPEECH_STOPPED`, `TURN_RECOVERING`, `SESSION_IDLE`, `SESSION_FAILED`.

`session.py`
Owns the live session coordinator. It receives channel events from desktop voice/wake/PTT and decides when to call the existing `TurnEngine`.

`turn_taking.py`
Owns turn boundaries. Initial implementation can support manual/PTT and committed audio chunks only. It should define the later seams for VAD, STT endpointing, and semantic endpointing without requiring them immediately.

`interruption.py`
Owns interruption decisions and event classification. Initial implementation can wrap existing `BargeInDetector`/playback stop behavior and record whether the interruption was a hard cancel, correction, or unknown.

`response_queue.py`
Owns progress phrases and future partial response sequencing. Initial implementation can be deterministic and non-streaming: queue ŌĆ£checking thatŌĆØ style messages only when explicitly invoked by tool/workflow code later.

`ledger.py`
Owns session-level event records. This should not replace `TurnArtifact`; it should complement it. `TurnArtifact` records committed turn results. The realtime ledger records the live event path around those turns.

## What not to add yet

Do not add autonomous agents.
Do not add semantic memory.
Do not add cloud LLM escalation.
Do not add telephony/SIP/WebRTC integration.
Do not add full streaming LLM/TTS.
Do not replace `TurnEngine`.
Do not move prompt/personality/tool/memory authority into the realtime layer.

Those are valid product goals, but the next boundary should make them attachable later without claiming them now.

## Required integration points

The slice should change `ResidentVoiceInvocationService` from ŌĆ£the voice coordinatorŌĆØ into a channel adapter that submits events to the realtime session boundary.

Current behavior:

```text
enqueue ŌåÆ capture 3 seconds ŌåÆ mark states ŌåÆ run_voice_turn ŌåÆ complete
```

New behavior:

```text
enqueue ŌåÆ start or reuse realtime session
ŌåÆ emit invocation event
ŌåÆ capture or receive audio
ŌåÆ commit user turn
ŌåÆ call TurnEngine
ŌåÆ emit response/playback/interruption events
ŌåÆ update SessionService status
```

`SessionService` should remain the status source consumed by API/desktop. It can receive summarized state updates from the realtime session boundary. It should not become the full event ledger.

`TurnEngine` should receive only committed turns. It should not be responsible for deciding whether a partial utterance is complete, whether a backchannel is an interruption, or whether a session remains open for follow-up.

`ConversationState` should remain the canonical coarse state vocabulary. The realtime boundary can add fine-grained events without expanding the top-level state enum prematurely.

## Slice framing

Suggested slice title:

`2026-06-13-realtime-conversation-session-boundary`

Purpose:

Establish the runtime boundary that turns the existing voice/text turn foundation into a live conversational session model, without replacing the proven turn engine, runtime selectors, tools, memory, personality policy, or desktop status surfaces.

Goal:

Create a typed realtime session coordinator and event ledger that can represent invocation, listening, committed user turn, assistant response, speech playback, interruption, recovery, failure, and idle transitions around existing `TurnEngine` execution.

Out of scope:

Streaming STT, streaming LLM, streaming TTS, semantic endpointing, full workflow/page engine, telephony, cloud escalation, autonomous agents, and semantic memory.

Those capabilities should become attachable through the new boundary, but they should not be claimed by this slice.

## Minimum implementation

The smallest useful implementation is:

1. Add typed realtime conversation events.
2. Add a `RealtimeConversationSession` coordinator.
3. Route resident voice invocation through the coordinator.
4. Preserve current PTT/wake behavior.
5. Record a session-level event ledger alongside turn artifacts.
6. Surface the current coarse state through existing `SessionService`.
7. Add unit tests proving event order for normal voice turn, empty transcript failure, and playback interruption.
8. Add one runtime/live test proving existing voice turn behavior still works through the new boundary.

Minimum event order for a normal PTT turn:

```text
SESSION_ACTIVE
INVOCATION_RECEIVED
AUDIO_CAPTURE_STARTED
AUDIO_CAPTURE_COMPLETED
USER_TURN_COMMITTED
TRANSCRIBING
REASONING
RESPONDING
SPEAKING
TURN_COMPLETED
SESSION_IDLE
```

Minimum event order for interruption:

```text
ASSISTANT_SPEECH_STARTED
USER_INTERRUPTION_DETECTED
ASSISTANT_SPEECH_STOP_REQUESTED
TURN_RECOVERING
SESSION_IDLE
```

This does not yet need to capture a second follow-up utterance after interruption. It needs to create the boundary that can support that next.

## Acceptance surface

This slice is complete when:

The existing voice path still works through the new coordinator.

Text turns still use the same cognition path and are not forced through voice-only mechanics.

The session status still reports meaningful visible states.

A session event ledger exists and records live interaction events around each committed turn.

The existing `TurnArtifact` still records prompt, transcript, response, personality profile, tool calls, memory refs, and interruption events.

Interruption behavior remains at least as capable as current `BargeInDetector` playback stop behavior.

No model routing, prompt personality, memory retrieval, tool authority, readiness, provisioning, or desktop layout boundary is reopened.

## Why this is the correct anchor

This slice gives the product a durable ŌĆ£conversation runtimeŌĆØ spine.

It connects directly to the visionŌĆÖs root identity: voice-first, interruptible, session-based, deterministic, personality-aware, hardware-aware, visibly degraded, and agent-capable later. It also gives later work a clean home:

Streaming STT attaches to `turn_taking.py`.

Semantic endpointing attaches to `turn_taking.py`.

Streaming LLM/TTS attaches to `response_queue.py`.

Tool progress phrases attach to `response_queue.py`.

Workflow/page state attaches above committed turns, not inside prompts.

Telephony/WebRTC attaches as new channel adapters.

Agents attach above the turn engine and consume session events, tools, memory, and runtime state.

The next slice should therefore be framed as **the live conversation runtime foundation**, not as ŌĆ£voice improvementŌĆØ and not as ŌĆ£agent prep.ŌĆØ
