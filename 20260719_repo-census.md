# JARVISv7 Repository Census — 2026-07-19

## Summary

This census inspected repository structure and wiring, not just keywords. It traced production entry points, imports, API routes, startup assembly, desktop imports, configuration consumers, tests, and the governance ledgers. `docs/archives/` and `desktop/node_modules/` were excluded under `.agentignore`.

The working backend, desktop, voice, memory, and local-runtime paths are substantially connected. The main structural debt is concentrated in four areas:

- clearly disconnected residue: an empty legacy frontend root, a zero-byte runtime module, and redundant keep-files;
- implemented but unreachable substrates: tools, several agent-role operations, and a duplicate search selector;
- explicitly incomplete runtime/config surfaces: cloud LLM providers, Porcupine wake, secondary ONNX-ASR STT, and vision-preview selection;
- stale structural authority: `repo_tree.md` describes files and directories that do not exist and labels some implemented capabilities as future work.

No deletion is performed or implied by this census. “Trash candidate” means no production consumer, operator workflow, or unique repository authority was found.

## Important non-findings

- `backend/app/conversation/realtime/` is connected through `ResidentVoiceInvocationService`; it is not an isolated experimental subsystem.
- Abstract base methods raising `NotImplementedError`, null/degraded runtimes, and exception-suppression cleanup blocks are normal boundaries and were not counted as scaffolds by themselves.
- SearXNG, DDGS, and Tavily runtime implementations are real and tested. Their application-level usefulness is blocked by missing tool registration, not by the runtime adapters themselves.
- Keep-files under `data/`, `cache/`, `models/`, and `reports/` preserve governed mutable/artifact roots and were not classified as trash.
- `scripts/run_jarvis.py` is intentionally a documented proving host, while `scripts/run_backend.py` is used by the Tauri shell; neither is dangling.

## Suggested triage order

1. Remove or justify C01–C05 after confirming no external, untracked operator dependency.
2. Decide whether tools are a product path now; either wire C12/C13 through one registry/selection authority or remove the unreachable surface until needed.
3. Keep incomplete providers impossible to select as “available” until their execution methods work (C16–C19).
4. Collapse duplicate agent/config helpers around the currently truthful status/ledger boundary (C06–C15).
5. Reconcile the structural map and stale capability metadata (C22–C23) against observed repository state and validation evidence.

## Evidence basis

This was a read-only structural inspection before this document was created. No runtime validation suite was rerun, so recorded capability claims come from `SYSTEM_INVENTORY.md` and `CHANGE_LOG.md`; connectivity findings come from current source/config/import/route tracing. The worktree was clean on `main` before creating this census.
