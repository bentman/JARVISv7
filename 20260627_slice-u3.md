# Slice U3 Implementation Guide: QNN STT Artifact Compatibility

## Purpose

Complete the ARM64 QNN acceleration slice correctly by preserving the Slice U split-package dependency configuration and focusing on the remaining QNN STT model artifact compatibility failure.

This guide is for Codex Agent running on Windows ARM64.

## Determination

Do not revert dependency or provisioning alignment.

Keep:

```text
onnxruntime>=1.24.4
onnxruntime-qnn>=2.3.0
onnx>=1.16
transformers>=4.40
```

Do not add:

```text
onnxruntime-qnn==1.24.3
onnxruntime==1.24.3
onnxruntime<1.25
--no-deps
force-reinstall QNN
uninstall onnxruntime on ARM64 QNN hosts
CPU fallback for QNN proof
committed binary model edits
```

The current failure is not dependency configuration. It is the current precompiled Whisper QNN EPContext artifact not being accepted fully by `QNNExecutionProvider` under the active package stack.

## Verified Inputs

### Architect U3

`20260627_architect-u3.md` is aligned with current repo evidence:

- Keep the Slice U split-package configuration.
- Keep the U2 QNN provider activation helper.

External docs support the artifact-focused direction:

- ONNX Runtime EPContext docs define the `EPContext` attributes, including `ep_cache_context`, `embed_mode`, `main_context`, `partition_name`, and `source`. They state QNN EP accepts EPContext nodes with `source = QNN` or `QnnExecutionProvider`, and that non-embedded context file paths are relative to the ONNX model file.
  - https://onnxruntime.ai/docs/execution-providers/EP-Context-Design.html
- `onnxruntime-qnn` v2.3.0 release notes state compatibility with ONNX Runtime `>=1.24.1`, compiled with `1.24.4`, QAIRT SDK `2.47.0`, and paired install of `onnxruntime==1.24.4` plus `onnxruntime-qnn==2.3.0`.
  - https://github.com/onnxruntime/onnxruntime-qnn/releases/tag/v2.3.0

## Decision Rules

### If EPContext Attributes Are Missing Required QNN-Compatible Metadata

Do not patch the binary or ONNX files by hand.

Acceptable next actions:

- Add a deterministic artifact verifier that marks this model as not QNN-ready with an explicit reason.
- Add a new model catalog entry for a current compatible QNN Whisper artifact if a source is identified.
- Add tests proving the old artifact is rejected for the documented reason.

### If Context Paths Are Incorrect

Only fix catalog/acquisition layout if the source artifact includes the required files and the repo extraction path is wrong.

Do not commit local binary edits.

### If Artifact Was Generated For An Older Incompatible QNN/QAIRT Stack

Mark the current model entry as incompatible with current QNN provider stack.

Add or propose a replacement acquisition path. If no replacement source is available, close U3 as blocked with:

```text
QNN provider activation: PASS
QNN hardware gate: PASS
QNN STT artifact compatibility: BLOCKED-no-compatible-artifact
```

### If A Compatible Artifact Source Is Found

Add it through existing model acquisition patterns in `config/models/stt.yaml` and `scripts/ensure_models.py`.

Keep the existing model source intact unless replacing it is explicitly validated and logged.

Validate:

```powershell
backend\.venv\Scripts\python scripts\ensure_models.py --family stt --verify-only
backend\.venv\Scripts\python -m pytest backend\tests\unit\runtimes\stt\test_stt_runtime.py -q
$env:JARVISV7_LIVE_TESTS = "1"

```
