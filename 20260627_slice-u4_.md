# ARM64 Agent Prompt: Slice U4 Qualcomm Blog Path

Goal:
Make ARM64 QNN STT work by using Qualcomm’s documented Snapdragon X Elite Whisper export path, then wire the generated model into JARVIS.

Research source:
https://www.qualcomm.com/developer/blog/2025/05/deploy-ai-models-on-snapdragon-x-elite-with-qualcomm-ai-hub

Key instruction:
Use Qualcomm’s documented Whisper path first:
- package extra: `qai_hub_models[whisper_base_en]`
- module: `qai_hub_models.models.whisper_base_en.export`
- device: `Snapdragon X Elite CRD`
- target runtime from blog: `onnx`

Do not:
- revert QNN runtime packages
- pin `onnxruntime-qnn==1.24.3`
- add `--no-deps`
- force reinstall QNN
- uninstall `onnxruntime`
- change AMD64 behavior
- commit if this fails

Keep runtime QNN stack:
- `onnxruntime>=1.24.4`
- `onnxruntime-qnn>=2.3.0`

Implementation path:
1. Add the Qualcomm AI Hub model export dependencies through the repo provisioning mechanism.
   Start with the blog package:
   - `qai-hub-models[whisper_base_en]`

   If pip resolver reports conflicts with current `onnxruntime>=1.24.4`, stop and report the exact resolver output. Do not downgrade runtime ORT silently.

2. Add a new STT model entry:
   - name: `whisper-base-en-qaihub-onnx-snapdragon-x`
   - runtime: `qnn_whisper`
   - devices: `[qnn]`
   - source type: `qai_hub_export`
   - module: `qai_hub_models.models.whisper_base_en.export`
   - target_runtime: `onnx`
   - device: `Snapdragon X Elite CRD`
   - required files: whatever export produces, but JARVIS runtime ultimately needs `encoder.onnx` and `decoder.onnx`
   - local_path: `models/stt/whisper-base-en-qaihub-onnx-snapdragon-x`

3. Extend `scripts/ensure_models.py` with source type `qai_hub_export`.
   Keep it simple:
   - call the export module as a subprocess or import the module if cleaner
   - output into the configured model local_path
   - be idempotent: if QNN verification passes, skip export

4. Add QNN artifact verification for `qnn_whisper`.
   File presence is not enough.
   Verification must:
   - locate encoder/decoder ONNX files
   - create QNN sessions with CPU fallback disabled
   - require `QNNExecutionProvider` primary

5. Only after verification passes:
   - update `select_stt_runtime()` to use `whisper-base-en-qaihub-onnx-snapdragon-x`
   - update `test_stt_qnn_live.py` to use the same model

Validation:
1. `backend\.venv\Scripts\python scripts\provision.py install`
2. `backend\.venv\Scripts\python scripts\provision.py verify`
3. `backend\.venv\Scripts\python scripts\ensure_models.py --family stt --model whisper-base-en-qaihub-onnx-snapdragon-x`
4. `backend\.venv\Scripts\python scripts\ensure_models.py --family stt --model whisper-base-en-qaihub-onnx-snapdragon-x --verify-only`
5. `backend\.venv\Scripts\python scripts\validate_backend.py profile`
6. `$env:JARVISV7_LIVE_TESTS="1"; backend\.venv\Scripts\python -m pytest backend\tests\runtime\hardware\test_qnn_gate_live.py -q`
7. `$env:JARVISV7_LIVE_TESTS="1"; backend\.venv\Scripts\python -m pytest backend\tests\runtime\voice\test_stt_live.py -q -k qnn`
8. `backend\.venv\Scripts\python scripts\validate_backend.py unit`
9. `backend\.venv\Scripts\python scripts\validate_backend.py regression`

Success:
- Qualcomm export produces a usable model
- QNN sessions load with CPU fallback disabled
- `QNNExecutionProvider` is primary
- live QNN transcript test passes

If failure:
- stop
- do not commit
- report exact failing command and output
