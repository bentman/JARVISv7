# 20260703_slice-z3.md
## Slice Z.3 — Conversation Debug and Turn Artifact Contract

Status: planned. This document is an implementation contract. It is not completion evidence.

---

## Purpose

Slice Z.3 makes the existing Conversation-area debug surface useful for both text and voice turns.

The target question is:

```text
What is the current session state, what was the latest turn, where is its artifact, what runtime context was observed, and why did it fail or degrade?
```

This work converts the current `Voice debug details` surface into `Conversation debug details`, keeps it collapsed, and renders only compact diagnostic facts that are not already obvious from the visible Conversation thread.

---

## Current Codebase Evidence

The repo already has the required session, artifact, and desktop surfaces:

- `backend/app/api/routes/session.py`
  - Exposes `/session/create`, `/session/close`, `/session/status`, and `/session/ptt`.
  - `/session/status` currently calls `SessionService.status()` and maps it into `SessionStatusResponse`.
  - Current response fields are session id, active flag, state, turn count, last transcript, last response, failure reason, invocation source, TTS output device, and voice capture diagnostics.

- `backend/app/api/schemas/session.py`
  - `SessionStatusResponse` currently carries the same flat live fields.
  - It has no structured latest-turn summary, artifact path, runtime context, or TTS degraded detail.

- `backend/app/services/session_service.py`
  - `SessionStatus` currently stores `session_id`, `active`, `state`, `turn_count`, `last_transcript`, `last_response`, `failure_reason`, `invocation_source`, `tts_output_device`, and `voice_capture_diagnostics`.
  - `status()` computes `turn_count` from `len(self._session_manager.turn_artifacts)`.
  - Voice invocation methods update the cached live fields.
  - Text turns currently rely on the engine and active `SessionManager` for turn recording.

- `backend/app/services/resident_voice_invocation.py`
  - Resident voice can fail before `TurnEngine` runs.
  - Those pre-engine failures update `SessionService` live status and may not create a turn artifact.
  - Capture diagnostics are recorded into `SessionService` when segment diagnostics exist.

- `backend/app/conversation/engine.py`
  - `TurnEngine` owns `self.stt`, `self.llm`, and `self.tts`.
  - Text turns enter the reasoning path from `run_text_turn(...)`.
  - Voice turns enter listening/transcribing first, then the reasoning path.
  - `TurnResult` already carries turn id, session id, transcript, response text, final state, failure reason, TTS degraded flag/reason, TTS output device, interruption data, tool calls/results, and raw audio path.
  - `_record_artifact(...)` writes observed `TurnResult` data into `TurnArtifact` through the active `SessionManager`.

- `backend/app/artifacts/turn_artifact.py`
  - `TurnArtifact` currently contains turn id, session id, input modality, hardware profile id, active personality profile id, raw audio path, transcript, final prompt text, retrieved memory refs, tools invoked, trace metadata, response text, audio output path, interruption events, final state, failure reason, TTS degraded flag/reason, TTS output device, and phase timestamps.
  - This is the natural place for compact per-turn runtime context because it is the artifact used to debug a real turn.

- `backend/app/artifacts/storage.py`
  - Turn artifacts are written to `data/turns/<session_id>/<turn_id>.json`.
  - Session artifacts are written to `data/sessions/<session_id>/session.json` on session close.

- `backend/app/api/routes/readiness.py`
  - `_runtime_label(...)` and `_runtime_labels(...)` already produce compact runtime labels from active runtime objects.
  - The same label style can be factored or mirrored for display context without changing readiness policy.

- `backend/app/runtimes/stt/base.py`, `backend/app/runtimes/tts/base.py`, and `backend/app/runtimes/llm/base.py`
  - STT and TTS base runtimes carry `device` and `model_path`.
  - LLM base runtimes expose `runtime_name()`.

- `backend/app/api/routes/task.py` and `backend/app/api/schemas/task.py`
  - Text turn responses already include turn id, session id, transcript, response text, final state, failure reason, and tool calls.
  - `text_turn(...)` refreshes the active engine through `SessionService.engine()` and returns the `TurnResult` response shape.

- `desktop/src/index.html`
  - The Conversation panel contains a collapsed `<details class="voice-debug">` block titled `Voice debug details` with `<pre id="voice-detail">`.
  - The same Conversation area contains the separate collapsed `Degraded list detail` surface.

- `desktop/src/components/resident-voice.js`
  - `setVoiceDetail(...)` renders line-oriented text into `#voice-detail`.
  - It currently prints state, source, resident mode, transcript, response, failure reason, TTS output device, and turn count.
  - `renderResidentVoiceStatus(...)` calls that renderer from `/session/status` refreshes.

- `desktop/src/main.js`
  - `refreshSessionStatus()` polls `/session/status` and passes the result into the resident voice presenter.
  - Text turns append user and assistant messages, then refresh session status.
  - Resident voice completions append conversation messages through the resident voice presenter.

- `desktop/src/api-client.js` and `desktop/src-tauri/src/lib.rs`
  - `getSessionStatus` already uses the existing `get_session_status` Tauri command.
  - The bridge passes backend JSON through to the desktop.

- `desktop/tests/static.test.mjs`
  - Protects required desktop DOM surfaces, API-client calls, and Tauri command exposure.
  - Rendering changes in this slice require matching desktop static contract updates.

## Target Mechanism

Use `/session/status` as the live contract. Use the latest active `TurnArtifact` as the latest-turn source when an artifact exists. Use live `SessionService` fields for current state and pre-engine resident voice failures.

Recommended backend shape:

```text
SessionStatusResponse
  existing fields
  latest_turn
    turn_id
    session_id
    input_modality
    final_state
    failure_reason
    degraded_reason
    tts_output_device
    raw_audio_path
    artifact_path
    runtime_context
  voice_capture_diagnostics
```

`latest_turn` should be absent or null when no turn artifact exists. Pre-engine resident voice failures should remain visible through existing live fields: `state`, `failure_reason`, `invocation_source`, and `voice_capture_diagnostics`.

Recommended turn artifact addition:

```text
TurnArtifact
  runtime_context: dict[str, str]
```

The compact `runtime_context` strings should be built from runtime objects that already exist in `TurnEngine`:

```text
stt: <runtime-label>/<device>
llm: <runtime-name>
tts: <runtime-label>/<device>
```

Use the turn phase timestamps to decide which runtime families are meaningful for a turn:

- text turn that reaches reasoning: `llm`
- voice turn that reaches transcribing: `stt`
- voice turn that reaches reasoning: `llm`
- voice turn that reaches speaking or records TTS degradation/output: `tts`

Artifact paths should follow the existing storage convention:

```text
data/turns/<session_id>/<turn_id>.json
```

## Desktop Rendering Target

Rename the existing user-visible title:

```text
Voice debug details -> Conversation debug details
```

Keep the same Conversation-area placement and collapsed behavior. The DOM id/class may remain stable where that reduces churn, but the visible label and tests should reflect conversation-level status.

Render only useful lines. Empty fields are omitted. Transcript and response remain in the normal Conversation thread and are not repeated in the debug block.

Idle/no-turn example:

```text
state: IDLE
turn_count: 0
```

Successful text turn example:

```text
state: IDLE
turn: text 4f31c2 IDLE
runtime: llm=llama.cpp
artifact: data\turns\<session_id>\4f31c2.json
```

Successful voice turn example:

```text
state: IDLE
turn: wake b7a901 IDLE
runtime: stt=onnx-whisper/qnn, llm=llama.cpp, tts=kokoro-onnx/cpu
tts_output_device: default
artifact: data\turns\<session_id>\b7a901.json
```

TTS degraded voice turn example:

```text
state: IDLE
turn: voice a15d77 IDLE
runtime: stt=onnx-whisper/cuda, llm=llama.cpp, tts=kokoro-onnx/cpu
degraded: tts=TTS runtime is unavailable
artifact: data\turns\<session_id>\a15d77.json
```

Pre-engine resident voice failure example:

```text
state: FAILED
source: ptt
failure: voice capture did not produce speech
capture: source=ptt, stage=segment, speech_started=false
turn_count: 2
```

This display is intentionally compact. The debug block should help locate the status, failure/degraded reason, artifact, and runtime context without becoming a second conversation transcript.

## Data Gathering Plan

Backend:

1. Add a compact latest-turn summary schema in `backend/app/api/schemas/session.py`.
2. Extend `SessionService.status()` or the session route response builder to derive `latest_turn` from `self._session_manager.turn_artifacts[-1]` when present.
3. Preserve existing flat `SessionStatusResponse` fields for current consumers.
4. Add `runtime_context` to `TurnArtifact` serialization.
5. Populate `runtime_context` inside `TurnEngine._record_artifact(...)` using runtime objects and phase timestamps already available there.
6. Represent pre-engine resident voice failures with current live status fields and capture diagnostics, since those failures may not produce a turn artifact.

Desktop:

1. Rename the visible details summary to `Conversation debug details`.
2. Move the line renderer out of voice-only semantics. A small `conversation-debug` renderer is appropriate when it keeps `resident-voice.js` focused on resident voice controls/status.
3. Render only non-empty useful fields from `/session/status`.
4. Keep Resident Voice panel rendering focused on stream, mode, VAD, barge-in, follow-up, and continuous state.
5. Keep `Degraded list detail` as the separate readiness-degradation surface.
6. Update `desktop/tests/static.test.mjs` for the renamed label and session-status/debug rendering contract.

## Acceptance Criteria

Slice Z.3 is complete when:

- `/session/status` remains backward compatible and includes a structured latest-turn summary when a turn artifact exists.
- Engine-created text and voice turn artifacts include compact runtime context for the runtime families that were meaningfully reached.
- Pre-engine resident voice failures remain debuggable through live status fields and capture diagnostics.
- The existing Conversation-area details block is titled `Conversation debug details`.
- The expanded `Conversation debug details` window has increase height for readability (~1.5 of existing).
- The details block renders only compact useful debug lines: state, latest turn summary, failure/degraded reason, runtime context, TTS output device when relevant, capture diagnostics when relevant, artifact path when available, and turn count when helpful.
- Transcript and response are not duplicated in the debug block.
- Normal idle/ready desktop state remains visually quiet.
- Existing Conversation messages continue to render as they do today.
- `Degraded list detail` remains the readiness-degradation surface.
- Desktop static tests protect the renamed debug surface and existing session-status wiring.
- Focused backend tests cover no-turn, successful text, successful voice, TTS degraded, failed artifact turn, and pre-engine resident voice failure cases.

## Stop Conditions

Stop and report before editing further when:

- latest-turn status cannot be derived from live `SessionService` state or latest active turn artifacts;
- runtime context requires changing runtime selection or readiness policy;
- compact display requires changing normal Conversation message layout;
- desktop rendering requires a new Tauri bridge command despite existing JSON pass-through;
- tests show existing session status semantics conflict with this contract.
