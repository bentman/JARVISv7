# 20260703_slice-z3.md
## Slice Z.3 — Session/Status Contract Polish

Status: planned. This document is an implementation contract. It is not completion evidence.

---

## Purpose

Slice Z.3 expands the live session/status contract so the desktop can show accurate current-turn and last-turn state without inferring meaning from scattered fields.

The target question is:

```text
What is the current session doing, what happened on the last turn, what runtime path was used, and why did it fail or degrade?
```

This slice should expose live session status and render compact per-turn detail in the existing Conversation thread. It should not create a new status system, rewrite turn artifacts broadly, infer operator guidance from incomplete data, or depend on closed session files as the live source of truth.

---

## Current Codebase Evidence

The repo already has the required foundation:

- `backend/app/api/routes/session.py`
  - Exposes `/session/create`, `/session/close`, `/session/status`, and `/session/ptt`.
  - `/session/status` currently maps `SessionService.status()` into `SessionStatusResponse`.
  - Current response includes session id, active flag, state, turn count, last transcript, last response, failure reason, invocation source, TTS output device, and voice capture diagnostics.

- `backend/app/api/schemas/session.py`
  - Defines `SessionStatusResponse` with the existing live fields.
  - It does not yet carry structured current status, last-turn result, runtime path, degraded reason, or artifact reference.

- `backend/app/services/session_service.py`
  - Owns the live in-memory `SessionStatus` dataclass.
  - Tracks current state, last transcript, last response, failure reason, invocation source, TTS output device, and voice capture diagnostics.
  - Updates state through voice invocation lifecycle methods.

- `backend/app/services/resident_voice_invocation.py`
  - Drives resident voice invocation and updates `SessionService` through begin/transient/complete/fail paths.
  - Handles PTT, wake, follow-up, barge-in, resident stream state, and capture diagnostics.

- `backend/app/conversation/session_manager.py`
  - Owns active session identity, live turn artifacts, timeline, working memory, continuity packet building, and closed-session artifact writes.
  - `data/sessions/<id>` artifacts are written when a session is closed and should not be treated as the live desktop status source.
  - `turn_artifacts` is the live in-memory list of recorded turns for the active session.

- `backend/app/artifacts/turn_artifact.py`
  - Current turn artifacts already include turn id, session id, modality, hardware profile id, active personality profile id, transcript, final prompt text, response text, interruption events, final state, failure reason, TTS degradation reason, TTS output device, and timestamps.
  - This slice should summarize existing latest-turn fields where possible. Deep turn artifact enrichment belongs to a later artifact-focused slice.

- `backend/app/api/routes/task.py` and `backend/app/api/schemas/task.py`
  - Text turn responses already include turn id, session id, transcript, response text, final state, failure reason, and tool calls.

- `desktop/src/main.js`
  - Renders the Conversation thread with `appendMessage(...)`.
  - Polls `/session/status` through `refreshSessionStatus()`.
  - Renders resident voice status through the existing resident voice presenter.
  - Text turns append user and assistant messages, then refresh session status.

- `desktop/tests/static.test.mjs`
  - Acts as the desktop UI/Tauri contract check for required DOM surfaces, API-client calls, and Tauri command exposure.
  - Desktop rendering changes should update this contract.

## Target Mechanism

Use the live session service and the latest active turn artifact as the source of truth.

Recommended backend shape:

```text
SessionStatusResponse
  existing fields
  current
    state
    label
    detail
  last_turn
    turn_id
    source
    final_state
    transcript
    response_text
    failure_reason
    degraded_reason
    artifact_path
  last_runtime_path
    stt
    llm
    tts
    wake
    tts_output_device
```

Names may be adjusted when implementation finds a simpler shape, but the contract should remain structured. Avoid adding a pile of unrelated flat fields.

Use existing data first:

- `SessionService` for live state, source, failure, TTS output, and diagnostics.
- `SessionManager.turn_artifacts[-1]` for latest turn summary when present.
- Existing readiness/runtime labels only when already available through API state or session-owned state; do not re-run readiness or runtime selection from the session route.

Do not use `data/sessions/<id>` as the live status source. Closed-session artifacts may remain unchanged for this slice.

Do not add operator guidance fields derived from inference rather than stored or observed data.

## Desktop Rendering Target

Keep the Conversation thread readable. Add a compact collapsed detail surface associated with the relevant turn or assistant message.

Example complete turn:

```text
USER
Hey Jarvis, what's the capital of the U.S.?

ASSISTANT
Washington, D.C. is the capital of the United States.

▸ Turn detail
```

Opened:

```text
▾ Turn detail
state: complete
source: wake
runtime path:
  stt: onnx-whisper / cuda
  llm: llama.cpp / gpu.cuda
  tts: kokoro-onnx / cpu
tts_output_device: default
failure_reason:
degraded_reason:
artifact: data\turns\<turn_id>.json
```

Example failed turn:

```text
ASSISTANT
No speech detected during PTT.

▾ Turn detail
state: failed
source: ptt
runtime path:
  stt: not reached
  llm: not reached
  tts: not reached
failure_reason: No speech detected during PTT
degraded_reason:
artifact: data\turns\<turn_id>.json
```

This detail should live in the Conversation thread. Do not move it into the left readiness pane or the right operator pane.

## Runtime Path Guidance

Runtime path should describe the selected path used or expected for the last turn. Keep it concise.

Acceptable values:

```text
stt: onnx-whisper / cuda
llm: llama.cpp / gpu.cuda
tts: kokoro-onnx / cpu
wake: openwakeword / cpu
```

When a stage was not reached, use an explicit value such as:

```text
stt: not reached
llm: not reached
tts: not reached
```

Do not invent runtime usage that is not available from current code or latest turn state. If exact runtime path is not observable without artifact enrichment, expose only the portion that is already available and leave deeper artifact work to the later turn-artifact slice.

## Behavior Contract

This slice is a live status contract and desktop presentation polish.

It must not:

- use `data/sessions/<id>` as the live session status source;
- rewrite session persistence;
- broaden turn artifact scope beyond a compact latest-turn summary;
- change runtime selection or readiness policy;
- change STT/TTS/LLM runtime behavior;
- change resident voice invocation behavior except to record already-known status details;
- add a new diagnostic route when `/session/status` can carry the data;
- add a new desktop panel outside the Conversation thread;
- add `next_step`, remediation, or operator-guidance fields derived from invented inference;
- make the UI noisy on idle/ready sessions.

It should:

- keep `/session/status` backward compatible for existing fields;
- prefer structured nested fields for new status/detail data;
- keep text and voice turn detail shape consistent where practical;
- use collapsed details so normal conversation reading remains clean;
- handle missing fields gracefully in the desktop.

## Acceptance Criteria

Slice Z.3 is complete when:

- `/session/status` returns the existing fields plus structured current status and latest-turn detail.
- The latest-turn detail includes final state, source where available, transcript/response where available, failure reason, degraded reason where available, and artifact reference where available.
- Runtime path is shown only where observable from current code or latest turn state; non-reached stages are explicit.
- No `next_step`, remediation, or operator-guidance field is added without a real stored/observed source.
- Desktop Conversation thread renders a collapsed `Turn detail` surface for completed or failed turns.
- Normal idle/ready desktop state remains clean and readable.
- Existing `Voice debug details` remains for live voice capture diagnostics and is not used as the main turn-history surface.
- Text-turn and voice-turn details are consistent enough that the operator can compare them.
- Existing `/session/status` consumers continue to work with old fields.
- Desktop static/Tauri contract tests protect the new Conversation turn-detail surface and existing session-status wiring.
- Focused backend tests prove the new session status contract for idle, completed, failed, and no-turn states.
