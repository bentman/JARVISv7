# 20260703_slice-z3.md
## Slice Z.3 — Session/Status Contract Polish

Status: planned. This document is an implementation contract. It is not completion evidence.

---

## Purpose

Slice Z.3 expands the live session/status contract enough for the existing desktop `Voice debug details` surface to show accurate current-session and latest-turn facts.

The target question is:

```text
What is the current session state, what happened on the latest turn, what observed runtime context is available, and why did the turn fail or degrade?
```

This slice should enhance the existing `Voice debug details` block under the Conversation area. It should not add per-message turn-detail expanders, a new status panel, a new diagnostic route, inferred remediation guidance, or a closed-session artifact dependency.

---

## Current Codebase Evidence

The repo already has the required live status and desktop surface:

- `backend/app/api/routes/session.py`
  - Exposes `/session/create`, `/session/close`, `/session/status`, and `/session/ptt`.
  - `/session/status` currently maps `SessionService.status()` into `SessionStatusResponse`.
  - Current response includes session id, active flag, state, turn count, last transcript, last response, failure reason, invocation source, TTS output device, and voice capture diagnostics.

- `backend/app/api/schemas/session.py`
  - Defines `SessionStatusResponse` with the existing live fields.
  - It does not yet carry structured latest-turn detail, TTS degraded detail, artifact reference, or runtime context.

- `backend/app/services/session_service.py`
  - Owns the live in-memory `SessionStatus` dataclass.
  - Tracks current state, last transcript, last response, failure reason, invocation source, TTS output device, and voice capture diagnostics.
  - Computes `turn_count` from `SessionManager.turn_artifacts`.
  - Updates state through voice invocation lifecycle methods.

- `backend/app/services/resident_voice_invocation.py`
  - Drives resident voice invocation and updates `SessionService` through begin/transient/complete/fail paths.
  - Handles PTT, wake, follow-up, barge-in, resident stream state, and capture diagnostics.

- `backend/app/conversation/session_manager.py`
  - Owns active session identity, live turn artifacts, timeline, working memory, continuity packet building, and closed-session artifact writes.
  - `data/sessions/<id>` artifacts are written when a session closes and are not the live desktop status source.
  - `turn_artifacts` is the live in-memory list of recorded turns for the active session.

- `backend/app/artifacts/turn_artifact.py`
  - Current turn artifacts already include turn id, session id, input modality, hardware profile id, active personality profile id, raw audio path, transcript, final prompt text, response text, interruption events, final state, failure reason, TTS degradation flag, TTS degradation reason, TTS output device, and phase timestamps.
  - This slice should summarize existing latest-turn fields where possible. Deep turn artifact enrichment belongs to a later artifact-focused slice.

- `backend/app/artifacts/storage.py`
  - Turn artifacts are written to `data/turns/<session_id>/<turn_id>.json`.
  - Session artifacts are written to `data/sessions/<session_id>/session.json` only on session close.

- `backend/app/conversation/engine.py`
  - `TurnResult` carries turn id, transcript, response text, final state, failure reason, TTS degradation detail, TTS output device, interruption data, tool calls/results, and raw audio path.
  - `_record_artifact(...)` writes those observed values into `TurnArtifact` through the active `SessionManager`.

- `backend/app/api/routes/readiness.py`
  - Runtime labels are already derived from the active `ApiState` using runtime objects and wake status.
  - This slice may reuse or factor that label logic for display context, but must not duplicate readiness policy or runtime selection.

- `backend/app/api/routes/task.py` and `backend/app/api/schemas/task.py`
  - Text turn responses already include turn id, session id, transcript, response text, final state, failure reason, and tool calls.

- `desktop/src/index.html`
  - The Conversation panel already contains a collapsed `<details class="voice-debug">` block titled `Voice debug details` with `<pre id="voice-detail">`.
  - The same area also contains the existing collapsed `Degraded list detail` block from Slice Z.2.

- `desktop/src/components/resident-voice.js`
  - `setVoiceDetail(...)` currently renders `state`, `source`, `resident_mode`, `transcript`, `response`, `failure_reason`, `tts_output_device`, and `turn_count` into `#voice-detail`.
  - `renderResidentVoiceStatus(...)` calls `setVoiceDetail(...)` from `/session/status` refreshes.

- `desktop/src/main.js`
  - Polls `/session/status` through `refreshSessionStatus()`.
  - Passes session status into `residentVoice.renderResidentVoiceStatus(status)`.
  - Text turns append user and assistant messages, then refresh session status.
  - Resident voice completions are appended through the resident voice presenter.

- `desktop/src/api-client.js`
  - `getSessionStatus` already calls the existing Tauri command `get_session_status`.

- `desktop/tests/static.test.mjs`
  - Acts as the desktop UI/Tauri contract check for required DOM surfaces, API-client calls, and Tauri command exposure.
  - Desktop rendering changes must update this contract.

## Target Mechanism

Use `/session/status` as the live status contract and `Voice debug details` as the desktop rendering surface.

Recommended backend shape:

```text
SessionStatusResponse
  existing fields
  latest_turn
    turn_id
    session_id
    input_modality
    final_state
    transcript
    response_text
    failure_reason
    tts_degraded
    tts_degraded_reason
    tts_output_device
    raw_audio_path
    artifact_path
    phase_timestamps
  runtime_context
    stt
    llm
    tts
    wake
```

Names may be adjusted when implementation finds a simpler shape, but the contract should remain structured. Avoid adding a pile of unrelated flat fields.

Use existing data first:

- `SessionService` for live state, source, failure, TTS output, capture diagnostics, and turn count.
- `SessionManager.turn_artifacts[-1]` for latest-turn summary when present.
- `data/turns/<session_id>/<turn_id>.json` path convention for artifact reference when a latest artifact exists.
- Active `ApiState` runtime objects for runtime context labels where available.

Do not use `data/sessions/<id>` as the live status source. Closed-session artifacts may remain unchanged for this slice.

Do not add operator guidance fields derived from inference rather than stored or observed data.

## Desktop Rendering Target

Enhance the existing `Voice debug details` output. Keep it collapsed by default and keep the Conversation thread messages clean.

Current style is line-oriented text in `#voice-detail`. Keep that simple format unless a small renderer improves readability without adding a new surface.

Example complete state:

```text
state: IDLE
source: wake
resident_mode: Wake
turn_count: 2
latest_turn:
  turn_id: <turn_id>
  modality: voice
  final_state: IDLE
  transcript: Hey Jarvis, what's the capital of the U.S.?
  response: Washington, D.C. is the capital of the United States.
  failure_reason:
  tts_degraded: false
  tts_degraded_reason:
  tts_output_device: default
  artifact: data\turns\<session_id>\<turn_id>.json
runtime_context:
  stt: onnx-whisper
  llm: llama.cpp
  tts: kokoro-onnx
  wake: openwakeword
voice_capture_diagnostics:
```

Example failed state:

```text
state: FAILED
source: ptt
resident_mode: Wake
turn_count: 3
latest_turn:
  turn_id: <turn_id>
  modality: voice
  final_state: FAILED
  transcript:
  response:
  failure_reason: No speech detected during PTT
  tts_degraded: false
  tts_degraded_reason:
  tts_output_device:
  artifact: data\turns\<session_id>\<turn_id>.json

  stt: onnx-whisper
  llm: not reached
  tts: not reached
  wake: openwakeword
voice_capture_diagnostics:
  source: ptt
  stage: segment
```

Display only data that exists in `/session/status`. Use empty values or omit lines for missing data. Do not invent next steps, remediation advice, or unobserved runtime usage.

## Runtime Context Guidance

Runtime context is display context, not proof of exact per-stage execution unless the turn artifact records that proof.

Acceptable values from current runtime/readiness state:

```text
stt: onnx-whisper
tts: kokoro-onnx
llm: llama.cpp
wake: openwakeword
```

Acceptable values for stages not reached in the latest turn:

```text
llm: not reached
tts: not reached
```

Do not claim accelerator/device usage, per-turn runtime usage, or fallback behavior unless that data is recorded or already exposed by an authoritative backend object. Preserve truthful labeling over appearance.

## Behavior Contract

This slice is live status contract and existing desktop diagnostic-surface polish.

It must not:

- use `data/sessions/<id>` as the live session status source;
- rewrite session persistence;
- broaden turn artifact scope beyond a compact latest-turn summary;
- add per-message Conversation turn-detail expanders;
- add a new desktop panel or diagnostic surface;
- change runtime selection or readiness policy;
- change STT/TTS/LLM runtime behavior;
- change resident voice invocation behavior except to record already-known status details;
- add a new diagnostic route when `/session/status` can carry the data;
- add `next_step`, remediation, or operator-guidance fields derived from invented inference;
- make the UI noisy on idle/ready sessions.

It should:

- keep `/session/status` backward compatible for existing fields;
- prefer structured nested fields for latest-turn and runtime-context data;
- keep text and voice latest-turn details consistent where practical;
- keep `Voice debug details` collapsed by default;
- preserve `Degraded list detail` as the separate readiness-degradation surface from Slice Z.2;
- handle missing fields gracefully in the desktop.

## Acceptance Criteria

Slice Z.3 is complete when:

- `/session/status` returns the existing fields plus structured latest-turn detail from live session state and latest active turn artifact data.
- Latest-turn detail includes turn id, session id, modality, final state, transcript/response where available, failure reason, TTS degraded flag/reason, TTS output device, raw audio path where available, artifact reference where available, and phase timestamps where available.
- Runtime context is shown only where observable from current backend objects or latest turn state; non-reached stages are explicit.
- No `next_step`, remediation, or operator-guidance field is added without a real stored/observed source.
- Desktop enhances the existing `Voice debug details` block rather than adding Conversation-thread turn detail, a new panel, or a new stacked diagnostic surface.
- Normal idle/ready desktop state remains clean and readable.
- Existing Conversation messages remain unchanged except for facts already appended today from voice/text completion.
- Existing `/session/status` consumers continue to work with old fields.
- Desktop static/Tauri contract tests protect the enhanced `Voice debug details` surface and existing session-status wiring.
- Focused backend tests prove the new session status contract for idle/no-turn, completed, failed, and TTS-degraded states.
