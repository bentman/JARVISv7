# Slice AA — Resident Voice Stabilization and Conversation Flow

Status: planned. This is a guide and request for implementation for a coding agent. It is not completion evidence and must not be treated as proof that any listed capability exists.

## Purpose

Make resident voice reliable before expanding wake or hands-free behavior.

Current user-observed quality is flakey/inconsistent. Treat this as a voice interaction system problem, not only an STT/TTS model problem. Stabilize the default interaction path first: PTT-only, explicit user control, observable phase timing, clear failure reasons, and no automatic wake startup. After that, harden endpointing, hands-free follow-up, barge-in, and wake toggle behavior.

The target outcome is a desktop voice loop where a normal PTT turn consistently captures the intended utterance, transcribes it, routes it through the existing Turn engine, speaks or degrades truthfully, and exposes enough diagnostics to distinguish capture/VAD, STT, LLM, TTS, playback, and turn-state failures.

## Agent Contract

- Preserve the existing architecture: backend owns voice, turn, readiness, and runtime truth; desktop only displays and invokes backend APIs.
- Do not create a parallel conversation engine, voice engine, or runtime family.
- Do not make Wake the startup/default path in this slice.
- Make `PTT-only` the startup/default resident voice mode (not `PTT+Wake`)
- Wake remains available only as an explicit operator toggle.
- Hands-free/continuous behavior must not be expanded until PTT-only is stable and measured.
- Do not claim model/runtime acceleration improvements unless live evidence proves them.
- Add focused tests with every behavior change.
- Use existing diagnostics/artifacts before adding new storage roots.
- Governance changelog entry only after each sub-slice with validation evidence and User/Operator verification.
- Governance inventory entry only after all sub-slices are complete, and User/Operator verification.
- Governance updates must not record future, skipped, degraded, or unvalidated behavior as complete.

- No agent execution.
- No semantic/vector memory.
- No new cloud voice provider.

## Target Observable Behavior

```text
Startup:
  Resident Voice mode: PTT-only
  Resident stream: started if available
  Wake: inactive, available toggle shown if backend reports readiness

PTT turn:
  User explicitly starts/commits a voice turn
  UI shows LISTENING -> TRANSCRIBING -> REASONING/ACTING -> RESPONDING -> SPEAKING/IDLE or FAILED
  Conversation log receives transcript and response once per completed turn
  Latest-turn debug shows raw_audio_path, capture diagnostics, STT runtime, LLM runtime, TTS runtime/output device, and phase timing

Wake:
  Wake does not auto-start
  Wake toggle can start/stop monitoring only after resident stream/readiness is valid
  PTT-only mode stops wake if wake is active

Hands-free:
  Not default
  Enabled only after PTT turn reliability and endpointing are validated
```

## Design Direction

Industry-aligned order for this repo:

```text
Manual PTT stability
  -> voice diagnostics and replayable evidence
  -> reliable endpointing/VAD
  -> truthful Turn phase/status hooks
  -> hands-free follow-up
  -> barge-in/false-interruption handling
  -> Wake as optional toggle
  -> streaming/pipelined STT/LLM/TTS only after the serial path is stable
```

Do not jump directly to wake/hands-free while PTT-only is unreliable.

## Execution Order

```text
AA.0 baseline capture
AA.1 safe default and Operator layout
AA.2 diagnostics and replay evidence
AA.3 PTT-only turn hardening
AA.4 Turn phase/status truth
AA.5 endpointing/VAD stabilization
AA.6 hands-free/barge-in/wake reintegration
AA.7 voice selector surface
AA.8 closeout evidence and governance
```

Each sub-slice must pass focused validation and receive User/Operator verification before moving to governance update for that sub-slice.

- User/Operator can inspect whether the issue is capture/VAD, STT, LLM, TTS synthesis, playback, or state orchestration.

### Scope

Extend existing diagnostics, not storage layout:

  - include voice phase timing and latest voice failure class.

  - record per-phase timing for STT, LLM generation, TTS synth, playback wait.
  - preserve existing raw audio persistence.

- `backend/app/artifacts/turn_artifact.py`
  - add fields only if existing runtime_context/latest-turn shape cannot carry timing cleanly.

- `backend/app/api/schemas/session.py`
  - expose timing/debug fields in latest-turn/session status.

- `desktop/src/components/conversation-debug.js`

- Tests under existing service/conversation/API locations.

### Diagnostics to Expose

```text

  source

  duration_s
  rms
  peak
  max_chunk_rms
  speech_chunks
```

## AA.4 — Turn Phase/Status Truth

### Goal

Review and correct the voice-specific Turn mechanism so status is not post-hoc or misleading.

### Desired Outcomes

- Session status reflects actual voice turn progress while the turn is running.
- TRANSCRIBING, REASONING, RESPONDING, and SPEAKING are live states, not only reconstructed after completion.
- Failures are tied to the phase where they occurred.
- Text turn behavior remains unchanged.

### Current Concern

`RealtimeConversationSession` calls `TurnEngine.run_voice_turn(...)` after capture, and `TurnEngine` performs STT, LLM, TTS, and playback internally. Some realtime/session events are recorded after the result returns. That makes the UI less useful during long or flakey STT/TTS/LLM phases.

### Scope

Extend existing TurnEngine/session boundaries with the smallest mechanism that reports real phase transitions.

Preferred shape:

```text
TurnEngine context.advance(...)
  -> optional phase observer/hook
  -> SessionService marks live transient state
  -> desktop polling displays current phase
```

Do not move TurnEngine logic into desktop, routes, or resident service.

Likely files:

- `backend/app/conversation/engine.py`
- `backend/app/conversation/turn_manager.py`
- `backend/app/conversation/realtime/session.py`
- `backend/app/services/session_service.py`
- `backend/tests/unit/conversation/`
- `backend/tests/unit/services/`
- `desktop/src/components/desktop-state.js`
- `desktop/src/components/resident-voice.js`

### Acceptance

```text
During a live or fake voice turn, session status can reflect actual TRANSCRIBING, REASONING, RESPONDING, and SPEAKING phases.
Phase timing is recorded.
Failures identify the phase where they occurred.
Existing text turn behavior is unchanged.
```

## AA.5 — Endpointing and VAD Stabilization

### Goal

Reduce cutoffs, missed speech, and silence/noise false positives.

### Desired Outcomes

- PTT capture includes the intended utterance more consistently.
- Brief pauses or hesitations do not prematurely end capture.
- Background noise does not trigger speech start.
- Diagnostics show why endpointing ended: silence, max-duration, no-speech, stream-ended, or too-short.

### Scope

Start with repo-local improvements before adding dependencies:

- Calibrate or adapt RMS threshold against observed stream noise floor.
- Add hysteresis/debounce/hangover behavior around speech start/end.
- Keep pre-roll.
- Preserve fixed max-duration and no-speech timeout fail-closed behavior.
- Add tests for:
  - quiet speech above calibrated threshold
  - background noise below threshold
  - brief hesitation inside utterance
  - trailing silence endpoint
  - too-short/empty speech
  - stream drop/timeout

If EnergyVAD is still insufficient after measured evidence, propose a separate dependency-backed sub-slice for model VAD such as Silero ONNX. Do not add that dependency silently in AA.5.

Likely files:

- `backend/app/runtimes/vad/energy_runtime.py`
- possibly new file under `backend/app/runtimes/vad/` if adaptive wrapper is justified
- `backend/app/services/utterance_segmenter.py`
- `backend/app/core/settings.py`
- `.env.example` only if new operator-safe thresholds are needed
- unit tests under existing VAD/service locations

### Acceptance

```text
Endpointing tests cover hesitation and noise cases.
PTT live/user validation improves or failure evidence clearly identifies STT/TTS as the remaining issue.
No wake/hands-free expansion occurs in this sub-slice.
```

## AA.6 — Hands-free, Barge-in, and Wake Reintegration

### Goal

### Desired Outcomes

- Barge-in does not trigger from ordinary noise or assistant playback artifacts as much as current behavior allows.
- Wake is reintegrated as a toggle-only feature using the stabilized endpointing path.

### Sequence

1. Validate hands-free follow-up after a successful PTT turn.
2. Validate continuous mode only as explicit opt-in.
3. Harden barge-in to distinguish real interruption from backchannel/noise.
4. Add false-interruption behavior:
   - if barge-in fires but STT returns empty/no-speech, resume or recover cleanly.

5. Keep Wake as toggle-only.
6. Validate wake-sourced command capture after resident stream and endpointing are stable.

Likely files:

- live runtime voice tests where hardware is available

### Acceptance

```text
Hands-free does not trigger unless explicitly selected.
Barge-in is unavailable in PTT-only and ptt+wake modes unless explicitly supported.
False interruptions do not strand the session in FAILED.
```

## Parent Slice Acceptance Criteria

Slice AA is accepted only when all applicable criteria below are satisfied:

1. Resident Voice starts in PTT-only mode by default.
2. Wake does not auto-start and remains an explicit toggle.
3. Resident Voice is positioned above Wake in the Operator panel.
4. The desktop provides a clear, simple PTT path that can complete repeated turns without Wake enabled.
5. No-speech, empty transcript, STT failure, LLM failure, TTS failure, playback failure, and turn-state failure are distinguishable through diagnostics or latest-turn status.
6. Voice turn phase/status reporting is truthful while the turn is running, not only after completion.
7. Endpointing/VAD behavior is tested against silence, noise, quiet speech, hesitation, trailing silence, max duration, and no-speech timeout cases.
8. Hands-free, continuous, barge-in, and Wake behavior are opt-in and validated only after PTT-only stability work.
9. The voice selector, if implemented in this slice, is backend/config-owned and truthfully reports whether changes are live-applied or restart-required.
10. Focused backend tests pass for changed resident voice, wake, turn, API, STT/TTS diagnostic, and VAD behavior.
11. Desktop static/UI tests pass for startup posture, Operator layout, Wake toggle behavior, Resident Voice status, and any voice selector surface.
12. Required validator commands pass or are explicitly skipped with acceptable reason.
13. Each sub-slice has a `CHANGE_LOG.md` entry only after validation evidence and User/Operator verification.
14. `SYSTEM_INVENTORY.md` is updated only after the parent Slice AA outcome is complete and User/Operator verification is complete.
15. No governance entry claims unvalidated accelerator behavior, hidden always-listening behavior, autonomous agent behavior, or production-ready hands-free/wake behavior beyond the evidence.

## Risks and Mitigations

| Risk                                           |   Severity | Mitigation                                                         |
| ---------------------------------------------- | ---------: | ------------------------------------------------------------------ |
| Treating Wake as primary before PTT is stable  |       High | Default PTT-only; wake toggle only                                 |
| Tuning VAD without evidence                    |       High | Add diagnostics/replay before endpoint changes                     |
| UI shows phases that backend cannot prove      |     Medium | Add backend phase hooks/status first                               |

## Finish Line

Slice AA is complete when PTT-only resident voice is the startup default, Wake is toggle-only, desktop status/layout matches that posture, voice turns expose enough diagnostics to classify flakiness, PTT turns repeatedly complete or fail clearly, and any hands-free/wake/voice-selector behavior included in the final scope is validated without overclaiming.
