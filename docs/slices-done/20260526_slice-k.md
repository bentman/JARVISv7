**Date:** 2026-05-26
**Status:** Proposed

---

## Purpose

Group J produced a shell that surfaces readiness data accurately, labels interaction states clearly, and communicates the wake/PTT activation paths. What the operator still cannot do from the UI is adjust anything. Every configurable setting requires editing files directly and restarting the backend manually.

Group K adds the interactive control surface. It is scoped tightly: settings visibility, read/write of an explicit allowlist of `.env` fields, restart-required UX, service health display, and lightweight appearance adjustments using the token set J.4 already established. Nothing in K re-engineers backend behavior, changes personality runtime logic, or introduces model management.

## Sub-Slice K.2a ‚Äî Settings Inventory + Backend Read/Write Route

### Why Here

Before any settings UI is built, the backend must define exactly which fields are operator-configurable and provide a safe read/write surface. The server-side allowlist is the authoritative guard ‚Äî the UI only reflects what the backend declares.

### Goal

`GET /config/operator` returns the current values of allowlisted `.env` fields. `POST /config/operator` writes only allowlisted fields to `.env` using a key-preserving strategy. Non-allowlisted keys are rejected with a named error.

### Scope

**Allowlist (from Gate K-1 enumeration of `.env.example` ‚Äî these are the likely fields; Gate K-1 finalizes):**

| Field | Type | Label |
|-------|------|-------|
| `USE_OLLAMA` | bool | Use Ollama |
| `OLLAMA_BASE_URL` | text | Ollama endpoint |
| `OLLAMA_MODEL` | text | Ollama model |
| `OLLAMA_NUM_CTX` | int | Context window |
| `USE_LOCAL_MODEL` | bool | Use local model |
| `USE_SEARXNG` | bool | Use SearXNG |
| `SEARXNG_BASE_URL` | text | SearXNG URL |
| `USE_DDGS` | bool | Use DDGS |
| `USE_TAVILY` | bool | Use Tavily |
| `TAVILY_API_KEY` | password | Tavily API key |
| `REDIS_HOST` | text | Redis host |
| `REDIS_PORT` | int | Redis port |

All of these are present in `.env.example` under their existing names. Gate K-1 produces the final list.

**`backend/app/api/routes/config.py`** ‚Äî new:

- `GET /config/operator` ‚Üí `{ fields: [{ key, value, type, label, restart_required }] }`. Reads current environment values. Password-type fields return `"***"` if set, `""` if unset.
- `POST /config/operator` ‚Üí accepts `{ fields: { key: value } }`. Key-preserving `.env` write: reads existing lines, updates matching keys in place, preserves all other lines. Returns `{ written: [key, ...], rejected: [{ key, reason }] }`. Never writes a key outside the allowlist. **Requires an existing `.env` file ‚Äî if the file is absent, the route returns a `409` error with `{ error: "env_file_missing" }` and writes nothing.** Does not create or delete `.env`.

**`backend/app/api/schemas/config.py`** ‚Äî new: Pydantic schemas for the above.

**`backend/tests/unit/api/test_routes.py`** ‚Äî extend:
- `GET /config/operator` returns expected shape with all allowlisted keys
- `POST /config/operator` with a valid key returns `written: [key]`
- `POST /config/operator` with a non-allowlisted key returns `rejected: [{ key, reason }]`
- `POST /config/operator` does not modify lines outside the allowlist in the `.env` file

## Sub-Slice K.4 ‚Äî Appearance Adjustments

### Why Here

J.4 established a token-driven style system. Appearance adjustments are safe because all visual values are CSS custom properties ‚Äî overriding a small set of tokens on `:root` changes density and accent without structural changes.

### Goal

The operator adjusts font size (two steps), panel density (two steps), and accent color (two presets) from a compact control in the sidebar. Changes apply immediately. Preferences persist between sessions. Semantic state tokens are never overridden.

### Scope

**Token overrides ‚Äî only these tokens are modified at runtime:**

- Font size step: overrides `--text-sm`, `--text-md`, `--text-lg` (larger step increases each by ~15%)
- Density step: overrides `--space-2`, `--space-3`, `--space-4` (compact reduces each by ~25%)
- Accent preset A (default): restores `--color-accent` to its token-defined value (`#00bfff`)
- Accent preset B (neutral): sets `--color-accent` to `#8b949e`

Preset color values live only in the `appearance-controls.js` preset map ‚Äî they are not written into `style.css`. CSS rule bodies continue to reference `var(--color-accent)` exclusively. Semantic tokens (`--color-ready`, `--color-degraded`, `--color-failed`, `--color-capture`, `--color-speaking`, `--color-thinking`, `--color-transcribing`) are never present in any preset definition.

Semantic tokens (`--color-ready`, `--color-degraded`, `--color-failed`, `--color-capture`, `--color-speaking`, `--color-thinking`, `--color-transcribing`) are never overridden.
