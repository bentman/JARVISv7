## Slice S â€” Repo-Owned Local LLM Runtime Artifact Acquisition

**Date:** 2026-06-18
**Status:** Proposed
**Depends on:** Slice R managed local llama.cpp runtime path, post-R cleanup, current provisioning shape in `pyproject.toml` and `backend/app/hardware/provisioning.py`.

---

## Purpose

Slice S corrects the missing artifact-acquisition boundary left after Slice R. Slice R made local llama.cpp runtime selection, sidecar launch, HTTP runtime calls, and CPU fallback behavior truthful, but the runtime binaries themselves are still not repo-acquired. Existing CPU binaries are consumed from `runtimes/llama.cpp/...`, but no committed code currently fetches those binaries. That is not acceptable for bootstrap, provisioning, or repeatable workstation setup.

This slice makes JARVIS own local LLM runtime binary acquisition through the existing model acquisition surface. Users must not manually download or place llama.cpp binaries. When local model acquisition is enabled for normal bootstrap flow with `USE_LOCAL_MODEL=true` and `LOCAL_MODEL_FETCH=true`, the repo should acquire the selected GGUF model and the runtime artifact required by the active hardware profile. Explicit operator commands such as `scripts/ensure_models.py --family llm --model ...` may still perform requested fetch/verify actions even when normal automatic fetch is disabled, but that behavior must be defined deliberately in this slice rather than emerging accidentally.

This slice also reorganizes `config/models/llm.yaml` so LLM runtime profiles align with the actual provisioning shape: CPU baseline, AMD64 base, ARM64 base, NVIDIA CUDA, AMD GPU, Intel GPU, and Qualcomm NPU/QNN. Each hardware profile must be represented in its own readable node so future operator editing can target one profile without modifying unrelated profiles.

This file is a new repo artifact. If the document is currently only local or conversational, the first implementation step is to add it as `20260618_slice-s.md`.

## Out of Scope

* No new LLM runtime family.
* No `llama-cpp-python`.
* No Python in-process inference.
* No selector override to force accelerator profiles.
* No desktop workaround to display CUDA/QNN without selected runtime evidence.
* No rewrite of `HardwareProfile`, provisioning extras, or preflight framework.
* No autonomous agent, memory, voice streaming, realtime interruption, or Group S voice-loop work.
* No Linux/macOS artifact acquisition unless a small metadata shape must leave room for future extension.
* No claim that AMD64 CUDA, AMD/Intel GPU, or ARM64 QNN acceleration works until the relevant live sidecar completion proof exists.
* No manual user download or manual binary placement as an accepted closeout path.

* **Gate S-G0 â€” Runtime artifact source census.** Confirm the exact existing repo surfaces that read, resolve, download, verify, and launch local LLM artifacts. This gate must inspect `config/models/llm.yaml`, `scripts/ensure_models.py`, `backend/app/models/catalog.py`, `backend/app/models/llm_profiles.py`, `backend/app/services/local_llm_sidecar.py`, `backend/app/services/local_llm_startup.py`, `backend/app/core/settings.py`, `scripts/provision.py`, and existing LLM tests before implementation begins.

* **Gate S-G1 â€” Binary source viability table.** Confirm current upstream artifact sources for every profile this slice will represent. The table in this document is planning input only. Asset names, archive composition, checksum strategy, dependency archive needs, and pinned release tag must be verified before implementation. Do not encode mutable `latest` release URLs in committed logic.

* **Gate S-G2 â€” Host evidence.** Before closing any live accelerator sub-slice, capture the current host fingerprint and preflight readiness. AMD64 CUDA validation requires a Windows AMD64 NVIDIA/CUDA host. ARM64 QNN validation requires a Windows ARM64 Qualcomm NPU host with QNN provider/DLL readiness.

* **Gate S-G3 â€” No manual artifact acceptance.** If a binary is already present locally, Agent must prove whether it was acquired by the new repo-owned downloader before using it as closeout evidence. Existing local files may be used as comparison evidence, not as proof of acquisition.

* **Gate S-G4 â€” Explicit fetch policy.** Define the relationship between `USE_LOCAL_MODEL`, `LOCAL_MODEL_FETCH`, automatic bootstrap acquisition, and explicit operator commands. Automatic paths must honor fetch-disable settings. Explicit commands must have deterministic behavior and must not become unusable unless the CLI clearly reports the disabled policy and provides an intentional override.
