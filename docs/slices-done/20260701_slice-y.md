## Slice Y — Local LLM Wiring and Response Contract Hardening

Status: planned. This document is an implementation contract for a coding agent. It is not completion evidence.

---

## Purpose

Slice Y hardens the existing local LLM wiring and validation path so live tests prove the same selected model, serve profile, sidecar command, prompt payload shape, and response contract used by the real backend. The immediate user-observed issue is flakey ARM64 local LLM behavior: repeated answers from one or two turns earlier, malformed/gibberish text, and tests that still pass because they only require non-empty responses.

This slice does not assume the ARM64 model, Adreno/OpenCL sidecar, Qwen3 GGUF, Ollama fallback, session continuity, or memory system is individually at fault. It first compares the existing working paths across Windows AMD64 CUDA, Windows AMD64 CPU fallback, Windows ARM64 CPU fallback, and Windows ARM64 Adreno/OpenCL. The result must be a smaller, evidence-backed correction to the existing llama.cpp/Ollama/turn-engine wiring, not a new runtime family or ARM-only workaround.

Every sub-slice, file touch, validation command, and closeout criterion must trace back to this purpose: prove and harden the actual local LLM path that produces user-visible responses.

## Target Observable Behavior

No direct new user-facing control is required. Behavior is observable through tests, readiness metadata, turn artifacts, sidecar command diagnostics, and live response checks.

Expected observable results:

- `LLM_MODEL_MODE=dev` live tests start and report the dev-selected model `assistant-small-q4` through the same selection/startup path as the backend.
- `LLM_MODEL_MODE=prod` live tests start and report the policy-selected Qwen3 model for the current host, including `assistant-qwen3-8b-q5-balanced` on AMD64 CUDA and `assistant-qwen3-4b-q4-portable` on current non-CUDA selectors unless the catalog policy is intentionally changed.
- AMD64 CUDA, AMD64 CPU fallback, ARM64 CPU fallback, and ARM64 Adreno/OpenCL use the same selection/profile/runtime contract; only hardware facts and serve-profile launch metadata differ.
- Live llama.cpp response tests fail if the runtime returns a non-empty but wrong answer to deterministic prompts.
- Multi-turn tests fail if the model answers a previous turn when the latest request is explicit and unrelated.
- Runtime diagnostics can distinguish: selected model, served model endpoint payload, serve profile, accelerator, sidecar command, prompt rendering mode, generation defaults, and response contract failure.
- Readiness and trace surfaces remain truthful about selected model/mode/policy/profile/accelerator and do not overclaim quality, coherence, or QNN/NPU acceleration.

Internal-only status sketch:

```text
Runtime: llama.cpp
Model mode: dev|prod
Selected model: <selection.model_id>
Serve profile: <resolution.serve_profile_id>
Accelerator: cpu|gpu.cuda|gpu.opencl.adreno|...
Prompt transport: chat-messages|flat-prompt-over-chat|completion
Response contract: deterministic sanity PASS|FAIL with reason
```

## Vocabulary / Contract Terms

- **Selection path**: `select_llm_model(...)` chooses the model id from route, mode, policy, override, and hardware facts.
- **Serve-profile path**: `resolve_llm_serve_profile(...)` chooses the local model path, llama.cpp binary, launch metadata, accelerator, and degraded candidates for the selected model.
- **Backend startup path**: `prepare_managed_local_llm(...)` composes model selection, serve-profile resolution, sidecar startup, readiness probing, and `LlamaCppLLM` construction.
- **Prompt transport**: the exact HTTP/API representation sent to the LLM runtime, such as OpenAI chat messages, raw flat prompt, or Ollama generate prompt.
- **Deterministic sanity prompt**: a short prompt with one acceptable normalized answer, used to detect prompt-template, stale-context, or response-contract failure.
- **Stale-answer failure**: a response that answers a prior user request rather than the latest unrelated explicit request.
- **Coherence failure**: a response with no parseable sentence or no relation to the deterministic expected answer, even if it is non-empty.
- **Quality validation**: deterministic contract checks only. This slice does not attempt subjective model benchmarking.

```text
A. Test fixture bypasses policy selection
   -> update fixture/helper to use existing selection/startup path

B. PromptEnvelope is rendered flat and then wrapped as a single chat user message

C. Session continuity / working memory is too salient for small local models
   -> reduce or label same-session context only within existing continuity policy/prompt assembly
   -> prove latest request still controls output

D. Sidecar readiness accepts stale/wrong process or wrong model
   -> harden readiness probe and diagnostics without changing sidecar architecture
```

## Slice-Level Closeout Criteria

1. Live llama.cpp tests use the same model-selection and serve-profile path as backend startup.
2. Prod live tests cannot silently exercise `assistant-small-q4` unless `LLM_MODEL_ID` explicitly requests it.
3. Deterministic live response tests fail on non-empty but wrong output.
4. A stale-answer two-turn guard exists and fails if the latest request is ignored.
5. Prompt transport is explicitly represented in tests and is shared across CPU, AMD64 CUDA, and ARM64 Adreno/OpenCL paths.
6. Session continuity and working memory remain bounded and subordinate to the latest request.
7. Readiness/trace diagnostics distinguish endpoint reachability from selected-model truth and response quality.
8. No new runtime family, prompt architecture, agent orchestration, or desktop manager was introduced.
9. Required validation commands pass or skip with explicit acceptable reason.
10. `CHANGE_LOG.md` is updated only after validation evidence exists.
11. `SYSTEM_INVENTORY.md` is updated only after both Windows AMD64 and Windows ARM64 evidence exists, unless explicitly approved otherwise.
