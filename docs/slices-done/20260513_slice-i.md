## Purpose

**Operational use note:** This document is an implementation guide for Slice I execution. It is not archival verification evidence. Only command outputs captured during Slice I execution constitute proof.

Group H produced evidence of what actually works per device per host class. Slice I consumes that evidence to normalize the selector, readiness deriver, and preflight reporting into coherent, documented acceleration paths across both host classes.

Without this normalization, the current state has a patchwork of per-sub-slice readiness decisions — some written before H.3 was proven, some accounting for H.5/H.6 deferral, none forming a single coherent documented fallback chain. I.1 and I.2 resolve this: one documented, deterministic priority order per host class, derived entirely from the H-group evidence already in `CHANGE_LOG.md`.

Slice I is also the gate for Group J. Before UX/readiness surfacing work begins (J), the data being surfaced must be correct and coherent. Normalization comes before presentation.

**UX caveat — Path A is the planned J/K surface.** After Slice I closes, Group J will expand the Tauri shell toward v6 UX quality and v5-style panel depth: per-subsystem readiness indicators, named interaction states, memory panel, search visibility, settings, and personality controls. The backend for all of this is already verified in Groups E, F, and G. Slice I is the prerequisite that ensures the readiness data those panels will display is accurate and stable.

## Sub-Slice I.1 — ARM Acceleration Sequence Normalization

### Why Here

H.3 proved QNN STT on ARM64. H.7 proved CPU TTS with `provider-override-missing` fail-close for accelerated TTS. The current readiness derivers reflect per-sub-slice additions. I.1 consolidates them into a single documented ARM64 priority chain.

### Goal

ARM64 path fully documented and normalized: `stt` selects QNN when proven, falls back to CPU explicitly. `tts` selects CPU. `config/hardware/notes.md` documents the ARM64 acceleration chain. Selector and readiness deriver reflect the H-group evidence with no ambiguity.

### Scope

**`backend/app/hardware/readiness.py` — normalize `derive_stt_device_readiness()` for ARM64:**

Current ARM64 logic evaluates QNN first (correct), then CUDA, then DirectML, then CPU. CUDA and DirectML are unreachable on ARM64 but are shared with the x64 path. I.1 does not remove them; it adds inline comments documenting which branches apply per host class.

If any inconsistency is found, correct it to match:
- `npu_available and npu_vendor == "qualcomm"` → evaluate `qnn_tokens_present` (all three: `import:onnxruntime-qnn`, `ep:QNNExecutionProvider`, `dll:QnnHtp`)
  - `qnn_tokens_present=True` AND artifact on disk → `("qnn", True, "qnn prerequisites proven; selecting qnn")`
  - `qnn_tokens_present=True` AND artifact absent → `("cpu", True, "qnn prerequisites proven but artifact absent; selecting cpu")`
  - `qnn_tokens_present=False` → `("cpu", True, "selecting cpu")`
- Artifact guard must use `get_model_path("stt", "whisper-base-en-qnn-snapdragon-x-elite")` — not a hardcoded path.

**`backend/app/hardware/readiness.py` — normalize `derive_tts_device_readiness()` for ARM64:**

No logic change expected. Add inline comments documenting the `provider-override-missing` rationale per H.7.
