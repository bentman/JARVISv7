## Slice A Goal

Establish the **host reality** layer for JARVISv7: one callable hardware profiler, one declarative provisioning authority, one evidence-backed readiness rail, one arch-aware test harness, a validated provisioning gate on both Windows host classes, and a first-class QNN slot defined but not wired.

Current repo truth entering this slice:

- the repository skeleton is in place (top-2-level directories with `.gitkeep`, governance docs, `pyproject.toml` with hardware extras declared, `.gitignore` aligned, `docker-compose.yml` scaffolded for Group E)
- `backend/requirements.txt` does not yet exist and will only ever exist as a generated lockfile (per AGENTS.md §1 and `pyproject.toml` comment)
- no `backend/app/hardware/` source exists yet; the profiler, provisioning resolver, and preflight are the first code this repository ships
- `scripts/` contains only a `.gitkeep`; `provision.py`, `validate_backend.py`, `bootstrap.py`, and `ensure_models.py` are all introduced by this slice
- `backend/tests/` contains only a `.gitkeep`; the `conftest.py` and marker harness are introduced by this slice
- the x64 CUDA rail must be implemented against v7's boundaries without importing ML packages into the profiler; `preflight.py` owns every DLL bootstrap

This slice closes **all six** Group A sub-slices defined in `slices.md`. Subsequent groups (B onward) depend entirely on Slice A being green on both Windows x64 and Windows ARM64 before they may begin.

**What Slice A is not:** a runtime (STT/TTS/LLM/Wake) slice, a model-acquisition slice, an application-surface slice, an inference slice, or an agent-framework slice. No voice-family runtime exists at the end of this slice. No model is loaded. No turn executes. What exists is an honest answer, on every supported host class, to the question: *what can this machine do, what packages does it need, and are those packages usable?*

## Canonical Identifiers

To eliminate naming ambiguity, Slice A uses these exact strings everywhere — profiler output, resolver output, evidence tokens, script output, test assertions, SYSTEM_INVENTORY entries:

**Architecture values** (normalized at the `cpu_detector` boundary):
- `"amd64"` — Windows x64 and Linux x86_64 (normalized from `AMD64` / `x86_64`)
- `"arm64"` — Windows ARM64 and Linux aarch64 (normalized from `ARM64` / `aarch64`)

**GPU vendor values:** `"nvidia"`, `"amd"`, `"intel"`, `"qualcomm"`, `"unknown"`, `None` (no GPU).

**NPU vendor values:** `"qualcomm"`, `"intel"`, `"amd"`, `"apple"`, `"unknown"`, `None` (no NPU).

**Hardware extras** (exactly as declared in `pyproject.toml`):
- `hw-cpu-base`, `hw-x64-base`, `hw-arm64-base`, `hw-gpu-nvidia-cuda`, `hw-gpu-amd`, `hw-gpu-intel`, `hw-npu-qualcomm-qnn`, `hw-wake-porcupine`, `dev`

**Evidence token namespace** (colon-separated two-field strings):
- `import:<package>` — package importable in current venv
- `ep:<ExecutionProviderName>` — ONNX Runtime EP registered and available
- `dll:<library_basename>` — DLL discoverable and loadable
- `env:<variable_name>` — operator-supplied env var is set and points at a valid path

All readiness claims surface these tokens verbatim so reasoning traces reconstruct without re-probing.

## Architectural Approach

### Profiler is stdlib + `psutil` only

`backend/app/hardware/profiler.py` and every file under `backend/app/hardware/detectors/` import only from the standard library, `psutil`, and `platform`.

Vendor-specific probes that require vendor libraries live in `backend/app/hardware/preflight.py`, not the profiler. The profiler uses only `platform.machine()`, `platform.system()`, `psutil`, subprocess calls to vendor CLI tools (`nvidia-smi`, `rocm-smi`, `xpu-smi`, PowerShell `Get-CimInstance`), and WMI / registry probes on Windows for NPU identity.

### Provisioning resolver is the only translator between hardware facts and `pip` extras

`backend/app/hardware/provisioning.py::resolve_required_extras(profile: HardwareProfile) -> list[str]` is the single function that translates detected hardware facts into an ordered, additive list of `pyproject.toml` extras. It is deterministic (same input → same output, documented order), serializable (output is plain `list[str]`), and loggable (included in every script's host-fingerprint line).

PEP 508 environment markers in `pyproject.toml` already gate arch-detectable extras (`platform_machine == 'ARM64'` etc.). The resolver adds the gating PEP 508 cannot express: NPU vendor detection, CUDA availability, optional extras like `hw-wake-porcupine`. The resolver never guesses; it reads only from the profile.

### Preflight owns every DLL / backend-path bootstrap and every evidence token

`backend/app/hardware/preflight.py` owns:

- DLL / backend-path bootstrap — `os.add_dll_directory()` calls for Windows CUDA (cuDNN, cuBLAS) and Qualcomm QAIRT (`QnnHtp.dll`). These are **opt-in per bootstrap method** and never run automatically at import time.
- Import probes — wrapped `importlib.import_module()` calls that emit `import:<pkg>` tokens on success, `import:<pkg>:MISSING` on failure.
- EP registration probes — `onnxruntime.get_available_providers()` comparison against expected EPs (CUDA/DML/QNN/CPU) emitting `ep:<Name>` tokens.
- Optional deep probes (session-creation) — exist but are opt-in; they may load model weights and are never called in the default readiness pass.

`derive_{stt,tts,llm,wake}_device_readiness(preflight_result)` helpers translate token sets into `(selected_device, selected_device_ready, reason)` tuples. In Slice A these derivers exist with **device-family-correct return values but no concrete runtime to select** — the tuple shape is fixed, the reason strings name the evidence, and Group B runtimes will consume the tuples unchanged.

### The QNN slot is a structural commitment, not a runtime feature

`hw-npu-qualcomm-qnn` is declared in `pyproject.toml` (already present). In Slice A:
- the resolver returns it on Qualcomm NPU hosts
- preflight's import / EP / DLL probes exist and execute, emitting their tokens
- `CapabilityFlags.qnn_available` is set from NPU vendor detection
- the STT readiness deriver recognizes `qnn` as a valid device **value** — but always returns `(selected_device="cpu", ready=True, reason="qnn defined; STT QNN inference pending H.2")` or equivalent when asked to evaluate a Qualcomm host
- `.env.example` already documents `QAIRT_SDK_PATH`; `config/hardware/notes.md` already explains the quantization-on-x64 workflow

No QNN inference. No quantized model. No runtime file. Activation in H.2 adds exactly two things: the runtime branch that builds a session with `QNNExecutionProvider` + `backend_path: QnnHtp.dll`, and the quantized Whisper catalog entry.

### Tests are arch-aware by marker, never by directory

`backend/tests/conftest.py` runs the profiler + preflight **once per pytest session**, caches the result, and exposes `skip_unless_x64`, `skip_unless_arm64`, `skip_unless_cuda`, `skip_unless_directml`, `skip_unless_qnn`, `skip_unless_ollama`, `skip_unless_redis`, `skip_unless_searxng`, `skip_unless_docker`, `skip_unless_qairt`, `skip_unless_live` as importable `pytest.mark.skipif` constants. Tests declare their requirements with standard markers (already registered in `pyproject.toml`); the conftest-exposed constants are used wherever a synthetic host-class override is needed.

No test file contains raw `if platform.system() == ...` guards. Every host-class gate flows through the harness so the validator can filter deterministically.

### Scripts emit a host-fingerprint line first

Every script introduced by Slice A — `provision.py`, `validate_backend.py`, `bootstrap.py` — and by future slices emits the host-fingerprint as the **first stdout line**:

```
[fingerprint] arch=arm64 python=3.11.8 extras=[hw-arm64-base,hw-cpu-base] profiled=2026-04-23T14:22:18Z
```

This convention is enforced by a shared helper in `backend/app/core/logging.py` that all scripts import. Logs become self-identifying; no log excerpt is ambiguous about which host produced it.

**`backend/app/hardware/detectors/cpu_detector.py`** — returns `{"cpu_name", "cpu_physical_cores", "cpu_logical_cores", "cpu_max_freq_mhz", "arch"}`. **Normalizes arch** to `"amd64"` | `"arm64"` | `"unknown"` regardless of source casing (`AMD64` / `x86_64` → `amd64`; `ARM64` / `aarch64` → `arm64`).

**`backend/app/hardware/detectors/memory_detector.py`** — returns `{"memory_total_gb", "memory_available_gb"}` from `psutil.virtual_memory()`.

**`backend/app/hardware/detectors/gpu_detector.py`** — returns `{"gpu_available", "gpu_name", "gpu_vendor", "gpu_vram_gb", "gpu_vram_source"}`. Vendor-ordered probes (NVIDIA → Intel → AMD) using CLI tools first (`nvidia-smi`, `xpu-smi`, `rocm-smi`), then PowerShell `Get-CimInstance Win32_VideoController` as fallback on Windows. Never imports `pynvml`, `torch`, or `openvino` in the profiler path (those probes live in preflight). If no vendor CLI and no PowerShell evidence, returns `gpu_available=False`.

**`backend/app/hardware/detectors/cuda_detector.py`** — returns `{"cuda_available", "cuda_version"}`. Probe via `nvidia-smi --query-gpu=driver_version` and `nvcc --version` subprocess only. No `torch.cuda.is_available()` — that would require importing torch.

**`backend/app/hardware/detectors/npu_detector.py`** — returns `{"npu_available", "npu_vendor", "npu_tops"}`. Vendor order: Apple MPS hint (Darwin + arm64) → Qualcomm (`platform.machine()` + `platform.processor()` tokens + Windows `Get-CimInstance Win32_PnPEntity` filtering for NPU/Hexagon/neural processing) → Intel (Windows PnP) → AMD (Windows PnP). Returns `("unknown", None, None)` for "I see something NPU-shaped but can't identify vendor" cases. Never imports `qaic`, `openvino`, or similar; those probes live in preflight.

**`backend/app/hardware/profiler.py`** — the root composer:
- `run_profiler() -> FullCapabilityReport` — call each detector in fixed order, merge outputs into `HardwareProfile`, derive `CapabilityFlags` from profile facts, compute a stable `profile_id` (hash of normalized facts), set `profiled_at` (UTC ISO-8601), return `FullCapabilityReport`.
- Handles detector failures by catching exceptions at the profiler boundary and substituting schema-correct unknown values — the profiler itself never raises.

## Sub-Slice A.2 — Provisioning Resolver + Script

### Objective

Introduce `resolve_required_extras()` and `scripts/provision.py`. The profiler alone determines what gets installed.

### Files to Create

**`backend/app/hardware/provisioning.py`** — `resolve_required_extras(profile: HardwareProfile) -> list[str]`:
- Returns an ordered list of extras names (strings matching `pyproject.toml` `[project.optional-dependencies]` keys).
- Deterministic order: `hw-cpu-base` → arch extra (`hw-x64-base` | `hw-arm64-base`) → GPU extras if applicable → NPU extras if applicable → `dev` (always, for developer hosts).
- Rules:
  - Always include `hw-cpu-base`.
  - Include exactly one of `hw-x64-base` or `hw-arm64-base` based on `profile.arch`.
  - If `profile.gpu_available` and `profile.gpu_vendor == "nvidia"` and `profile.cuda_available`: include `hw-gpu-nvidia-cuda`.
  - If `profile.gpu_available` and `profile.gpu_vendor == "amd"`: include `hw-gpu-amd`.
  - If `profile.gpu_available` and `profile.gpu_vendor == "intel"`: include `hw-gpu-intel`.
  - If `profile.npu_available` and `profile.npu_vendor == "qualcomm"`: include `hw-npu-qualcomm-qnn`.
  - `hw-wake-porcupine` is never auto-included — it is opt-in via a separate CLI flag in `provision.py`.
- Returns `list[str]`. No side effects. No I/O.

**`backend/app/core/logging.py`** — shared logging setup consumed by all scripts and future backend modules. Exposes `configure_logging(level, trace_to)` and `emit_host_fingerprint(profile, extras, out=sys.stdout)`. Host-fingerprint helper is used by every script to emit the first-line convention.

**`backend/app/core/paths.py`** — canonical filesystem paths (`REPO_ROOT`, `BACKEND_DIR`, `DATA_DIR`, `MODELS_DIR`, `REPORTS_DIR`, `CONFIG_DIR`). Import-only; no I/O.

**`backend/app/core/settings.py`** — pydantic-settings model reading `.env`. Populates runtime-config values (`APP_NAME`, `CONFIG_PATH`, `DATA_PATH`, `MODEL_PATH`, `OLLAMA_BASE_URL`, etc.). No hardware-family values; those are profile-driven.

**`scripts/provision.py`** — the sole provisioning entry point. Subcommands (exit codes per AGENTS.md §1):
- `install` — run profiler → call `resolve_required_extras()` → compose `pip install -e '.[extra1,extra2,...]'` invocation → execute it. Accepts `--with-porcupine` to add `hw-wake-porcupine` opt-in. Exit 0 on install success, 1 on install failure, 3 if venv is missing or Python is wrong arch.
- `verify` — re-run the resolver against the current host and compare to the installed distribution set (`importlib.metadata.distributions()`). Reports missing / unexpected extras. No install. Exit 0 if aligned, 1 if drift detected.
- `lock` — emit `backend/requirements.txt` as a platform-neutral lockfile of the base extra only. Includes top-of-file comment identifying the file as generated and naming this script + timestamp.
- `dry-run` — print the plan (resolved extras + the exact `pip install` command) without executing. Exit 0.
- `explain` — print each extra chosen and a one-line reason referencing the profile fact that selected it. Exit 0.

Shared flags on every subcommand: `--verbose`, `--dry-run` (overrides mode to not install even on `install`), `--trace-to <dir>`, `--profile` (force re-emit fingerprint line before anything else).

**`backend/tests/unit/hardware/test_provisioning.py`** — unit coverage:
- `test_resolver_returns_base_plus_arch_for_cpu_only_x64`

## Key Design Decisions

**Profiler is stdlib + `psutil` only; preflight owns every hardware-family probe.** The profiler must run before any extras are installed. Separating "what hardware is here" (profiler, stdlib-only) from "what packages are usable" (preflight, after provisioning) preserves both layers' ability to fail closed without polluting the other.

**Extras are the provisioning surface; the resolver translates facts to extras.** `pyproject.toml` is the single source of truth for package sets. The resolver is the only function in the codebase that decides which extras apply. Adding a hardware family later is one extra + one resolver branch + one preflight probe branch — no other file changes.

**Evidence tokens are strings, not booleans.** A failure's reason string cites `ep:CUDAExecutionProvider:MISSING` or `dll:QnnHtp:MISSING` rather than a generic "unavailable." Debugging reconstructs without re-probing.

**The QNN slot is defined structurally, not runtime-active, in Slice A.** Every layer that later needs to accept `"qnn"` as a device value already accepts it; every reason string that later needs to cite `ep:QNNExecutionProvider` already does so. H.2 adds only a runtime branch and a quantized model.

**Tests are marker-gated, not directory-split.** One test file per behavior; markers + `skip_unless_*` decide run/skip per host. Directory structure reflects test purpose (unit/integration/runtime) + functional domain (hardware/voice/turn/desktop/agents/acceleration_matrix), never host class.

**Scripts are orchestrators over `backend/app/**`, never duplicators.** `provision.py` imports `resolve_required_extras()` from the application; `validate_backend.py` imports the profiler and preflight from the application; `bootstrap.py` composes both plus pytest. There is no "script-side" logic that re-implements application rules.

**The host-fingerprint first line is a hard convention.** Every script emits it. Every validator run emits it. Every CHANGE_LOG evidence block can cite it as its first line. Logs become self-identifying.

**`backend/requirements.txt` is only ever generated.** It carries a top-of-file comment declaring so. Any agent or developer who sees it and thinks about hand-editing stops at that comment.

**No runtime code exists at Slice A closeout.** `backend/app/runtimes/` does not exist. No model is acquired. No inference happens. The full Slice A stack answers "what can this host do and can I get the right packages installed?" — nothing more. This is a deliberate architectural commitment that Group B's voice runtime work begins from a proven, stable foundation.

## Risks / Sequencing Notes

- **Detector isolation matters more than coverage.** A single detector that raises would poison the profile and destabilize every downstream decision. Every detector must catch its own exceptions and return schema-correct unknowns.
- **CPython ARM64 availability by point release.** Some CPython point releases may lack a Windows ARM64 installer or have timing gaps before PyPI wheels catch up. Pin the `requires-python` range to a tested window; document in A.5 the exact CPython patch version used.
- **`pip` PEP 508 marker handling varies across versions.** Pin the supported `pip` version in the `dev` extra and verify marker evaluation explicitly in the provision-script unit tests.
- **Qualcomm NPU identity on Windows ARM64.** WMI / PowerShell `Get-CimInstance Win32_PnPEntity` output includes NPUs under various names (Hexagon, Snapdragon, NPU, neural processing unit). The NPU detector must tolerate vendor-specific naming without guessing from OS alone.
- **DirectML candidate flag is best-effort.** `CapabilityFlags.directml_candidate` means "this host plausibly supports DirectML" (Windows + GPU present). Actual DirectML readiness is a preflight concern; runtime-selection honesty lives in the readiness deriver.
- **`CUDAExecutionProvider` versus `CUDAExecutionProvider:MISSING`.** The EP probe must distinguish "onnxruntime installed but CUDA EP not registered" (token: `ep:CUDAExecutionProvider:MISSING` — reason: cuDNN/cuBLAS DLLs not discoverable) from "onnxruntime not installed" (token: `import:onnxruntime:MISSING`). Reasons must point at the actual gap.
- **Idempotent DLL bootstrap.** Multiple callers in one process must produce identical token sets without re-running subprocess / registry work. Memoize at the module boundary keyed on `(arch, vendor, extras_hash)`.
- **The gate on A.5 is the blocker for Group B.** No Group B work begins until both Windows host classes are green on `validate_backend.py regression`. If either host cannot be validated, the blocker is surfaced and the dependent Group B slices do not begin.
- **Do not promote `validate_backend.py` state beyond what Slice A covers.** The `regression` set at Slice A closeout includes hardware unit, provisioning unit, preflight unit, readiness unit, script unit, QNN-slot unit, and the three live baseline tests. It does **not** include any voice-family, turn-engine, desktop, substrate, tool, memory, or agent tests — those extend regression as their slices close.
- **QAIRT SDK availability on developer hardware.** If the only Qualcomm NPU host in developer hands does not yet have QAIRT installed, A.6 may complete with `import:onnxruntime-qnn:MISSING` being the honest token. That is still valid Slice A evidence — the slot is defined; inference is deferred.

## Finish line

Slice A is complete when the repository has:

- a profiler that honestly reports what it sees on every supported host class
- a provisioning authority that translates those facts to a deterministic extras list
- a readiness rail that emits evidence tokens every downstream layer can cite
- an arch-aware test harness that every later slice writes against
- a validation-script surface with documented subcommands and exit codes
- a bootstrap script with inspectable checkpoints that a new developer runs once
- a QNN slot present structurally from day one so later activation is pure wiring
- both Windows host classes green on `validate_backend.py regression` from a clean venv
- no runtime, service, shell, substrate, tool, memory, or agent code anywhere in the repo

Host reality is the foundation. Every slice that follows consumes this foundation unchanged.
