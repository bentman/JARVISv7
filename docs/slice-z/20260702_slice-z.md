# Slice Z — Product Readiness and Operational Polish

Date: 2026-07-02

## Purpose

Slice Z is a stabilization slice. It turns the existing working project into a more coherent repo-run product preview by smoothing startup, launch, readiness, diagnostics, artifacts, desktop state, validation, and documentation.

This slice is not a new core engineering milestone. It should reduce friction, reduce duplicated startup logic, improve user-visible truth, and make the current system easier to start, inspect, and trust.

## General contract

All sub-slices share this contract:

- Preserve existing runtime behavior unless a sub-slice explicitly corrects a product-facing gap.
- Prefer existing modules, scripts, commands, and UI surfaces.
- Keep changes small, observable, and reversible.
- Treat the desktop shell and backend API as the product preview surface.
- Treat `scripts/` as operator/developer entry points, not a second product architecture.
- Improve clarity before adding capability.
- Update tests only where they prove changed behavior or protect a simplified path.
- Update `SYSTEM_INVENTORY.md` and `CHANGE_LOG.md` only after validated observable changes.

## Out of scope

Slice Z does not include:

- semantic/vector memory
- autonomous agents
- new model families
- new accelerator support
- new runtime families
- full installer/distribution pipeline
- major desktop redesign
- broad test-suite expansion
- cloud-provider expansion
- new service substrate
- new governance framework

## Parent acceptance

Slice Z is acceptable when the current repo-run product preview has one clear launch path, accurate readiness/degraded status, useful desktop/backend diagnostics, richer turn artifacts for real turns, and one focused product smoke validation path.
