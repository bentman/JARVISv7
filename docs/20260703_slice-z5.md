# 20260703_slice-z5.md
## Slice Z.5 — Personality Check-up

Status: planned. This document is an implementation contract. It is not completion evidence.

---

## Purpose

Slice Z.5 makes personality profiles observable in real local turns.

The target question is:

```text
When a profile is selected, does the next local LLM request receive profile instructions in the strongest request shape the selected runtime supports?
```

The outcome is not a larger personality catalog. The outcome is a verified personality path from YAML profile, through backend policy compilation, into the prompt envelope, into the runtime-specific request payload used by llama.cpp and Ollama.

---

## Current Codebase Evidence

The repo already has most of the profile plumbing, but the final local runtime request weakens the profile signal.

- `config/personality/*.yaml`
  - Existing profiles use a small flat schema with fields such as `profile_id`, `display_name`, `identity_summary`, `tone`, `brevity`, `formality`, `warmth`, `assertiveness`, `humor_policy`, and speech settings.

- `backend/app/personality/schema.py`
  - Defines strict allowed profile fields and enum values.
  - Accepts `system_prompt_addendum`, but that field is not currently used by the compiled prompt policy.
  - Rejects prohibited authority fields such as tool, routing, memory, safety, hidden-instruction, and raw system-prompt controls.

- `backend/app/personality/loader.py`
  - `list_personality_profiles()` loads every `*.yaml` under `config/personality`.
  - A malformed or invalid profile file can break the whole profile list endpoint because per-file load errors are not isolated.

- `backend/app/personality/policy.py`
  - `compile_personality_policy(...)` emits `identity_summary` and enum-derived style/speech rules.
  - The compiled policy includes lines such as tone, brevity, warmth, humor policy, response style, and voice pacing.
  - The compiled policy does not include `system_prompt_addendum`.

- `backend/app/cognition/prompt_assembler.py`
  - Builds a `PromptEnvelope` with separate segments for application instruction, persona style, session context, memory context, retrieval context, tool context, user input, and output contract.
  - This envelope has the right internal shape for runtime-specific rendering.

- `backend/app/runtimes/llm/base.py`
  - `LLMBase.generate_envelope(...)` currently renders the entire `PromptEnvelope` into one flat prompt string, then calls `generate(...)`.
  - This default is a compatibility path, not the strongest path for chat-style local runtimes.

- `backend/app/runtimes/llm/local_runtime.py`
  - `LlamaCppLLM` uses `/v1/chat/completions`.
  - Its current request places the full rendered prompt into one `user` message.
  - This loses the role separation already present in the prompt envelope.

- `backend/app/runtimes/llm/ollama_runtime.py`
  - `OllamaLLM` currently uses `/api/generate` with one flat `prompt`.
  - It does not use `/api/chat` messages or `/api/generate`'s separate `system` field.

- `backend/app/api/routes/personality.py`
  - `/personality/list` returns the active profile id and summaries.
  - `/personality/select` loads a profile, updates API state, updates `state.engine.personality`, and updates `SessionService`.
  - This means selected profiles are intended to affect later turns without replacing the active session.

- `desktop/src/main.js`
  - The desktop lists/selects profiles and updates the visible personality metadata.
  - Presence text is currently hard-coded for `default`, `concise`, and `warm`; profiles outside those ids receive default presence messages.
  - Presence text is separate from model personality and should not be used as proof that personality changed in the model request.

## Research Findings

The local runtimes already support stronger request shapes than the repo currently uses.

- llama.cpp server documentation describes OpenAI-compatible chat completion, responses, and Anthropic-compatible messages routes. The OpenAI-compatible chat route supports message-based requests, which is the appropriate path for preserving application/persona/user separation.
  - Reference: `https://raw.githubusercontent.com/ggml-org/llama.cpp/master/tools/server/README.md`

- Ollama API documentation supports `/api/chat` with a `messages` array. Ollama `/api/generate` also supports a separate `system` field that overrides the model's system prompt for that request.
  - Reference: `https://raw.githubusercontent.com/ollama/ollama/main/docs/api.md`

- OpenAI's current text-generation documentation uses high-priority `instructions` and role-separated `developer` / `user` messages for behavior and style instructions. This is the cloud escalation pattern to preserve when cloud routes are reactivated.
  - Reference: `https://developers.openai.com/api/docs/guides/text`

Cloud LLM work is not part of the immediate local-runtime fix. Z.5 should leave a clean extension point so cloud runtimes can map the same prompt envelope into provider-native instruction/message roles later.

## Target Mechanism

Keep `PromptEnvelope` as the internal source of truth. Add a runtime-specific rendering layer that preserves role separation where the target runtime supports it.

Recommended shape:

```text
PromptEnvelope
  -> runtime-neutral structured messages
       system/developer instruction text
       persona instruction text
       untrusted context text
       current user request
       output contract
  -> runtime-specific payload
       llama.cpp OpenAI-compatible chat messages
       Ollama /api/chat messages or /api/generate system + prompt
       future cloud provider request shapes
```

Recommended helper location:

```text
backend/app/cognition/prompt_chat_renderer.py
```

Recommended helper outputs:

```text
ChatPromptPayload
  system_text: str
  user_text: str
  messages: list[dict[str, str]]
```

Use trust boundaries from the existing envelope:

- Application instruction, personality style/speech rules, and output contract are trusted application-controlled instructions.
- User transcript is user content.
- Memory, retrieval, and tool context remain labeled context and must not be promoted into higher-authority instruction roles.

## Runtime Adaptation Targets

### llama.cpp server

Update `LlamaCppLLM.generate_envelope(...)` to send OpenAI-compatible chat messages instead of flattening the whole envelope into one `user` message.

Preferred payload shape:

```json
{
  "model": "<model>",
  "messages": [
    {"role": "system", "content": "<application + persona + output contract>"},
    {"role": "user", "content": "<context labels + latest user request>"}
  ],
  "stream": false
}
```

Preserve existing generation defaults such as temperature, top-p, top-k, repeat penalty, max tokens, stop strings, and chat template kwargs.

Keep flat `generate(prompt, ...)` as a compatibility method for callers that do not have a `PromptEnvelope`.

### Local Ollama app

Update `OllamaLLM.generate_envelope(...)` to use one of the supported personality-preserving paths.

Preferred path:

```text
POST /api/chat
```

with payload:

```json
{
  "model": "<model>",
  "messages": [
    {"role": "system", "content": "<application + persona + output contract>"},
    {"role": "user", "content": "<context labels + latest user request>"}
  ],
  "stream": false,
  "options": {"num_ctx": "<existing setting when present>"}
}
```

Fallback path when `/api/chat` is not used:

```text
POST /api/generate
```

with separate `system` and `prompt` fields:

```json
{
  "model": "<model>",
  "system": "<application + persona + output contract>",
  "prompt": "<context labels + latest user request>",
  "stream": false
}
```

### Cloud escalation placeholder

Add an interface boundary for future cloud runtimes to consume `PromptEnvelope` without forcing a flat prompt. Provider-specific implementations should map the envelope into each provider's native instruction and message fields.

Examples to preserve as design targets:

- OpenAI Responses: `instructions` or `developer` message for application/persona policy, user message/input for user content.
- Anthropic Messages: provider-native system/instruction field plus user messages.

No cloud provider activation is required for Z.5.

## Profile Contract Cleanup

Make profile files reliable and the UI truthful.

Backend profile handling:

- Keep the current allowlist/prohibited-field model.
- Isolate invalid profile files in `list_personality_profiles()` so one bad YAML file does not make `/personality/list` return 500.
- Report profile-load problems through logs, diagnostics, or an explicit response field when the schema is expanded.
- Either wire `system_prompt_addendum` into `compile_personality_policy(...)` as an application-controlled style addendum, or remove it from the profile contract and existing configs.
- Keep profile values from changing tool, routing, memory, or safety authority.

Desktop profile handling:

- Keep current selected-profile display.
- Preserve session continuity when selecting a profile.
- Add a low-noise indication when profile list loading partially succeeds with skipped invalid profiles if the backend exposes that detail.
- Treat desktop presence lines as UI feedback only. They do not prove that the selected profile changed the model prompt.

## Observable Verification Target

Z.5 needs tests that prove both routing and effect.

Minimum unit-level proof:

- Selecting a profile updates the active engine personality for the next turn.
- Compiled personality policy includes the selected profile's observable identity/style data.
- `system_prompt_addendum` behavior is tested according to the chosen contract.
- The prompt envelope structured renderer preserves separate trusted instruction text and user/context text.
- `LlamaCppLLM.generate_envelope(...)` posts role-separated chat messages to `/v1/chat/completions`.
- `OllamaLLM.generate_envelope(...)` posts role-separated messages to `/api/chat`, or separate `system` and `prompt` fields to `/api/generate`.
- Invalid YAML/enum profile files do not break the whole `/personality/list` route.

Manual product proof:

- Select `default`, submit a style-sensitive prompt, record response.
- Select `concise`, submit the same prompt in the same session, observe a materially shorter/directer answer.
- Select `warm` or another expressive profile, submit the same prompt, observe a materially different style.
- Inspect the recorded turn artifact or test-captured runtime payload to confirm the selected profile id and structured persona instruction reached the runtime request path.

Use deterministic-ish prompts for verification, for example:

```text
Explain how you would report a degraded local LLM sidecar to the operator in two sentences.
```

## Acceptance Criteria

Slice Z.5 is complete when:

- Profile selection still preserves the active session and applies to the next turn.
- The selected profile materially affects the runtime request payload, not just desktop metadata.
- llama.cpp receives role-separated chat messages through its OpenAI-compatible chat endpoint.
- Ollama receives role-separated messages through `/api/chat`, or separate `system` and `prompt` fields through `/api/generate`.
- The default flat prompt renderer remains available as a compatibility fallback.
- Invalid profile files no longer make `/personality/list` fail wholesale.
- `system_prompt_addendum` is either wired into compiled policy with tests or removed from accepted config and existing profiles.
- Tests capture the exact llama.cpp and Ollama request payload shape for selected profiles.
- Manual validation shows a visible response-style difference between at least two profiles on the same local runtime.
- Cloud LLM runtime paths are left with a clear envelope-to-provider mapping boundary for future escalation work.

## Stop Conditions

Stop and report before editing further when:

- a local runtime cannot support a stronger structured request path with the currently selected model/server;
- preserving role separation requires changing tool, memory, routing, or safety authority;
- personality changes require a new model family or cloud activation;
- the desktop profile selector must be redesigned beyond current profile selection and status display;
- validation cannot prove the selected profile reached the runtime payload.
