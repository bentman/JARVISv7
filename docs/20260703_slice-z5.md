# 20260703_slice-z5.md
## Slice Z.5 — Personality Check-up

Status: planned. This document is an implementation contract. It is not completion evidence.

---

## Purpose

Slice Z.5 makes personality profiles observable in real local turns.

The target question is:

```text
When a profile is selected, does the next local LLM request receive profile instructions through the standard structured request contract used by that runtime?
```

The outcome is a verified personality path from YAML profile, through backend policy compilation, into the prompt envelope, into the runtime-specific request payload used by llama.cpp and Ollama.

---

## Current Codebase Evidence

The repo already has most profile plumbing. The final local runtime request weakens the profile signal.

- `config/personality/*.yaml`
  - Existing profiles use flat fields: `profile_id`, `display_name`, `identity_summary`, `tone`, `brevity`, `formality`, `warmth`, `assertiveness`, `humor_policy`, `response_style`, `acknowledgment_style`, `confirmation_style`, `interruption_style`, `voice_pacing`, `voice_energy`, and `enabled`.

- `backend/app/personality/schema.py`
  - Defines strict allowed profile fields and enum values.
  - Accepts `system_prompt_addendum`.
  - Rejects prohibited authority fields: tool, routing, memory, safety, hidden-instruction, and raw authority override controls.

- `backend/app/personality/loader.py`
  - `list_personality_profiles()` loads every `*.yaml` under `config/personality`.
  - A malformed profile file can break the whole profile list endpoint because per-file load errors are not isolated.

- `backend/app/personality/policy.py`
  - `compile_personality_policy(...)` emits `identity_summary` and enum-derived style and speech rules.
  - The compiled policy includes tone, brevity, warmth, humor policy, response style, acknowledgement style, confirmation style, interruption style, voice pacing, and voice energy.
  - The compiled policy does not include `system_prompt_addendum`.

- `backend/app/cognition/prompt_assembler.py`
  - Builds a `PromptEnvelope` with separate segments for application instruction, persona style, session context, memory context, retrieval context, tool context, user input, and output contract.
  - This envelope has the required internal shape for runtime-specific rendering.

- `backend/app/runtimes/llm/base.py`
  - `LLMBase.generate_envelope(...)` renders the entire `PromptEnvelope` into one flat prompt string, then calls `generate(...)`.
  - This remains the compatibility path for non-chat runtimes.

- `backend/app/runtimes/llm/local_runtime.py`
  - `LlamaCppLLM` calls `/v1/chat/completions`.
  - Its current payload places the full rendered prompt into one `user` message.
  - This discards the role separation present in `PromptEnvelope`.

- `backend/app/runtimes/llm/ollama_runtime.py`
  - `OllamaLLM` calls `/api/generate` with one flat `prompt`.
  - It does not call `/api/chat`.
  - It does not pass persona policy through a separate system message.

- `backend/app/api/routes/personality.py`
  - `/personality/list` returns active profile id and summaries.
  - `/personality/select` loads a profile, updates API state, updates `state.engine.personality`, and updates `SessionService`.
  - Selected profiles are intended to affect later turns while preserving the active session.

- `desktop/src/main.js`
  - The desktop lists profiles, selects profiles, and updates visible personality metadata.
  - Presence text is hard-coded for `default`, `concise`, and `warm`.
  - Presence text is UI feedback, not evidence that model personality changed.

## Research Findings

The local runtimes support standard structured instruction patterns that the repo is not using.

- llama.cpp server exposes OpenAI-compatible chat completions, responses, embeddings, and Anthropic-compatible messages routes. Use the OpenAI-compatible chat completion route for Z.5.
  - Reference: `https://raw.githubusercontent.com/ggml-org/llama.cpp/master/tools/server/README.md`

- llama.cpp server exposes sampling fields that match common OpenAI-compatible request vocabulary and existing repo defaults: `temperature`, `top_k`, `top_p`, `repeat_penalty`, `presence_penalty`, `frequency_penalty`, `max_tokens`, and `stop`.
  - Reference: `https://raw.githubusercontent.com/ggml-org/llama.cpp/master/tools/server/README.md`

- Ollama API supports `/api/chat` with a `messages` array. Use `/api/chat` for Z.5.
  - Reference: `https://raw.githubusercontent.com/ollama/ollama/main/docs/api.md`

- Ollama Modelfile uses the established config vocabulary `SYSTEM`, `MESSAGE`, `PARAMETER`, and `TEMPLATE`. Runtime profile switching in this app must use `/api/chat` messages. Modelfile fields define the vocabulary and static model defaults.
  - Reference: `https://raw.githubusercontent.com/ollama/ollama/main/docs/modelfile.mdx`

- OpenAI text-generation guidance uses the `instructions` parameter and role-separated `developer` and `user` messages. This defines the cloud escalation shape to preserve.
  - Reference: `https://developers.openai.com/api/docs/guides/text`

- Anthropic guidance uses a `system` parameter with `messages` and recommends clear roles, examples, and explicit style constraints. This defines the Claude escalation shape to preserve.
  - Reference: `https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices`

Cloud LLM activation is not part of Z.5. Z.5 creates the envelope-to-provider boundary used by future cloud escalation.

## Canonical Personality Config Contract

Replace the current weak profile contract with fields that map directly to standard runtime request concepts.

Target YAML fields:

```yaml
profile_id: jarvis
display_name: JARVIS
locale: en_GB
system_prompt: >-
  Application-controlled personality and behavior instructions.
style_rules:
  - A short imperative style rule.
speech_rules:
  - A short TTS-safe speech rule.
example_messages:
  - role: user
    content: Example user prompt.
  - role: assistant
    content: Example assistant answer that demonstrates style.
generation:
  temperature: 0.7
  top_p: 0.9
  top_k: 40
  repeat_penalty: 1.08
  max_tokens: 220
  stop:
    - "\nUser:"
    - "\nAssistant:"
enabled: true
```

Mapping rules:

- `system_prompt` becomes trusted application/persona instruction text.
- `style_rules` become trusted application/persona instruction text.
- `speech_rules` become trusted application/persona instruction text for spoken output safety and pacing.
- `example_messages` become role-separated demonstration messages after the system message and before the current user message.
- `generation` becomes validated runtime generation defaults for the selected profile.
- `locale` replaces `response_language` in new profile files.
- `identity_summary`, enum style fields, and `system_prompt_addendum` are migrated into the new fields during implementation.
- Prohibited authority fields remain rejected.

## Target Mechanism

Keep `PromptEnvelope` as the internal source of truth. Add one structured renderer that produces the exact fields required by local chat runtimes.

Required helper:

```text
backend/app/cognition/prompt_chat_renderer.py
```

Required output:

```text
ChatPromptPayload
  system_text: str
  user_text: str
  messages: list[dict[str, str]]
  generation: dict[str, object]
```

Trust-boundary mapping:

- Application instruction, personality policy, speech policy, output contract, and example assistant messages become system/developer-level instruction content.
- User transcript becomes user content.
- Memory, retrieval, and tool context remain labeled context inside user content.
- Memory, retrieval, and tool context never become system/developer content.

## Runtime Adaptation Targets

### llama.cpp server

Implement `LlamaCppLLM.generate_envelope(...)` with OpenAI-compatible chat completion messages.

Required endpoint:

```text
POST /v1/chat/completions
```

Required payload shape:

```json
{
  "model": "<model>",
  "messages": [
    {"role": "system", "content": "<system_text>"},
    {"role": "user", "content": "<user_text>"}
  ],
  "stream": false,
  "temperature": 0.7,
  "top_p": 0.9,
  "top_k": 40,
  "repeat_penalty": 1.08,
  "max_tokens": 220,
  "stop": ["\nUser:", "\nAssistant:"],
  "chat_template_kwargs": {"enable_thinking": false}
}
```

`generate(prompt, ...)` remains the flat compatibility method. `generate_envelope(...)` is the production path for turns.

### Local Ollama app

Implement `OllamaLLM.generate_envelope(...)` with Ollama chat messages.

Required endpoint:

```text
POST /api/chat
```

Required payload shape:

```json
{
  "model": "<model>",
  "messages": [
    {"role": "system", "content": "<system_text>"},
    {"role": "user", "content": "<user_text>"}
  ],
  "stream": false,
  "options": {
    "temperature": 0.7,
    "top_p": 0.9,
    "top_k": 40,
    "repeat_penalty": 1.08,
    "num_predict": 220,
    "num_ctx": 4096,
    "stop": ["\nUser:", "\nAssistant:"]
  }
}
```

`/api/generate` remains the flat compatibility method. `generate_envelope(...)` is the production path for turns.

### Cloud escalation boundary

Create a provider-neutral boundary that preserves the same `ChatPromptPayload` data for future cloud runtimes.

Required mapping targets:

- OpenAI Responses: `instructions` receives `system_text`; `input` receives role-separated developer and user messages.
- Anthropic Messages: `system` receives `system_text`; `messages` receives user and assistant examples plus current user message.

Cloud runtime activation remains outside Z.5.

## Profile Contract Cleanup

Backend profile handling:

- Keep the current allowlist and prohibited-field model.
- Isolate invalid profile files in `list_personality_profiles()` so one bad YAML file does not make `/personality/list` return 500.
- Expose skipped profile diagnostics through a response field named `profile_errors`.
- Compile `system_prompt`, `style_rules`, `speech_rules`, `example_messages`, `generation`, and `locale` into `PersonalityPolicy`.
- Remove `system_prompt_addendum` from active config files after migration.
- Keep profile values from changing tool, routing, memory, and safety authority.

Desktop profile handling:

- Keep current selected-profile display.
- Add profile load diagnostics when `profile_errors` is present.
- Preserve active session continuity when selecting a profile.
- Stop treating hard-coded presence lines as personality proof.

## Observable Verification Target

Z.5 tests must prove routing and effect.

Minimum unit-level proof:

- Selecting a profile updates the active engine personality for the next turn.
- Compiled policy includes selected profile `system_prompt`, `style_rules`, `speech_rules`, `example_messages`, and `generation` fields.
- The structured renderer preserves separate trusted instruction text and user/context text.
- `LlamaCppLLM.generate_envelope(...)` posts role-separated chat messages to `/v1/chat/completions`.
- `OllamaLLM.generate_envelope(...)` posts role-separated chat messages to `/api/chat`.
- Invalid YAML and invalid enum profile files do not break the whole `/personality/list` route.

- Select `default`, submit the verification prompt, record response.

- Select `warm`, submit the same verification prompt in the same session, observe warmer wording.
- Select `jarvis`, submit the same verification prompt in the same session, observe precise British-influenced dry wit.
- Inspect captured runtime payload in test evidence to confirm selected profile id and structured persona instruction reached the local runtime request.

## Acceptance Criteria

Slice Z.5 is complete when:

- Profile selection preserves the active session and applies to the next turn.
- The selected profile changes the runtime request payload, not only desktop metadata.
- llama.cpp receives role-separated chat messages through `/v1/chat/completions`.
- Ollama receives role-separated chat messages through `/api/chat`.
- The default flat prompt renderer remains available as a compatibility fallback.
- Invalid profile files no longer make `/personality/list` fail wholesale.
- `profile_errors` exposes skipped profile diagnostics.
- `system_prompt_addendum` is removed from active config files and replaced by `system_prompt`.
- Tests capture exact llama.cpp and Ollama request payloads for selected profiles.
- Manual validation shows visible response-style differences between `default`, `concise`, `warm`, and `jarvis` on the same local runtime.
- Cloud LLM runtime paths have a provider-neutral envelope mapping boundary for future escalation work.

## Stop Conditions

Stop and report before editing further when:

- selected local runtime lacks `/v1/chat/completions` and `/api/chat` support in the inspected environment;
- preserving role separation requires changing tool, memory, routing, and safety authority;
- personality changes require a new model family;
- cloud activation becomes necessary for local personality validation;
- validation cannot prove selected profile reached the runtime payload.
