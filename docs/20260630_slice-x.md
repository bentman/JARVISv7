# 20260630_slice-x.md
## Slice X — Local LLM Tier Activation

Status: planned. This document is an implementation contract. It is not completion evidence.

---

## Purpose

Turn the existing LLM model-policy mechanism into useful runtime behavior by adding real, defined local model tiers and mapping them through the current catalog, setup, readiness, Operator, and llama.cpp runtime paths.

Use the existing flow:

```text
config/models/llm.yaml -> select_llm_model(...) -> resolve_llm_serve_profile(...) -> ensure_models.py -> llama.cpp sidecar -> local runtime/readiness
```

The work is complete only when the repository functions better end-to-end:

- new clones can set up the selected model through existing setup commands
- existing repos can move from the legacy development model to the new defined model ladder without manual repair
- normal policy-based operation does not depend on the legacy development model artifact
- Operator/readiness surfaces show truthful selected model state
- supported runtime behavior improves without breaking existing local assistant functionality

---

## Implementation Discipline

KISS/YAGNI means do not over-complicate or over-implement. It does not mean reducing the intended function.

Required scope must still be implemented. Avoid extra architecture, extra workflows, and speculative features, but do not skip setup, existing-repo transition, Operator/readiness truth, legacy model independence, hardware-safe selection, or validation needed to prove the outcome.

Do not add:

- a parallel model selector
- a new model manager
- a new downloader workflow
- a new runtime launcher
- a desktop-only model system
- a new setup entry point
- model binaries committed to git

Do add or adjust whatever existing repo-owned code is necessary for the intended Slice X function to work safely.

## Current Repo Baseline

`config/models/llm.yaml` already defines:

- `llm_selection.default_policy: auto`
- Operator env keys `LLM_MODEL_POLICY` and `LLM_MODEL_ID`
- roles `portable`, `balanced`, `quality`, `vision_preview`, and `diagnostic`
- hardware-profile policy selectors for AMD64, ARM64, CUDA, AMD GPU, Intel GPU, Qualcomm Adreno OpenCL, Qualcomm NPU base, and Qualcomm QNN
- llama.cpp serve profiles under each model entry

Current limitation:

```text
All policy roles still resolve to assistant-small-q4.
The policy selector works, but the model ladder is not functionally active.
```

Slice X activates the ladder using existing repo mechanisms.

## Required Outcomes

1. New clone setup works.
   - `scripts/bootstrap.py` remains the normal setup entry point.
   - `scripts/ensure_models.py` fetches/verifies the selected model and current-host runtime sidecar.
   - No manual model-path editing is required for normal use.

2. Existing repos transform safely.
   - Blank or absent `LLM_MODEL_ID` uses policy selection.
   - Existing explicit legacy overrides remain explicit and visible.
   - Clearing a legacy override returns the repo to policy selection.
   - Legacy artifacts may remain on disk but are not required by normal policy paths.

3. Legacy model independence is achieved.
   - Auto, Portable, Balanced, and Quality do not require `assistant-small-q4` artifacts.
   - If `assistant-small-q4` remains, it is legacy compatibility only or an explicitly named diagnostic fallback.
   - Removing `models/llm/assistant-small-q4/` does not break normal policy-selected operation.

4. Operator experience stays simple.
   - Operator selects policy first, not raw paths.
   - Active model, policy, role, serve profile, accelerator/degraded state, and selection reason are visible in readiness/status.
   - Raw model id/path/binary overrides remain advanced escape hatches.

5. Runtime behavior improves without false claims.
   - At least one non-legacy model is selected by default on a supported path.
   - Unsupported accelerators degrade cleanly.
   - No vision, AMD/Intel GPU acceleration, Adreno acceleration, NPU, or QNN LLM support is claimed without validation evidence.

6. Application compatibility is preserved.
   - Existing voice, text, local turn, readiness, setup, Operator, and desktop static behavior must not regress.
   - Ollama fallback/development settings must remain separate from llama.cpp policy selection.
   - Existing advanced overrides must remain visible and fail closed when invalid.
   - Schema/API changes must be backward-compatible unless a focused test proves the intentional contract change.

## Model Ladder

Use Qwen-family GGUF models for the first activation unless implementation evidence proves a different family is safer. Gemma remains a future alternate, especially for a separate vision or long-context slice. Slice X does not activate multimodal input.

Recommended roles:

| Role | Candidate | Function |
|---|---|---|
| `diagnostic` | Qwen3 0.6B or 1.7B GGUF | Fast smoke tests, lowest-cost fallback |
| `portable` | Qwen3 4B Q4-class GGUF | Default cross-host local assistant model |
| `balanced` | Qwen3 8B Q4/Q5-class GGUF | Better reasoning where acceleration is proven |
| `quality` | Qwen3 14B Q4-class GGUF | Explicit quality trial on capable AMD64 CUDA only |
| `vision_preview` | unchanged placeholder | No runtime activation in this slice |

Implementation must verify exact source repo, filename, quantization, expected size, local path, license/provenance, and route support before adding a model entry. Do not invent artifact metadata.

## Hardware Policy Targets

These are best-effort targets from the existing hardware profiles. Unproven accelerators must remain degraded or unavailable until validated.

| Hardware selector | Auto target | Required behavior |
|---|---|---|
| `windows_amd64_cpu` | `portable` or `diagnostic` if latency requires | CPU local llama.cpp must remain usable; no CUDA claim. |
| `windows_arm64_cpu` | `portable` or `diagnostic` if latency requires | Must work without Adreno or QNN. |
| `windows_amd64_gpu_nvidia_cuda` | `balanced` | Primary activation target; validate on RTX 3060-class CUDA if available. |
| `windows_amd64_gpu_amd` | `portable` | Do not claim Vulkan/HIP acceleration without pinned artifact and proof. |
| `windows_amd64_gpu_intel` | `portable` | Do not claim Vulkan/OpenVINO/SYCL acceleration without pinned artifact and proof. |
| `windows_arm64_gpu_qualcomm_adreno_opencl` | `portable` | Do not require OpenCL sidecar for setup; report acceleration only if present and proven. |
| `windows_arm64_npu_qualcomm_base` | `portable` via non-NPU runtime | No Hexagon/NPU LLM claim. |
| `windows_arm64_npu_qualcomm_qnn` | `portable` via non-QNN runtime | No QNN LLM claim. QNN STT proof does not prove QNN LLM. |

Policy behavior:

```text
Auto:
- AMD64 CUDA -> balanced
- all other current selectors -> portable unless validation requires diagnostic

Portable:
- all selectors -> portable

Balanced:
- AMD64 CUDA -> balanced
- other selectors -> portable or explicit unsupported/degraded result

Quality:
- AMD64 CUDA only after proof -> quality
- other selectors -> explicit unsupported/degraded result

Diagnostic:
- all selectors -> diagnostic

Vision Preview:
- placeholder only; no runtime claim
```

## Application Compatibility Requirements

Slice X is not only a catalog edit. The selected model must work through the application paths that depend on model selection.

Required compatibility checks:

- catalog loading still works
- model selection still resolves by route, policy, override, and hardware selector
- serve-profile resolution still returns a valid llama.cpp sidecar profile
- setup still uses `scripts/bootstrap.py`
- model/runtime acquisition still uses `scripts/ensure_models.py`
- local llama.cpp startup still passes the selected model id into serve-profile resolution
- readiness/status still exposes selected model truth
- Operator settings still expose policy first and raw overrides as advanced controls
- desktop settings rendering remains metadata-driven unless a functional display gap requires a minimal change
- text/local-turn routes still use the selected local LLM path
- voice routes are not broken by the model change
- Ollama remains a separate fallback/development path, not the default policy-selected local LLM path
- unsupported hardware paths degrade explicitly instead of silently selecting an unavailable artifact

A change that passes isolated catalog tests but breaks setup, readiness, Operator, or local turn behavior is not complete.

## Implementation Order

### X.0 Baseline

Confirm current selection and setup behavior before edits.

Acceptance:

```text
Existing selected model, policy, role, serve profile, setup behavior, and ensure_models behavior are known before changes.
Pre-existing failures are recorded before edits.
```

### X.1 Model Entries

Add verified model entries for the ladder.

Acceptance:

```text
Catalog loads.
Each new model id resolves.
Each new model declares required routes and local artifact metadata.
Serve-profile resolution works for every selector that can select the model.
Artifact metadata is verified rather than invented.
```

### X.2 Role Mapping

Map policies to real model roles.

Acceptance:

```text
Auto selects a non-legacy model on at least one supported path.
Every hardware selector has a deterministic result.
Unsupported acceleration remains explicit and truthful.
No normal policy role depends on assistant-small-q4 unless it is intentionally retained as diagnostic legacy.
```

### X.3 Setup and Existing Repo Transform

Make setup and existing repo transition work through existing commands.

Acceptance:

```text
bootstrap dry-run reports selected model behavior.
ensure_models verify-only reports selected model id, role, policy, and hardware profile.
Default policy operation does not require assistant-small-q4 artifacts.
Explicit legacy override remains explicit and visible.
Clearing a legacy override returns to policy selection.
```

### X.4 Operator and Readiness

Keep model selection understandable and truthful.

Acceptance:

```text
Operator can select policy without editing raw paths.
Readiness/status reports active model id, role, serve profile, degraded state, and selection reason.
Desktop remains metadata-driven unless a functional display gap exists.
Unsupported hardware or unavailable artifacts produce visible degraded/unavailable state.
```

### X.5 Holistic Runtime Proof

Prove function on available hardware and prove no adjacent application breakage.

Minimum validation:

```powershell
backend\.venv\Scripts\python -m pytest backend\tests\unit\models backend\tests\unit\scripts\test_ensure_models_llm_runtime_artifacts.py backend\tests\unit\services\test_local_llm_startup.py backend\tests\unit\api\test_routes.py backend\tests\unit\desktop\test_desktop_static_contract.py -q

backend\.venv\Scripts\python scripts\bootstrap.py --dry-run
backend\.venv\Scripts\python scripts\validate_backend.py profile
```

Runtime proof, when hardware is available:

```powershell
$env:JARVISV7_LIVE_TESTS = "1"
backend\.venv\Scripts\python -m pytest backend\tests\runtime\voice\test_llm_live.py backend\tests\runtime\turn\test_local_llm_turn_live.py -q
```

Closeout validation should include the normal backend unit/regression validator when implementation touches setup, runtime selection, readiness, or Operator/API behavior:

## Out of Scope

- multimodal or image input runtime
- Gemma vision activation
- NPU/QNN LLM support claim
- AMD/Intel GPU acceleration claim without pinned artifact and validation
- new setup workflow
- new desktop model manager
- committing model binaries
- automatic deletion of legacy artifacts
- changelog or inventory update during implementation

## Finish Line

Slice X is complete when:

- real model entries exist for the selected ladder
- Auto policy maps every current hardware selector to a safe role
- at least one non-legacy model is selected by default on a supported path
- new clone setup can fetch/verify selected artifacts through existing commands
- existing repos can clear legacy overrides and use policy selection
- normal Auto, Portable, Balanced, and Quality behavior does not require `assistant-small-q4`
- Operator/readiness surfaces report truthful selected model state
- setup, readiness, Operator, desktop static, local turn, and available voice/runtime paths do not regress
- unsupported acceleration remains degraded or unavailable
- focused tests, setup dry-run, selected model verify-only, host profile validation, unit/regression validation, and available runtime proof pass

Governance updates are separate closeout work after validation evidence exists.
