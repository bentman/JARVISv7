# 20260630_slice-w.md
## Slice W — Environment and Operator Settings Posture Cleanup

Status: planned. This document is an implementation contract for a coding agent. It is not completion evidence.

---

## Purpose

Reduce `.env` / `.env.example` and Operator Settings sprawl without breaking the existing shell -> `.env` -> `.env.example` -> backend default fallback behavior. The slice should make normal configuration smaller and safer by moving repo-safe defaults into `backend/app/core/settings.py`, deriving redundant local-runtime values where appropriate, and keeping advanced/power-user overrides available without making them the primary Operator experience.

This is a housekeeping slice, not a redesign. Fix tight mismatches discovered during implementation, such as a base URL/port pair that can drift between backend and Docker, but do not broaden into a new settings system.

---

## Agent Contract

Implementation agents must follow these rules:

- Preserve existing validated behavior unless this slice explicitly changes it.
- Do not delete an env var from active support merely because it is removed from primary `.env.example` posture.
- Do not create a parallel settings registry or new setup workflow.
- Prefer centralizing classification/default behavior in existing settings/config surfaces over adding helper docs or scripts.
- Repurpose and extend existing settings, config-route, desktop static, and setup tests before creating new tests.
- Add new tests only when no existing test file is the natural owner.
- If Operator API metadata, grouping, visibility, or field ordering changes, update the desktop rendering contract and desktop/static tests in the same sub-slice.
- Keep this slice tight: if a mismatch is directly adjacent to `.env.example`, `Settings`, Operator Settings, Docker service URLs, or local LLM defaults, fix it; otherwise record it as out of scope.
- Append `CHANGE_LOG.md` only after validation evidence exists for each slice.
- Do not update `SYSTEM_INVENTORY.md` until the capability is verified on both Windows AMD64 and Windows ARM64, unless the user explicitly approves a narrower inventory state.

## Baseline Consumed

- `.env.example` currently acts as both documentation and runtime fallback when `.env` is absent. This slice may reduce and reorganize it, but must preserve safe no-`.env` behavior.
- `backend/app/core/settings.py` is the central backend settings loader and default source. This slice should add repo-safe defaults and comments/classification there rather than relying on `.env.example` to carry every safe value.
- `backend/app/api/routes/config.py` owns Operator Settings allowlisting and field metadata. This slice should improve grouping/visibility there instead of adding a second Operator settings source.
- `desktop/src/components/settings-panel.js` renders Operator settings based on API metadata. This slice must update it if API metadata changes require different grouping, visibility, or advanced-field behavior.
- `backend/tests/unit/desktop/test_desktop_static_contract.py` is the static desktop/API contract owner. Any Operator Settings contract change must update this test rather than leaving stale assertions.
- `scripts/ensure_models.py` and `backend/app/services/local_llm_startup.py` already consume local-model settings. This slice may derive fetch/managed behavior when local model mode is enabled, while preserving explicit override behavior.
- `docker-compose.yml` owns Redis/SearXNG host port mapping. This slice may fix direct backend-vs-Docker env drift where a backend base URL and Docker port can disagree.
- Existing tests under `backend/tests/unit/core/test_settings.py`, `backend/tests/unit/api/test_routes.py`, `backend/tests/unit/desktop/test_desktop_static_contract.py`, `backend/tests/unit/scripts/test_ensure_models_llm_runtime_artifacts.py`, and related local LLM startup tests are the preferred validation homes.

## Out of Scope

- No new settings subsystem or generated settings registry.
- No deletion of compatibility env names without explicit migration tests and fallback behavior.
- No change to model family/quant selection or addition of new model artifacts.
- No Docker service redesign.
- No secrets manager or credential store.
- No broad desktop redesign beyond Operator Settings grouping/visibility.
- No stale desktop/static contract assertions after Operator Settings changes.
- No docs-only workaround for a mismatch that should be resolved in code or tests.

## Target Observable Behavior

Normal users should see a smaller, clearer Operator Settings surface. Primary controls should represent intent, not every low-level override.

Expected posture:

```text
Model: local model enabled, model policy/id, active selection reason
Search: search toggles and effective endpoint state
Services: Redis/SearXNG connection settings only where user-relevant
Advanced: raw model paths, llama.cpp sidecar/binary/base URL overrides, secrets, SDK paths, live-test gates, tuning thresholds
```

`.env.example` should become a safe starter posture, not a complete list of every power-user override. Power-user-only env names can remain supported in `Settings` and Operator Advanced, with comments in `settings.py` or nearby code explaining when to add them manually.

When `USE_LOCAL_MODEL=true`, the system should be able to derive sensible local-model defaults without requiring the user to set redundant low-level toggles. Explicit env values still win.

If a backend endpoint and Docker port can drift, the slice should either derive one from the other or move the non-authoritative value out of primary Operator posture.

Desktop behavior must remain API-driven. If the API advertises primary/advanced/section metadata differently, the desktop must render that metadata correctly and the static contract must assert it.

---

## Vocabulary / Contract Terms

- **primary setting**: a normal user/operator control that expresses intent.
- **derived setting**: a value computed from a primary setting and current repo/catalog/runtime state when no explicit env override exists.
- **advanced override**: a supported env value for power users, diagnostics, or non-default runtime wiring.
- **compatibility alias**: an older env name that remains supported but should not be advertised as normal posture.
- **effective value**: the value actually used after shell env, `.env`, `.env.example`, defaults, and derivation are applied.
- **desktop contract**: the backend API metadata and desktop/static expectations that keep Operator Settings rendering aligned with backend-owned settings.

---

## Design Shape / Contract Sketch

Keep the shape close to existing files:

```text
shell env / .env / .env.example
  -> backend.app.core.settings.Settings
  -> derived/effective settings where explicit env is absent
  -> existing runtime/setup/operator paths
  -> readiness/status/tests expose effective behavior
```

Operator Settings should continue to be API-driven. Field metadata may gain clearer classification, but the desktop should not hard-code a separate settings model.

W.0 env/settings census -> W.1 settings defaults/classification -> W.2 derived local-model behavior -> W.3 direct mismatch fixes -> W.4 Operator Settings posture + desktop contract -> W.5 .env.example reduction -> W.6 closeout validation/docs

### Goal

Make Operator Settings reflect primary intent first and advanced overrides second, and update the desktop/API contract in the same pass.

### Scope

## Sub-Slice W.5 — `.env.example` Posture Reduction

### Goal

Reorganize `.env.example` into a safe starter posture that matches `Settings` defaults and Operator grouping, without pretending it is a complete power-user catalog.

### Scope

- `.env.example` — keep only safe primary starter values plus clearly marked optional placeholders; remove or comment out values that are now derived or advanced-only.
- `backend/tests/unit/core/test_settings.py` — update existing `.env.example` coverage to assert required starter posture, no advertised compatibility aliases, and no missing safe defaults.

- `backend/tests/unit/desktop/test_desktop_static_contract.py` if `.env.example` posture changes the static Operator contract.

- `.env.example` must remain safe when loaded as runtime fallback.

- It must not advertise stale or unused variables as normal posture.
- If `.env.example` and Operator posture are intentionally different, tests must assert the reason and the user-facing behavior.

### Files Not to Modify

- Runtime behavior unless a test exposes a direct mismatch.
- New documentation files.

## Risks and Sequencing Notes

| Risk | Severity | Mitigation |
|---|---|---|
| Removing an env name from `.env.example` breaks fallback behavior | High | Move safe default into `Settings` first and test no-`.env`/no-example behavior. |
| Operator hides a value still needed for normal operation | Medium | Classify primary vs advanced with tests before UI changes. |
| Operator API and desktop/static contract drift apart | High | Require W.4 to update API tests, desktop static contract tests, and npm desktop checks together. |
| `USE_LOCAL_MODEL` derivation accidentally fetches or starts runtime unexpectedly | High | Explicit env wins; tests cover absent/env true/env false cases; fail closed where ambiguous. |
| SearXNG port/base URL drift persists | Medium | Fix direct mismatch in code/config or remove the non-authoritative value from primary posture. |
| Cleanup creates more test/doc sprawl | Medium | Reuse existing tests first; add new files only with clear ownership gap. |

## Finish Line

Slice W is complete when the environment posture is smaller and safer, Settings owns repo-safe defaults and effective derived behavior, Operator Settings presents primary intent before advanced overrides, desktop/API contracts match that posture, direct env mismatches are fixed, existing tests cover the cleanup, and governance records only validated behavior.
