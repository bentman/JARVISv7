# 20260702_slice-z1.md
## Slice Z.1 — Startup Context Cleanup

Status: planned. This document is an implementation contract. It is not completion evidence.

---

## Purpose

Slice Z.1 centralizes the startup facts used by the backend API and operational scripts:

```text
hardware profile -> provisioning extras -> preflight evidence -> family readiness map
```

The current codebase constructs those facts in multiple places. This slice creates one backend-owned helper for that sequence and updates the existing consumers to use it.

This is a cleanup and stabilization slice. It must not add a new product command, new runtime family, new launch workflow, or new validation framework.

---

## Research Basis

Use the established mechanism of a typed startup/context object passed into application and command paths.

Relevant proven patterns:

- FastAPI lifespan/startup guidance favors building shared application resources before request handling and cleaning them up during shutdown, rather than loading heavy state at module import time.
- Click/Typer-style CLI design uses a context/state object to avoid rebuilding or hiding invocation state across command paths.

Apply the pattern without adding Click, Typer, or another CLI dependency. The repo already uses `argparse`; keep it.

---

## Current Codebase Evidence

The duplicated startup-facts sequence is currently present in these locations:

- `scripts/bootstrap.py`
  - Runs `run_profiler()` and `resolve_required_extras(...)` before provisioning.
  - Later imports and runs `run_preflight(...)` plus `derive_*_device_readiness(...)` after provisioning/model setup.
  - Checkpoint order is part of observable behavior and must remain stable.

- `scripts/run_backend.py`
  - Runs `run_profiler()`, `resolve_required_extras(...)`, and `run_preflight(...)` directly.
  - Owns a local readiness summary helper.

- `scripts/run_jarvis.py`
  - Defines its own `StartupContext`.
  - Runs profiler, extras, preflight, readiness derivation, and readiness summary locally.
  - Also owns proving-host engine construction, local LLM sidecar trace handling, and text/voice turn execution. Those remain local to the script.

- `scripts/validate_backend.py`
  - Runs profiler, extras, and preflight directly.
  - Owns a separate readiness summary used for validation fingerprinting.

- `backend/app/api/app.py`
  - Builds API startup state by running profiler, extras, preflight, readiness, personality loading, runtime selection, local LLM preparation, session service, resident voice, and wake monitor wiring.
  - Only the profile/extras/preflight/readiness portion belongs in the shared helper. Runtime construction remains API-owned.

## Target Mechanism

Create a backend-owned helper:

```text
backend/app/services/startup_context.py
```

Recommended public surface:

```python
ReadinessMap = dict[str, tuple[str, bool, str]]

@dataclass(slots=True)
class ProfileContext:
    report: FullCapabilityReport
    profile: HardwareProfile
    extras: list[str]

@dataclass(slots=True)
class StartupContext:
    report: FullCapabilityReport
    profile: HardwareProfile
    extras: list[str]
    preflight: PreflightResult
    readiness: ReadinessMap


def load_profile_context() -> ProfileContext: ...
def complete_startup_context(context: ProfileContext) -> StartupContext: ...
def load_startup_context() -> StartupContext: ...
def derive_readiness_map(preflight: PreflightResult, profile: HardwareProfile) -> ReadinessMap: ...
def readiness_summary(context: StartupContext) -> str: ...
def selected_path_readiness_summary(context: StartupContext) -> str: ...
```

Name changes are allowed only when the replacement is simpler and used consistently.

## Behavior Contract

The shared helper owns only startup facts:

```text
profile
extras
preflight
readiness map
readiness summary text
```

It must not own:

- API app creation
- FastAPI route registration
- session service construction
- resident audio stream construction
- wake monitor construction
- STT/TTS/LLM runtime selection beyond readiness derivation
- local LLM sidecar startup
- proving-host text/voice turn execution
- product doctor behavior
- launch orchestration
- validation tier policy

Runtime construction stays where it is today.

## Acceptance Criteria

Slice Z.1 is complete when:

- `backend/app/services/startup_context.py` is the only place that performs the full profile/extras/preflight/readiness startup-facts sequence.
- `bootstrap.py` still preserves its five checkpoint flow and deferred preflight/readiness import behavior.
- `run_backend.py`, `run_jarvis.py`, `validate_backend.py`, and `backend/app/api/app.py` consume the shared helper instead of duplicating the full sequence.
- API runtime/service construction remains in `backend/app/api/app.py`.
- Proving-host engine construction remains in `scripts/run_jarvis.py`.
- Existing user-visible script output is equivalent or clearer.
- Focused tests cover the shared helper and touched script/API behavior.
