# Slice DD — Toolbox Wiring and Host Filesystem Grants

Status: planned. This document is an implementation contract for a coding agent. It is not completion evidence.

---

## Purpose

Complete the existing tool execution path so the application owns one usable toolbox, existing tools are registered during startup, and every `TurnEngine` receives that toolbox.

Expand the filesystem tool package from one fixed read-only sandbox into user-designated host folder grants. Each grant has a stable identifier, one resolved root, and one access level: `read-only` or `read-write`. Filesystem operations must remain inside the selected grant and enforce the selected access level at execution.

---

## Agent Contract

- Preserve the existing `ToolBase -> ToolRegistry -> ToolExecutor -> TurnEngine` execution path.
- Keep the toolbox explicit and deterministic. Register built-in tools directly; do not add module scanning or a plugin loader.
- Keep management logic thin. The toolbox decides which concrete tools are available; each filesystem operation enforces its own grant and path boundary.
- Preserve existing tool names unless this slice explicitly adds a new filesystem operation.
- Reuse existing startup state, settings, tool tests, and conversation tests.
- Add focused tests with each behavior change and record command evidence before governance updates.

## Out of Scope

- No automatic tool discovery, dynamic imports, marketplace, plugin installation, or third-party tool protocol.
- No new planner or autonomous tool-selection behavior.
- No filesystem delete, rename, move, execute, permission-change, ownership-change, link creation, or binary-file editing.
- No unrestricted absolute-path tool input.
- No access to a folder that is not present in the resolved grant set.
- No desktop folder-picker or broader operator UI in this slice.
- No changes to search provider routing, hardware profiling, memory storage, or agent enablement.

---

## Target Observable Behavior

- Application startup constructs one `ToolRegistry` containing the available built-in tools.
- `ApiState` retains that registry, and every engine created by `build_engine()` or `bind_session()` receives the same toolbox.
- `time`, `hardware.info`, and `search` are invocable through the application-built registry.
- Filesystem tools are registered only when at least one valid user-designated folder grant is configured.
- A filesystem call selects a grant by identifier and supplies a path relative to that grant.
- `read-only` grants permit listing and reading.
- `read-write` grants permit listing, reading, creating directories, creating text files, and explicitly overwriting text files.
- All filesystem operations reject traversal or link/reparse resolution outside the selected grant.
- Tool failures remain explicit and flow through the existing `ToolExecutor` and turn artifact path.

Operator configuration shape:

```text
Grant id: documents
Folder: C:\Users\<operator>\Documents\JARVIS
Access: read-write

Grant id: reference
Folder: D:\Reference
Access: read-only
```

Tool input shape:

```json
{
  "grant": "documents",
  "path": "notes/today.txt"
}
```

## Vocabulary / Contract Terms

- **Toolkit**: the small startup composition function that constructs the application toolbox from existing dependencies and configured filesystem grants.
- **Toolbox**: the concrete `ToolRegistry` retained by application state and supplied to `TurnEngine`.
- **Filesystem grant**: one user-designated host directory, identified by a stable grant id and resolved to a canonical root with one access level.
- **Read-only**: permits `filesystem.list` and `filesystem.read` only.
- **Read-write**: permits read-only operations plus `filesystem.mkdir` and `filesystem.write`.
- **Relative path**: a path interpreted below the selected grant root. Absolute paths are not accepted as tool input.

## Design Shape / Contract Sketch

```text
startup settings + cached hardware profile
  -> build_tool_registry(...)
  -> explicit built-in tool registration
  -> ApiState.tool_registry
  -> build_engine(..., tool_registry=state.tool_registry)
  -> ToolExecutor.execute(...)
  -> ToolRegistry.invoke(...)
  -> concrete tool.run(...)
```

Filesystem boundary:

```text
{grant id, relative path, operation}
  -> resolve configured grant
  -> resolve candidate below canonical grant root
  -> enforce grant access level
  -> perform bounded filesystem operation
  -> deterministic output or explicit error
```

Suggested grant value shape in runtime code:

```python
@dataclass(frozen=True, slots=True)
class FilesystemGrant:
    grant_id: str
    root: Path
    access: Literal["read-only", "read-write"]
```

The exact settings serialization may follow the existing `Settings` conventions, but it must produce a validated `list[FilesystemGrant]` before registry construction. User-specific host paths remain local operator configuration and are not committed as active paths.

## Sub-Slice DD.3 — Host Filesystem Grants

### Goal

Replace the single fixed read sandbox assumption with multiple user-designated host folder grants and operation-specific filesystem tools.

### Scope

Modify or add only the smallest coherent filesystem surfaces, expected to include:

- `backend/app/tools/filesystem/__init__.py`
- `backend/app/tools/filesystem/read_tool.py`
- A shared grant/path resolver under `backend/app/tools/filesystem/`
- Operation-specific `list`, `write`, and `mkdir` tool modules under `backend/app/tools/filesystem/`
- `backend/app/core/settings.py` and its existing tests when settings own grant serialization
- `.env.example` only if a new operator setting must be documented
- `backend/tests/unit/tools/test_filesystem_tool.py`

### Grant resolution

A configured grant must contain:

```text
grant id: non-empty, unique, stable identifier
root: existing host directory resolved to a canonical absolute path
access: exactly read-only or read-write
```

Invalid grant entries must not silently broaden access. Duplicate ids, unsupported access values, missing roots, non-directory roots, or malformed configuration must fail explicitly during grant resolution or startup.

### Shared path boundary

Every filesystem operation must:

1. require a valid grant id;
2. require a non-empty relative path, except listing the grant root may use `.`;
3. reject absolute input paths;
4. combine the path with the canonical grant root;
5. resolve the candidate and verify it remains under that root;
6. reject symlink, junction, or reparse-point traversal that resolves outside the grant;
7. perform the operation only after access-level enforcement.

Windows drive, UNC, reserved-name, and alternate-data-stream inputs must fail closed when they could bypass the relative grant boundary.

### Operations

#### `filesystem.list`

- Allowed for `read-only` and `read-write`.
- Lists one directory inside the selected grant.
- Returns deterministic sorted entries with at least name and type.
- Does not recurse in this slice.
- Rejects missing paths and non-directory targets explicitly.

#### `filesystem.read`

- Allowed for `read-only` and `read-write`.
- Reads UTF-8 text only.
- Preserves the existing `filesystem.read` tool name.
- Rejects missing paths, directories, invalid UTF-8, and paths outside the grant.
- Apply a bounded maximum byte count through an existing setting or a small explicit default covered by tests.

#### `filesystem.mkdir`

- Requires `read-write`.
- Creates exactly the requested directory.
- Parent directory must already exist.
- Existing target returns an explicit error rather than silently succeeding.

#### `filesystem.write`

- Requires `read-write`.
- Writes UTF-8 text only.
- Requires explicit mode:

```text
create     target must not already exist
overwrite  target must already exist and be a regular file
```

- Parent directory must already exist.
- No implicit directory creation.
- Apply a bounded maximum byte count through an existing setting or a small explicit default covered by tests.
- A failed write must not leave a partially replaced destination. Use the narrowest existing atomic-replacement pattern available in the standard library when overwriting.

### Settings behavior

- User-specific active paths must come from local operator configuration.
- Settings must normalize raw configuration into validated grants before registry construction.
- Empty grant configuration is valid and produces no filesystem tools.
- The selected access level is per grant, not global.
- Do not expose repository data, memory, session, turn, model, cache, or other folders unless the operator explicitly grants that exact root.

### Required tests

Extend `backend/tests/unit/tools/test_filesystem_tool.py` to cover:

- multiple grants selected by id;
- read-only list/read success;
- read-only write/mkdir rejection;
- read-write create, overwrite, read, list, and mkdir success;
- create refusing an existing file;
- overwrite refusing a missing file;
- missing and duplicate grant ids;
- relative traversal rejection;
- absolute path rejection;
- sibling directory rejection;
- symlink/junction/reparse escape rejection where supported by the test host;
- invalid UTF-8 read failure;
- read and write size bounds;
- deterministic listing order;
- no filesystem tool registration when no grants exist.
