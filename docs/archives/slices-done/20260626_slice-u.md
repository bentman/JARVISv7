# Slice U - Implementation Guide: ARM64 QNN Dependency Alignment

## Objective

Align the Windows ARM64 Qualcomm QNN provisioning path with the current upstream package model while preserving all AMD64 selection and provisioning behavior. This is an implementation guide for Codex Agent running on Windows ARM64.

## Governing Constraints

- Use `backend/.venv/Scripts/python` for all Python commands from the repo root.
- Do not use global Python packages.
- Do not run `pip install` directly. Use `scripts/provision.py`.
- Treat AMD64 behavior as out of scope except for validation that it remains unchanged.
- Do not change QNN runtime/session logic in this slice.
- Do not change NumPy, sounddevice, audio warning handling, or live audio behavior in this slice.
- Do not update `CHANGE_LOG.md` or `SYSTEM_INVENTORY.md` until implementation has validation evidence.

## Upstream Evidence

- PyPI currently reports `onnxruntime-qnn` latest version `2.3.0`, `requires_python >=3.11`, and Windows ARM64 wheels for CPython 3.11, 3.12, 3.13, and 3.14:
  - `https://pypi.org/project/onnxruntime-qnn/`
  - Verified by PyPI JSON query on 2026-06-26.
- The `onnxruntime-qnn` `v2.3.0` GitHub release says:
  - ONNX Runtime compatibility: `>= 1.24.1`
  - compiled with ONNX Runtime `v1.24.4`
  - QAIRT SDK compatibility: `2.47.0`
  - install example installs both `onnxruntime==1.24.4` and `onnxruntime-qnn==2.3.0`
  - `https://github.com/onnxruntime/onnxruntime-qnn/releases/tag/v2.3.0`

Repo nuance: `pyproject.toml` currently declares `requires-python = ">=3.11,<3.14"`. Do not expand repo Python support in this slice.

## Current Repo State To Change

The current repo still encodes the older workaround:

- `pyproject.toml`
  - `hw-npu-qualcomm-qnn` pins `onnxruntime-qnn==1.24.3`.
- `backend/app/hardware/provisioning.py`
  - `_EXTRA_REQUIREMENT_SPECS["hw-npu-qualcomm-qnn"]` pins `onnxruntime-qnn==1.24.3`.
- `scripts/provision.py`
  - `_run_install()` uninstalls `onnxruntime` on ARM64 QNN hosts.
  - `_run_install()` force-reinstalls `onnxruntime-qnn==1.24.3` with `--no-deps`.
  - `_run_verify()` rejects paired `onnxruntime` and `onnxruntime_qnn`.
- `backend/tests/unit/hardware/test_provisioning.py`
  - expects QNN requirement names to exclude `onnxruntime`.
  - expects `onnxruntime-qnn==1.24.3`.
- `backend/tests/unit/scripts/test_provision_script.py`
  - expects uninstall plus force reinstall plus `--no-deps`.
  - expects exact QNN version drift behavior.
  - expects paired `onnxruntime` and `onnxruntime_qnn` to be rejected.
- `backend/tests/unit/hardware/test_preflight.py`
  - has assertion messages and fixtures that still say `ORT 1.24.x QNN` / `1.24.3`.

## Required Implementation

### 1. Update ARM64 QNN Dependency Metadata

In `pyproject.toml`, change only the `hw-npu-qualcomm-qnn` extra.

Target:

```toml
hw-npu-qualcomm-qnn = [
  "onnxruntime>=1.24.4; platform_machine=='ARM64' and sys_platform=='win32'",
  "onnxruntime-qnn>=2.3.0; platform_machine=='ARM64' and sys_platform=='win32'",
  "onnx>=1.16; platform_machine=='ARM64' and sys_platform=='win32'",
  "transformers>=4.40; platform_machine=='ARM64' and sys_platform=='win32'",
]
```

Also update the nearby comment to reflect current 2.3.0 Windows ARM64 wheel support. Do not change AMD64 extras.

### 2. Update Resolver Requirement Specs

In `backend/app/hardware/provisioning.py`, update only `_EXTRA_REQUIREMENT_SPECS["hw-npu-qualcomm-qnn"]`.

Target:

```python
"hw-npu-qualcomm-qnn": (
    "onnxruntime>=1.24.4",
    "onnxruntime-qnn>=2.3.0",
    "onnx>=1.16",
    "transformers>=4.40",
),
```

Do not change `resolve_required_extras()` ordering unless a test proves it is required. The existing resolver should still select `hw-npu-qualcomm-qnn` for ARM64 Qualcomm NPU and should still omit `hw-arm64-ort-cpu` when that accelerated ORT path is selected.

### 3. Remove ARM64 QNN Post-Install Workaround

In `scripts/provision.py`, remove only the ARM64 QNN post-install branch that:

- uninstalls `onnxruntime`
- force-reinstalls `onnxruntime-qnn==1.24.3`
- passes `--no-deps`

The normal editable install should be the only ARM64 QNN install action:

```text
backend\.venv\Scripts\python -m pip install -e .[hw-cpu-base,hw-arm64-base,hw-npu-qualcomm-qnn,dev]
```

Do not remove or modify the AMD64 CUDA post-install handling in this slice. It is separate behavior:

```python
onnxruntime-gpu>=1.17
```

### 4. Remove ARM64 QNN Verify Conflict Rule

In `scripts/provision.py`, remove the conflict that rejects:

```python
{"onnxruntime", "onnxruntime_qnn"} <= installed_requirements
```

Paired `onnxruntime` and `onnxruntime_qnn` is expected for the new ARM64 QNN path.

Keep exact-version mismatch logic for any remaining exact pins elsewhere. With QNN moved to `>=`, verify should not demand an exact QNN version.

### 5. Update Focused Tests

Update `backend/tests/unit/hardware/test_provisioning.py`:

- QNN requirement names should include both `onnxruntime` and `onnxruntime-qnn`.
- QNN specs should include:
  - `onnxruntime>=1.24.4`
  - `onnxruntime-qnn>=2.3.0`
  - `onnx>=1.16`
  - `transformers>=4.40`
- Keep the existing assertion that ARM64 Qualcomm QNN omits `hw-arm64-ort-cpu`.

Update `backend/tests/unit/scripts/test_provision_script.py`:

- Replace the QNN reinstall test with one proving `_run_install()` runs only the normal editable install for ARM64 QNN.
- Replace or remove the exact QNN version drift test. QNN is no longer exact-pinned, so verify should not report drift for `onnxruntime_qnn` version differences when expected specs are `>=`.
- Invert the paired-package conflict test: verify should pass when both `onnxruntime` and `onnxruntime_qnn` are installed and expected.
- Keep tests for exact pins if they are testing generic exact-pin behavior using another exact requirement. Do not keep QNN as the exact-pin example.

Update `backend/tests/unit/hardware/test_preflight.py` only where existing test text is stale:

- Replace assertion messages that say `onnxruntime_qnn must not be imported for ORT 1.24.x QNN`.
- Update mocked `onnxruntime-qnn` versions from `1.24.3` to `2.3.0` where the version value is meant to represent the current QNN distribution.
- Do not introduce an import of `onnxruntime_qnn` unless the production preflight code changes. Current production preflight imports `onnxruntime`, checks the `onnxruntime-qnn` distribution, probes providers from `onnxruntime`, and discovers packaged `QnnHtp.dll`.

## Adjacent Code To Leave Alone

Do not change these unless validation proves a direct need:

- `backend/app/hardware/preflight.py`
  - Already imports `onnxruntime` for provider probing.
  - Already checks the `onnxruntime-qnn` distribution separately.
  - Already discovers packaged `QnnHtp.dll`.
- `backend/app/hardware/readiness.py`
  - Already selects QNN STT only when `import:onnxruntime-qnn`, `ep:QNNExecutionProvider`, and `dll:QnnHtp` are present.
- `backend/app/hardware/qnn_provider.py`
  - Already passes `backend_path` provider options.
  - Already disables CPU EP fallback when requested.
- `backend/app/runtimes/stt/onnx_whisper_runtime.py`
  - Already uses `create_qnn_session()` for QNN Whisper sessions and verifies `QNNExecutionProvider` is primary.
- `config/models/llm.yaml`
  - QNN/Hexagon LLM profile states are independent of this Python dependency alignment.
- AMD64 resolver/provisioning behavior.
  - In particular, do not change `hw-gpu-nvidia-cuda`, AMD64 CUDA selection, or CUDA post-install handling.

## Gotchas Found In Adjacent Tests

- `backend/tests/unit/hardware/test_provisioning.py` encodes both the old missing-`onnxruntime` expectation and the old exact QNN pin.
- `backend/tests/unit/scripts/test_provision_script.py` encodes the old uninstall, `--force-reinstall`, `--no-deps`, exact version, and package conflict assumptions.
- `backend/tests/unit/hardware/test_preflight.py` has stale `ORT 1.24.x QNN` assertion messages and `1.24.3` mocked version values.
- `backend/tests/unit/hardware/test_qnn_prerequisite.py` references `onnxruntime_qnn` in fake imports, but current production preflight does not import that module. Check before editing; this file may not need changes.
- `scripts/validate_backend.py` treats QNN probe errors as readiness-degrading when `stt_device == "qnn"` and probe error keys start with `onnxruntime.qnn` or equal `onnxruntime-qnn`. This remains compatible with the intended package alignment.

## Validation Plan For ARM64 Agent

Run from repo root on Windows ARM64.

1. Focused provisioning tests:

```powershell

```

2. Focused preflight tests if `test_preflight.py` is edited:

```powershell
backend\.venv\Scripts\python -m pytest backend\tests\unit\hardware\test_preflight.py backend\tests\unit\hardware\test_qnn_prerequisite.py backend\tests\unit\hardware\test_qnn_slot.py -q
```

3. Provision dry-run must show ARM64 QNN editable install with no post-install QNN workaround:

```powershell
backend\.venv\Scripts\python scripts\provision.py dry-run
```

Expected ARM64 Qualcomm QNN extras:

```text
hw-cpu-base,hw-arm64-base,hw-npu-qualcomm-qnn,dev
```

4. Install dry-run:

```powershell
backend\.venv\Scripts\python scripts\provision.py install --dry-run
```

```text
backend\.venv\Scripts\python.exe -m pip install -e .[hw-cpu-base,hw-arm64-base,hw-npu-qualcomm-qnn,dev]
```

5. Provision verify:

```powershell
backend\.venv\Scripts\python scripts\provision.py verify
```

- no conflict for paired `onnxruntime` and `onnxruntime_qnn`

6. Standard validation:

```powershell
backend\.venv\Scripts\python scripts\validate_backend.py unit
```


7. ARM64 QNN readiness/profile validation:


8. ARM64 live STT QNN proof only after the above is green and live gates/prerequisites are available:

```powershell

```
