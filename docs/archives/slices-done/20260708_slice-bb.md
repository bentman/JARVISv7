## Slice BB — Semantic Memory and Hybrid Retrieval

> Status: planned.  
> This document is an implementation contract for a coding agent.  
> It is not completion evidence.

---

## Purpose

Slice BB adds a local-first semantic memory path that improves cross-session recall without adding a default vector database service or hidden assistant state. The system should persist durable semantic memory records locally, retrieve relevant prior context through a hybrid lexical/vector path, and inject retrieved context through the existing untrusted retrieval segment of the prompt envelope.

This slice must preserve the validated voice/text turn loop, episodic memory behavior, retrieval prompt boundary, Redis-as-cache-only rule, and desktop/runtime surfaces. Semantic recall is additive over existing turn artifacts; it must not replace the conversation engine, prompt assembler, episodic memory, Redis cache, or runtime-selection architecture.

## Vocabulary / Contract Terms

- **Semantic memory**: durable cross-session memory records intended for meaning-based retrieval. It is not hidden state and not a replacement for turn artifacts.
- **Semantic record**: one stored memory item with text, source references, metadata, vector metadata, and vector bytes.
- **Vector**: a numeric representation of text produced by the deterministic CPU text vectorizer in this slice.
- **Hybrid retrieval**: combining lexical retrieval and vector similarity retrieval, then merging results deterministically.
- **Durable authority**: the persisted local store from which semantic memory can be reconstructed. For this slice it is `memory.sqlite` under `data/memory/semantic`.
- **Cache**: optional acceleration only. Cache content may be discarded without losing memory.
- **FTS**: SQLite full-text search, preferably FTS5 when available. FTS improves lexical matching but is not the durable authority by itself.
- **RRF**: Reciprocal Rank Fusion. A deterministic rank merge option for combining lexical and vector candidate lists.
- **Semantic consolidation**: closeout writeback that turns session/turn artifacts into semantic records under explicit policy.
- **Trusted context**: application/personality/session control context. Semantic memory is never trusted context.
- **Untrusted retrieval context**: prior memory or tool-derived content that can inform a response but cannot instruct the assistant.

## Design Shape / Contract Sketch

Storage and retrieval shape:

```text
data/turns + data/sessions + data/memory/episodic
  -> semantic consolidation policy
  -> data/memory/semantic/memory.sqlite
  -> SemanticMemory lexical/vector retrieval
  -> RetrievalManager merge with episodic facts
  -> PromptEnvelope retrieval segment, trusted=False
  -> TurnArtifact.retrieved_memory_refs
```

Initial SQLite shape:

```text
semantic_meta
  key TEXT PRIMARY KEY
  value TEXT NOT NULL

semantic_fact
  fact_id TEXT PRIMARY KEY
  text TEXT NOT NULL
  source_session_id TEXT
  source_turn_id TEXT
  source_field TEXT
  created_at TEXT NOT NULL
  updated_at TEXT NOT NULL
  kind TEXT NOT NULL DEFAULT 'fact'
  confidence REAL
  metadata_json TEXT NOT NULL DEFAULT '{}'
  vectorizer_id TEXT NOT NULL
  vector_dim INTEGER NOT NULL
  vector_blob BLOB NOT NULL
  text_hash TEXT NOT NULL

semantic_fact_fts
  FTS5 virtual table over fact text when available
```

Rules:

- Store vectors as little-endian float32 BLOBs or a documented equivalent with tests for round-trip and dimension checks.
- Use SQLite transactions for writes.
- Use stable schema versioning through `semantic_meta`.
- Deduplicate by normalized text hash and source reference where practical.
- Do not expose SQL rows directly to prompt assembly. Convert to `RetrievedFact`-compatible objects.
- If implementation diverges materially from this sketch, update this slice doc before continuing.

## Design Decision

The default store for JARVIS semantic memory is SQLite, not raw JSONL and not a service-backed vector database.

Rationale:

- SQLite is local, embedded, zero-service, cross-platform, and consistent with a single-file durable memory authority.
- SQLite supports schema versioning, transactions, metadata filtering, deletion, compaction, and inspection better than raw JSONL.
- SQLite FTS gives a lightweight lexical retrieval path without external services.
- CPU cosine similarity over SQLite-stored vectors is sufficient for the first semantic memory scale and keeps this slice dependency-free.
- JSONL may be added later as export/audit output only if separately approved; it is not the primary query store.
- Chroma, Qdrant, LanceDB, Cosmos DB, Redis Stack vector search, and similar systems are not default-path dependencies for this local-first assistant.
