# 20260704_slice-z6.md
## Slice Z.6 — Desktop Backend Diagnostics

> Status: planned.  
> This document is an implementation contract.  
> It is not completion evidence.

---

## Purpose

Slice Z.6 makes backend launch failures actionable from the desktop UI.

When the desktop cannot start or reach the backend, the product surface should show the concrete startup facts needed to diagnose the failure instead of reducing the problem to a generic backend-unavailable message or raw Tauri error string.

The target question is:

```text
Which Python executable, backend script, working directory, endpoint, log paths, and recent log output were involved in the failed backend startup?
```

---

## Current Codebase Evidence

The repo already has most backend startup facts in the Tauri backend manager:

- `desktop/src-tauri/src/backend.rs`
  - `BackendDiagnostics` already contains `python_path`, `backend_script_path`, `working_directory`, `host`, `port`, `stdout_log`, and `stderr_log`.
  - `BackendProcessManager::diagnostics()` already derives those fields from the desktop repo root and backend lifecycle settings.
  - `BackendProcessManager::new()` writes startup logs to `reports/backend_startup.log` and `reports/backend_spawn_stderr.log`.
  - `spawn_backend()` already includes formatted diagnostics for missing Python, missing backend script, spawn failure, and backend exit during startup.
  - Early backend exit already includes stdout and stderr tails from the startup log files.
  - `format_diagnostics(...)` and `tail_file(...)` already provide line-oriented diagnostic text.

- `desktop/src-tauri/src/lib.rs`
  - `start_backend` currently returns JSON with `diagnostics` on success.
  - Health-wait failures append `startup diagnostics` as pretty JSON to the error string.
  - Direct `spawn_backend()` failures still travel through the Tauri command error channel as a string.

- `desktop/src/api-client.js`
  - `startBackend` currently parses success JSON from `invoke("start_backend")`.
  - Failed invocations are not normalized into a structured frontend diagnostic payload.

- `desktop/src/main.js`
  - `startDesktop()` catches backend startup errors and calls `showError(String(error), "BACKEND_UNAVAILABLE")`.
  - The current error surface is a single text panel and does not render fields or log tails as a dedicated backend diagnostics display.

- `desktop/src/index.html`
  - The Conversation panel already has a collapsed `Conversation debug details` block.
  - The Conversation panel also has a collapsed `Degraded list detail` block, but that block is hidden when there are no degraded readiness rows.

- `desktop/src/components/conversation-debug.js`
  - The conversation debug renderer already uses compact line-oriented text and omits empty values.
  - This is the interaction style to mimic for backend startup diagnostics.

- `desktop/src/components/degraded-list.js`
  - The degraded list renderer controls its surrounding `<details>` visibility from readiness data.
  - Backend startup diagnostics should not depend on degraded readiness data because readiness may be unavailable when backend startup fails.

- `desktop/tests/static.test.mjs` and `backend/tests/unit/desktop/test_desktop_static_contract.py`
  - Desktop static contract tests already protect DOM placement, Tauri command exposure, API client wiring, and diagnostic/debug labels.
  - Z.6 should extend adjacent static coverage instead of adding broad test sprawl.

## Target Outcome

Backend startup failure details are visible and copyable in the desktop UI.

The desktop should display:

```text
python_path: <repo>\backend\.venv\Scripts\python.exe
backend_script_path: <repo>\scripts\run_backend.py
working_directory: <repo>
endpoint: 127.0.0.1:8765
stdout_log: <repo>\reports\backend_startup.log
stderr_log: <repo>\reports\backend_spawn_stderr.log
stdout_tail: <recent backend stdout lines when available>
stderr_tail: <recent backend stderr lines when available>
failure: <startup failure summary>
```

Field names may follow existing JavaScript naming conventions, but the user-visible text must remain direct and copyable. Empty or unavailable tails should be omitted or shown as unavailable without hiding the path fields.

---

## UI Placement and Behavior

Add a collapsed backend diagnostics panel in the Conversation panel.

The panel should mimic the current `Conversation debug details` interaction pattern:

- use a `<details>` block;
- use a concise `<summary>` label, for example `Backend diagnostics`;
- render line-oriented diagnostic text into a `<pre>` or equivalent copyable text surface;
- omit empty fields;
- stay compact and avoid duplicating the visible error message beyond a short `failure:` line.

Placement must be:

```text
Conversation debug details
Backend diagnostics
Degraded list detail
```

This places backend diagnostics above `Degraded list detail`, because the degraded readiness detail is conditional and may not exist during backend startup failure.

The backend diagnostics panel should appear when backend startup fails or when startup diagnostics are otherwise useful. It should not turn into a general readiness or conversation transcript surface.

---

## Failure Contract

The implementation should preserve the current backend lifecycle behavior while improving the shape delivered to the UI.

Backend launch failure cases to preserve and surface:

- repo-local Python executable missing;
- backend script missing;
- backend process spawn failure;
- backend process exits during startup;
- backend starts but health check never passes before timeout.

For each case, the desktop should show the best available diagnostics fields and log paths. When stdout or stderr tail output is available, show it in the backend diagnostics panel. When tail output is unavailable, keep the log paths visible.

The desktop may continue to show a short error summary in `#error-panel`, but the detailed startup facts belong in the backend diagnostics panel.

---

## Out of Scope

Do not redesign backend startup.

Do not change backend runtime selection, readiness policy, LLM/STT/TTS runtime behavior, session behavior, personality behavior, Redis behavior, or conversation debug semantics.

Do not convert this into a broad desktop layout redesign.

Do not use backend diagnostics as a replacement for `Conversation debug details` or `Degraded list detail`.

Do not update `CHANGE_LOG.md`, `SYSTEM_INVENTORY.md`, or other bookkeeping until changes are validated by User/Operator and that update is explicitly approved.

---

## Suggested Implementation Shape

Prefer the existing startup truth path.

The current Rust backend manager already owns the startup facts and log paths. The missing outcome is structured desktop presentation. Keep Rust changes limited to exposing or normalizing those facts for failure cases when needed.

The frontend should treat backend startup diagnostics as structured data where possible and fall back to parsing/displaying the existing string only when no structured object is available.

A small component such as `desktop/src/components/backend-diagnostics.js` is acceptable if it reduces main-file churn and mirrors the existing conversation debug renderer style.

---

## Testing Guidance

Use adjacent desktop static tests.

Tests should prove:

- the desktop DOM contains a backend diagnostics details block;
- the backend diagnostics block is placed after `Conversation debug details` and before `Degraded list detail`;
- the API/client/startup path preserves diagnostics fields from backend startup;
- startup failures are not collapsed only into a generic `String(error)` display;
- the renderer includes python path, backend script path, working directory, endpoint, stdout log path, stderr log path, and available tail output;
- existing `Conversation debug details` and `Degraded list detail` behavior remains intact.

Avoid broad test-suite expansion. Add focused assertions to existing desktop static contract locations unless a small adjacent component test is clearly cleaner.

---

## Acceptance Criteria

Z.6 is complete when:

1. Backend startup failures in the desktop show python path, backend script path, working directory, endpoint, stdout log path, stderr log path, and useful stdout/stderr tail output when available.
2. The backend diagnostics panel is a collapsed Conversation-panel detail that visually follows the `Conversation debug details` pattern.
3. The panel appears above `Degraded list detail` and does not depend on readiness being available.
4. The short error panel still summarizes failure state, but detailed startup facts are available separately and copyably.
5. Existing backend startup behavior, readiness behavior, session behavior, and conversation debug behavior are preserved.
6. Focused desktop/static tests pass.
7. Unit validation passes.
8. No changelog or inventory entry is added until User/Operator validation is complete and explicitly approved.
