## Group R Boundary From `slices.md`

Group R exists because local LLM service work has resource and build-environment constraints distinct from voice acceleration. The group owns ARM64 memory limits, model quantization choices, and local LLM server binary availability.

R.0 is a non-mutating census. Its acceptance is a census table with close states for both host classes and no code changes.

R.1 is the later implementation slice. It depends on R.0 closing as viable on at least one host class. R.1 is where `LlamaCppLLM.is_available()` and `generate()` would be activated, local endpoint settings would be modeled, selector/readiness would consume real probe evidence, and Ollama would remain fallback.

## Existing Governance Shape

The governing pattern is already established:

- Hardware-dependent runtime selection starts in the profiler and readiness evidence, not inside runtime modules.
- Config files define catalog/default metadata.
- Runtime modules execute selected runtime/model/device; they do not invent hardware policy locally.
- `pyproject.toml` plus `backend/app/hardware/provisioning.py` are the provisioning authority.
- `backend/app/hardware/preflight.py` owns import/provider/DLL probe evidence.
- `backend/app/routing/runtime_selector.py` owns LLM/search escalation policy.
- `config/models/*.yaml` is the model catalog family surface.
- `models/` stores model artifacts only.
- `scripts/ensure_models.py` is the existing model acquisition/verification script.
- `scripts/run_jarvis.py` remains a proving host, not a durable product surface.

## Current Profile Shape

`backend/app/core/capabilities.py` defines `HardwareProfile` with:

- OS name/version
- architecture
- CPU name, physical/logical cores, max frequency
- total and available memory
- GPU presence, name, vendor, VRAM, VRAM source
- CUDA availability and version
- NPU presence, vendor, TOPS
- device class
- profile id
- timestamp

`backend/app/hardware/profiler.py` builds a stable profile id from the normalized profile and derives `CapabilityFlags`.

LLM-relevant existing flags:

- `supports_local_llm`
- `supports_gpu_llm`
- `supports_cuda_llm`
- `requires_degraded_mode`
- `qnn_available`
- `directml_candidate`

Current `supports_local_llm` is profile-derived from known platform, CPU core count, and memory total. Current GPU/CUDA LLM flags are capability flags only; LLM device acceleration is not wired as a verified runtime.

Shape implication for Group R census:

- R.0 should identify whether local LLM support is an external operator binary, an existing service, or a Python/runtime package concern before any provisioning change is proposed.

## Current Preflight And Readiness Shape

`backend/app/hardware/preflight.py` currently records:

- import tokens from resolved extras
- execution-provider tokens from `onnxruntime.get_available_providers()`
- DLL discovery for CUDA and QNN/QAIRT paths
- QNN provider/distribution/path evidence
- probe errors keyed by probe area

`backend/app/hardware/readiness.py` derives readiness tuples:

- STT: selects QNN, CUDA, DirectML, or CPU from profile plus preflight tokens.
- TTS: currently CPU with explicit provider-override limitations.
- Wake: CPU from `openwakeword` import evidence.
- LLM: always returns `("cpu", False, "local runtime unavailable")`.

Shape implication for Group R census:

- LLM readiness already has a named family slot.
- The missing piece is not a new readiness architecture; it is repo-grounded evidence for local LLM availability that can feed the existing LLM readiness tuple.
- Probe evidence should remain preflight/readiness-owned, not embedded in `LlamaCppLLM` as host policy.

## Current Model Catalog Shape

- `get_model_entry(family, model_name)` resolves default model or explicit model.
- `ModelEntry.local_path` resolves relative paths under repo root.

LLM catalog gap:

- It has no `default_model`, no `models` mapping, no per-model `runtime`, no `devices`, no quantization metadata, no artifact source metadata, and no required local files.
- `scripts/ensure_models.py` treats `llm` specially as `ollama_manages_models` instead of using `ModelEntry` verification.

Shape implication for Group R census:

- STT/TTS already provide the catalog pattern for local artifacts.
- Local LLM model artifacts can be inventoried against that existing pattern without inventing a separate manifest format.
- R.0 should distinguish Ollama-managed remote-to-Ollama model state from repo-managed `models/llm/` artifacts.

## Current Runtime And Selector Shape

`backend/app/runtimes/llm/base.py` defines the common LLM interface:

- `generate(prompt, **kwargs)`
- `generate_envelope(envelope, **kwargs)`
- `is_available()`
- `runtime_name()`

`backend/app/runtimes/llm/local_runtime.py` currently defines `LlamaCppLLM`:

- `is_available()` returns `False`.
- `generate()` raises `NotImplementedError`.
- `runtime_name()` returns `llama.cpp`.

`backend/app/runtimes/llm/ollama_runtime.py` currently defines `OllamaLLM`:

- reads settings through `load_settings()`
- defaults to `http://127.0.0.1:11434`
- defaults model to `phi4-mini`
- supports `OLLAMA_NUM_CTX`
- probes `/api/tags`
- generates through `/api/generate`
- fails on empty generation

`backend/app/routing/runtime_selector.py` currently:

- constructs `LlamaCppLLM()`
- selects local only if `local.is_available()`
- selects Ollama if reachable
- then tries policy-gated cloud runtimes
- returns `NullLLMRuntime` when none are available
- returns `SelectionTrace(runtime_name, reason)`

Current selector behavior is already ordered as local -> Ollama -> cloud/null, but local is inert because `LlamaCppLLM` is always unavailable.

Shape implication for Group R census:

- R.1 does not need a second LLM routing owner.
- The existing selector already names the policy point where local viability would surface.
- R.0 should collect the evidence needed to decide what `local` means in the existing selector: repo-managed artifact, external server endpoint, or existing Ollama service.

Shape implication for Group R census:

- R.0 should identify which concrete probe facts would be needed before `BackendReadiness.llm_local_ready` or equivalent readiness state can be truthfully populated in R.1.

## Repo-Grounded Findings

1. The local LLM routing slot already exists.

`select_llm()` already tries `LlamaCppLLM` before Ollama. The current blocker is evidence and implementation of local availability, not lack of a selector path.

2. The local LLM runtime class exists only as a boundary marker.

`LlamaCppLLM` is intentionally unavailable and raises `NotImplementedError`, matching inventory/changelog statements that llama.cpp is not wired as a verified runtime.

3. Ollama is the verified fallback surface.

`OllamaLLM` has HTTP probe/generate behavior, settings, unit tests, live-gated runtime tests, and CHANGE_LOG evidence. Its model state is server-managed, not repo artifact-managed.

4. The model catalog pattern exists but is not yet applied to LLM artifacts.

STT/TTS use `default_model`, `models`, `runtime`, `devices`, `source`, and `local_path`. LLM currently has endpoint/path config only.

5. The probe/readiness pattern exists but LLM has no local proof tokens.

Preflight already records import/provider/DLL evidence for voice runtimes and QNN. LLM readiness is a placeholder that always marks local runtime unavailable.

6. Operator knobs exist but are not enough to define local LLM behavior.

The repo exposes `LLAMA_CPP_MODEL_PATH`, `USE_LOCAL_MODEL`, and `LOCAL_MODEL_FETCH`, but no endpoint/probe/model-entry shape is currently tied to `LlamaCppLLM`.

7. Avoiding a parallel architecture means reusing the current chain.

The repo shape points to this chain for any future R.1 work: profile facts -> provisioning if needed -> preflight/readiness evidence -> model catalog/artifact path if repo-managed -> `LlamaCppLLM` through `LLMBase` -> `select_llm()` -> startup state -> `/readiness`.

## Minimum Evidence Needed Before Implementation Scope

For Windows x64:

- local server/runtime availability state
- model artifact format and quantization option
- endpoint or path ownership decision
- memory and context parameters
- fallback proof that Ollama remains available or degrades cleanly

For Windows ARM64:

- same as x64
- prebuilt binary availability or source-build-only classification
- observed or documented memory pressure for selected quantization/context
- close state: `PASS-census-viable`, `Degraded-memory-constrained`, `SKIP-no-viable-binary`, or `SKIP-build-toolchain`

For repo architecture:

- whether LLM artifacts should adopt `ModelEntry` shape
- whether `LLAMA_CPP_MODEL_PATH` is a model file, model directory, or server data root
- whether a local endpoint variable is needed later, and if so whether it mirrors the Ollama endpoint pattern
- which readiness token names would be evidence, not policy

## Census Conclusion

The existing repo already has the correct ownership chain for Group R:

`HardwareProfile` and `CapabilityFlags` describe host constraints; provisioning owns packages; preflight/readiness owns evidence; `backend/app/models/catalog.py` owns repo-managed model artifact metadata; `LLMBase` owns the runtime interface; `runtime_selector.select_llm()` owns local -> Ollama -> cloud/null selection; startup and readiness consume selected runtime state.

The current gap is not architectural. The gap is evidence and catalog shape for local LLM artifacts/server viability. R.0 should collect external model/binary/memory facts and map them into the existing chain before any R.1 implementation is proposed.
