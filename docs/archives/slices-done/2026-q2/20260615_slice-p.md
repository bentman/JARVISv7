# 20260615_slice-p.md
## Slice P — Agent Framework Correction (Agent System)

**Date:** 2026-06-15
**Status:** Proposed
**Depends on:** Group O implementation and validation recorded in `SYSTEM_INVENTORY.md` and `CHANGE_LOG.md`.

---

## Purpose

Correct the Group O agent framework before any agent runtime behavior expands.

Group O created useful boundary infrastructure: typed agent messages, a local agent ledger, disabled-by-default policy, truthful `/agents/status`, dry-run role helpers, and read-only trace diagnostics. It also introduced the wrong architectural pressure: Planner, Executor, Critic, Curator, and Learner became hardcoded framework-level role concepts through Python literals, `VALID_AGENT_ROLES`, and default role configuration.

Slice P changes the agent system to be **spec-first**:

- one `JarvisAgentSpec` schema defines JARVIS agents
- agent IDs come from validated specs, not Python `Literal` role sets
- `agent_creator` is the first spec-defined agent
- Agent Creator creates and validates agent specs only
- Agent Creator is then used to create and validate the ProjectVision-derived JARVIS agents
- planner, executor, critic, curator, and learner become disabled specs, not privileged framework concepts
- existing Group O policy, ledger, message, status, and trace surfaces remain valid

This slice is a correction. It does not add autonomous agents, background execution, tool-using agents, semantic memory, model routing, desktop UI, or normal conversation changes.

---

## ProjectVision Alignment

`ProjectVision.md` defines the Agent Layer as responsible for:

- role-separated task decomposition, execution, and validation
- typed, non-conversational inter-agent messaging
- persisted coordination through an agent ledger
- training-data curation from successful execution traces
- training-cycle orchestration with regression-gated deployment

It also states that the agent layer is optional, composes the conversation engine, tool registry, memory, and LLM runtime, does not replace the turn engine, leaves non-agent turns unchanged, and must be policy-gated and traceable.

Slice P keeps those target agents, but makes them data-defined specs instead of framework literals.

## External Reference Points

These are research references, not dependencies and not imported architecture:

- CrewAI documents YAML configuration as the recommended way to define agents because it is cleaner and more maintainable than direct code definitions. That supports JARVIS using config-backed specs rather than Python role literals. Reference: [CrewAI Agents documentation](https://docs.crewai.com/en/concepts/agents).
- CrewAI identifies role, goal, and backstory as critical agent-shaping parameters, and also separates tools, memory/context, and templates. JARVIS should adapt the pattern as `spec_id`, purpose, prompt reference, allowed tools, memory policy, and output contract, not copy CrewAI runtime behavior.
- OpenAI Agents SDK describes agents as LLMs configured with instructions, tools, optional handoffs, guardrails, and structured outputs. That supports a spec schema with instructions/prompt references, allowed tools, handoff policy, guardrail policy, and output contract. Reference: [OpenAI Agents SDK agents documentation](https://openai.github.io/openai-agents-python/agents/).
- OpenAI’s docs also distinguish manager-style orchestration from handoffs. JARVIS should keep those as future policy/spec fields, not hardcoded role behavior.
- AutoGen warns that a general assistant agent is a broad prototyping abstraction and points users toward custom agents once design choices are understood. It also documents structured output using schemas and tool iteration controls. JARVIS should favor explicit narrow specs and bounded tool policy over a general kitchen-sink agent. Reference: [AutoGen AgentChat agents documentation](https://microsoft.github.io/autogen/stable/user-guide/agentchat-user-guide/tutorial/agents.html).

Research conclusion: JARVIS should own a small, local-first `JarvisAgentSpec` that borrows accepted concepts such as role/purpose, prompt/instructions, tools, handoffs, guardrails, structured outputs, and lifecycle/trace events, while preserving JARVIS-specific constraints: disabled-by-default, no hidden execution, deterministic policy, explicit ledger, and no normal-turn behavior changes.

## Current Design Gap

The current Group O implementation makes these roles framework concepts:

- planner
- executor
- critic
- curator
- learner

This is wrong for the target JARVIS agent system. Roles must be specs that can be created, validated, enabled, disabled, listed, and audited. Adding a new JARVIS agent must not require editing Python role literals.

The correction is to keep framework code generic and let the repository carry default agent specs as data.

---

## Out of Scope

Slice P does not implement:

- autonomous agents
- background execution
- tool-using agents
- model-side function calling
- semantic/vector memory
- model routing
- training or adapter deployment
- desktop UI changes
- normal conversation changes
- wake/STT/TTS/LLM runtime changes
- provisioning or dependency changes

Agent Creator creates and validates specs only. It does not create Python code, auto-enable specs, execute specs, or call tools.

## P.1 — `JarvisAgentSpec` Schema and Spec Repository

**Goal.** Add `JarvisAgentSpec` as the single schema for all JARVIS agents.

**Scope.** Add a spec schema and loader. Minimum fields:

- `spec_id`
- `display_name`
- `description`
- `category` or `kind` (`system`, `default`, `user`)
- `enabled`
- `created_by`
- `prompt_path`
- `purpose`
- `instructions_summary`
- `allowed_message_types`
- `allowed_tools`
- `memory_policy`
- `handoff_policy`
- `output_contract`
- `guardrails`
- `version`

**Design decisions.**

- Specs are data.
- Specs are disabled by default unless policy explicitly enables them.
- Spec IDs are stable and validated.
- Prompt paths must be repo-relative and must not escape `config/prompts/agents/`.
- Tool permissions are declarative policy inputs only; they do not execute tools.

**Acceptance.**

- Unit tests validate schema, serialization, duplicate ID rejection, disabled default behavior, prompt path validation, and required fields.
- Adding a new valid spec file does not require editing Python role literals.

## P.4 — Agent Creator Generates ProjectVision Agent Specs

**Goal.** Use Agent Creator to create and validate the initial ProjectVision-derived JARVIS agent specs.

**ProjectVision-derived target specs.**

From `ProjectVision.md` Agent Layer responsibilities, create disabled specs for:

- `planner` — role-separated task decomposition and proposed plan records
- `executor` — execution-boundary evaluation against policy and allowed tools
- `critic` — validation/review of plans, outcomes, and policy risks
- `curator` — training-data/artifact curation from successful execution traces
- `learner` — training-cycle proposal and regression-gated learning/deployment planning

**Scope.**

- Agent Creator creates these specs from bounded seed inputs.
- Generated specs are written as durable repo-owned files in the configured spec repository.
- The app loads the generated specs from that repository; temporary returned artifacts are not sufficient.
- Each generated spec must validate through `JarvisAgentSpec`.
- Each generated spec is disabled by default.
- `/agents/status` can list each generated spec as an available disabled agent.
- Each creation/validation event is recorded in the existing agent ledger.

**Acceptance.**

- Unit tests prove Agent Creator generates all five ProjectVision-derived specs.
- Tests prove generated spec files exist in the spec repository that JARVIS loads.
- Tests prove each generated spec is valid and disabled by default.
- API/status tests prove each generated spec is visible as disabled and not executable.
- Tests prove no generated spec creates code, enables itself, calls tools, calls models, or changes conversation behavior.
- Ledger records include spec-created and spec-validated events for all five specs.

## Implementation Notes

- Keep the `JarvisAgentSpec` smaller than third-party agent definitions. It should carry only the fields needed for JARVIS policy, prompts, messages, traceability, and future orchestration.
- Do not import CrewAI, LangGraph, AutoGen, or OpenAI Agents SDK. They are reference patterns only.
- Do not make Agent Creator an LLM runtime in this slice. It can be deterministic/spec-only until an explicitly approved future slice adds model-assisted creation.
- Preserve local-first, explicit cognition: specs and ledger records are local artifacts, not hidden model state.
