# 20260630_slice-v.md
## Slice V — Model Selection Policy and Setup Simplification

Status: planned. This document is an implementation contract for the next coding agent. It is not completion evidence.

---

## Purpose

Mature local LLM model selection without replacing the existing catalog, serve-profile, sidecar, or readiness architecture. The slice introduces a small policy layer that selects an LLM model id from `config/models/llm.yaml` by host profile, route, and operator preference, then passes that model id into the existing `resolve_llm_serve_profile(...)` path. The first implementation must use the existing model across all hardware profiles to prove the mechanism before larger Qwen/Gemma model artifacts are introduced.

This slice should reduce user-facing setup complexity: normal bootstrap and Operator settings should expose a simple model policy, not raw model-path editing as the primary experience.

---

## Agent Contract

Implementation agents must follow these rules:

- Preserve the current model catalog -> serve profile -> sidecar command -> OpenAI-compatible runtime flow.
- Do not introduce a parallel model manager, downloader, runtime launcher, or desktop-only selector.
- Do not add new large model artifacts or new default model downloads in this slice.
- Use the existing LLM model entry as the selected model for every policy/profile during mechanism development.
- Make every selection decision observable in readiness/status output.
- Keep raw path overrides available only as advanced/operator escape hatches.
- Add tests before or with implementation for each behavior change.
- Do not update `SYSTEM_INVENTORY.md` until slice closeout evidence exists.
- Append to `CHANGE_LOG.md` only after each sub-slice has validation evidence.

## Baseline Consumed

- `config/models/llm.yaml` already defines one local LLM model, route declarations, generation defaults, and host-specific llama.cpp serve profiles. This slice extends it with selection metadata; it does not replace the model-entry shape.
- `backend/app/models/llm_profiles.py` already resolves one selected model entry to a current-host serve profile. This slice should feed it an explicit selected model id.
- `backend/app/services/local_llm_startup.py` already prepares managed local llama.cpp and starts the sidecar when configured. This slice should add model selection before serve-profile resolution.
- `backend/app/services/local_llm_sidecar.py` already builds `llama-server` commands from a resolved profile. This slice should not fork or duplicate that logic.
- `scripts/ensure_models.py` already verifies and fetches model artifacts and LLM runtime sidecars. This slice should make default LLM ensure behavior selection-aware.
- `scripts/bootstrap.py` already calls `scripts/provision.py install`, then `scripts/ensure_models.py`, then preflight/profile validation. This slice should preserve that simple entry point.
- `backend/app/api/routes/config.py` and `desktop/src/components/settings-panel.js` already provide an Operator settings surface. This slice should make the model portion curated instead of raw-first.

## Target User Experience

Normal setup should remain:

```powershell
backend\.venv\Scripts\python scripts\bootstrap.py
```

Default behavior should be:

1. Detect host profile.
2. Resolve required dependencies.
3. Resolve default LLM model policy.
4. Ensure only the selected local LLM model plus current-host applicable runtime sidecar artifacts by default.
5. Report the active model, role, serve profile, accelerator, and degraded candidates in readiness.
6. Let Operator choose a model policy from a small curated set.

Operator-facing model controls should eventually look like:

```text
Model policy: Auto | Portable | Balanced | Quality | Vision Preview | Diagnostic
Active model: <model id>
Selected because: <short reason>
Advanced: raw model path / binary path / base URL overrides
```

---

## Selection Vocabulary

Use these terms consistently:

- **model id**: a key under `models:` in `config/models/llm.yaml`.
- **model role**: a capability intent such as `portable`, `balanced`, `quality`, `vision_preview`, or `diagnostic`.
- **model policy**: the operator-selected strategy, such as `auto`, `portable`, `balanced`, `quality`, `vision_preview`, or `diagnostic`.
- **hardware selector**: a mapping from current host profile id to model role or model id.
- **serve profile**: existing per-host runtime launch profile under the selected model entry.
- **selected model**: the concrete model id handed to `resolve_llm_serve_profile(..., model_name=...)`.

## Proposed `llm.yaml` Shape

The exact YAML can be improved during implementation, but it should stay close to this structure:

```yaml
default_model: assistant-small-q4

llm_selection:
  default_policy: auto
  operator_env:
    policy: LLM_MODEL_POLICY
    model_override: LLM_MODEL_ID
  roles:
    portable:
      model: assistant-small-q4
      status: dev-alias
    balanced:
      model: assistant-small-q4
      status: dev-alias
    quality:
      model: assistant-small-q4
      status: dev-alias
    vision_preview:
      model: assistant-small-q4
      status: placeholder-no-vision-runtime
    diagnostic:
      model: assistant-small-q4
      status: active
  policies:
    auto:
      windows_amd64_cpu: portable
      windows_arm64_cpu: portable
      windows_amd64_gpu_nvidia_cuda: balanced
      windows_amd64_gpu_amd: portable
      windows_amd64_gpu_intel: portable
      windows_arm64_gpu_qualcomm_adreno_opencl: portable
      windows_arm64_npu_qualcomm_base: portable
      windows_arm64_npu_qualcomm_qnn: portable
    portable:
      "*": portable
    balanced:
      "*": balanced
    quality:
      "*": quality
    vision_preview:
      "*": vision_preview
    diagnostic:
      "*": diagnostic
```

All roles intentionally point at `assistant-small-q4` in this slice. Later slices can replace role targets with Qwen/Gemma model ids after artifact, runtime, and validation evidence exists.

## Sub-Slice V.2 — Backend Model Selection Resolver

### Goal

Create a small backend selector that turns route, hardware profile, model policy, and optional override into a concrete model id.

### Scope

Create:

- `backend/app/models/llm_selection.py` — pure selection logic and trace dataclasses.
- `backend/tests/unit/models/test_llm_selection.py` — policy, wildcard, override, invalid model, invalid role, and route compatibility tests.

Modify:

- `backend/app/core/settings.py` — add `LLM_MODEL_POLICY` and `LLM_MODEL_ID` settings.
- `backend/app/api/routes/config.py` — allowlist the new env keys.

Selection precedence:

1. Explicit `LLM_MODEL_ID`, if set and valid.
2. `LLM_MODEL_POLICY`, defaulting to `llm_selection.default_policy`.
3. Current host profile mapping under that policy.
4. Wildcard mapping under that policy.
5. `default_model` fallback.
6. Fail closed with a clear reason if the selected model does not exist or lacks the requested route.

### Files Not to Modify

- `local_llm_sidecar.py`
- `local_runtime.py`
- `desktop/`
- `scripts/ensure_models.py`

### Validation

backend\.venv\Scripts\python -m pytest backend\tests\unit\api\test_routes.py -q

## Host Profile Default Map

Initial role mapping should cover all existing LLM serve-profile ids. All roles map to the existing model during this slice.

| Serve profile id | Auto role | Rationale |
|---|---|---|
| `windows_amd64_cpu` | `portable` | CPU-safe default. |
| `windows_arm64_cpu` | `portable` | CPU-safe ARM64 default. |
| `windows_amd64_gpu_nvidia_cuda` | `balanced` | CUDA-capable host; still maps to current dev model in this slice. |
| `windows_amd64_gpu_amd` | `portable` | Accelerator path remains degraded/pending. |
| `windows_amd64_gpu_intel` | `portable` | Accelerator path remains degraded/pending. |
| `windows_arm64_gpu_qualcomm_adreno_opencl` | `portable` | OpenCL path exists as staged/build-required; avoid larger model until live proof. |
| `windows_arm64_npu_qualcomm_base` | `portable` | NPU LLM path remains pending. |
| `windows_arm64_npu_qualcomm_qnn` | `portable` | QNN LLM path remains pending; QNN STT proof does not imply QNN LLM proof. |

Future role targets, outside this slice:

| Role | Candidate future family | Notes |
|---|---|---|
| `portable` | Qwen3 4B or Gemma 3 4B text | CPU/Adreno-safe floor after proof. |
| `balanced` | Qwen3 8B | CUDA daily-driver candidate after proof. |
| `quality` | Qwen3 14B or comparable | CUDA quality tier only after live proof. |
| `vision_preview` | Gemma 3 vision-capable or Qwen vision variant | Must stay degraded until image payload/runtime path exists. |
| `diagnostic` | Existing small GGUF | Smoke/provision fallback. |

## Slice-Level Closeout Criteria

All of these must be true:

1. `config/models/llm.yaml` contains explicit model selection metadata and all roles resolve to existing model ids.
2. Backend selection chooses a model id before serve-profile resolution.
3. Managed startup passes the selected model id into the existing serve-profile resolver.
4. Readiness exposes selected model id, policy, role, serve profile, accelerator, and reason.
5. `ensure_models.py` defaults to the selected LLM model, not every LLM model entry.
6. `ensure_models.py --all-llm` remains available for explicit full-catalog validation.
7. Bootstrap remains a single normal setup entry point.
8. Operator model controls are curated policy-first, with raw overrides still available as advanced fields.
9. No new model artifacts or runtime launch architecture are introduced.
10. Focused tests and validators pass with exact command evidence.
11. Any host-class-specific SKIP is explicit and not reported as PASS.

## Risks and Sequencing Notes

| Risk | Severity | Mitigation |
|------|----------|------------|
| `ensure_models.py` begins fetching future large models by default | High | Implement selection-aware default before adding any new LLM entries. |
| Operator exposes raw env fields as primary UX | Medium | Add curated policy fields and demote raw overrides to Advanced. |
| `LLAMA_CPP_MODEL_NAME` remains misleading | Medium | Separate selected catalog model id from raw runtime model name; readiness must explain active source. |
| Vision role appears implemented before image payload support exists | High | Mark `vision_preview` as placeholder/degraded until multimodal runtime support is explicitly added. |
| Multiple model entries duplicate large serve-profile blocks | Medium | In this slice, use existing model only; future slice may factor common serve profiles if warranted. |
| ARM64 Adreno or QNN profiles overclaim capability | High | Preserve degraded/build-required metadata and require live proof before promotion. |

## Finish Line

Slice V is complete when JARVIS can select a local LLM model through a policy-aware catalog mechanism, setup ensures only the selected LLM by default, readiness explains the selected model and serve profile, Operator exposes a simple model-policy control, and the existing llama.cpp sidecar architecture remains intact. The slice does not complete larger Qwen/Gemma adoption, vision runtime support, or new accelerator proof; it prepares the mechanism those later changes will use.
