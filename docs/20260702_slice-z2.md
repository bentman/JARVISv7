# 20260702_slice-z2.md
## Slice Z.2 — Readiness Summary Cleanup

Status: planned. This document is an implementation contract. It is not completion evidence.

---

## Purpose

Slice Z.2 improves the existing readiness surfaces so the current desktop/backend product preview clearly answers:

```text
What is ready, what is degraded or failed, why, and what matters for the current product path?
```

This slice must use the existing `/readiness` API and desktop readiness UI. It must not create a new doctor command, new diagnostic service, new readiness policy, or new launch path.

This slice must preserve the normal working desktop layout. A healthy/ready system should remain visually clean and should not expose extra degraded-detail content that implies a problem where the selected product path is working.

---

## Current Codebase Evidence

The repo already has the core readiness data and product surfaces:

- `backend/app/api/routes/readiness.py`
  - Builds `/readiness` from API startup state.
  - Reports overall status from preflight probe errors.
  - Reports profile id, architecture, active personality, active LLM runtime, degraded flag, family readiness, preflight counts, service status, and resident audio readiness.
  - Refreshes LLM runtime readiness from selection/runtime trace where local `llama.cpp` state is available.

- `backend/app/api/schemas/readiness.py`
  - Defines `ReadinessResponse`, `FamilyReadiness`, `PreflightSummary`, `ServiceReadiness`, and `ResidentAudioReadiness`.
  - Family readiness already carries runtime, device, model, ready, reason, and LLM trace fields.

- `desktop/src/main.js`
  - Calls `api.getReadiness()` during desktop startup.
  - Renders readiness, degraded conditions, and service status through existing components.
  - Sets global desktop state to `READY` or `DEGRADED` from the readiness payload.

- `desktop/src/components/readiness-panel.js`
  - Renders summary facts for arch, profile, active LLM runtime, and status.
  - Renders family rows for LLM, PTT, STT, TTS, and Wake.
  - Currently places family readiness reason mostly in the row title/tooltip instead of visible diagnostic guidance.

- `desktop/src/components/degraded-list.js`
  - Exists as a named degraded surface.
  - Currently clears itself and remains hidden.
  - It must not become an always-visible left-pane list.

- `desktop/src/components/service-status.js`
  - Renders Redis and SearXNG reachability.
  - Service status is present but optional service degradation should not be confused with product-path failure.

- `scripts/validate_backend.py`
  - Uses `selected_path_readiness_summary(...)` for profile/validation fingerprinting.
  - Should remain a validation/operator surface, not a second readiness model.

## Target Mechanism

Keep `/readiness` as the source of truth.

Improve presentation and consistency around the existing readiness payload:

```text
/readiness -> desktop readiness panel -> collapsed degraded detail -> service status -> profile validation wording
```

Use existing fields first. Add schema fields only when they remove ambiguity that cannot be resolved from the current payload.

Recommended improvements:

- Keep the left readiness pane compact. Do not add a permanent visible degraded list there.
- Keep family rows concise in the readiness pane. Runtime/device should remain primary; reason text should not crowd the ready-state view.
- Add a collapsed diagnostic detail surface near the existing `Voice debug details` area, titled `Degraded list detail`.
- Use smaller title text for `Voice debug details` and `Degraded list detail` so the diagnostic area does not dominate the desktop.
- Render degraded/failure detail only inside the collapsed detail surface.
- Hide or omit the degraded detail surface when there are no selected-path blockers or meaningful degradation reasons.
- Distinguish product-path blockers from optional service degradation.
- Keep service status visible and separate, while making optional service state clearly optional.
- Keep `/readiness`, desktop readiness, and `validate_backend.py profile` wording consistent enough that they do not contradict each other.

## Behavior Contract

This slice is presentation and contract polish for existing readiness truth.

It must not:

- add a doctor script;
- add a new backend diagnostic service;
- add a new readiness policy engine;
- change STT/TTS/LLM/Wake runtime selection;
- change QNN/CUDA/DirectML/CPU readiness rules;
- make optional Redis or SearXNG failures block the core product preview;
- start services, install packages, acquire models, or run live probes beyond existing readiness/service checks;
- add an always-visible degraded list to the left readiness pane;
- expose unused runtimes, intentionally skipped accelerators, expected fallbacks, or non-selected paths as apparent product problems;
- hide actual selected-path degraded state behind hover-only UI.

---

## Desired Desktop Behavior

When the desktop loads readiness on a ready system:

- the existing readiness panel remains compact;
- no empty degraded panel is shown;
- no degraded detail is expanded or visually prominent;
- optional service status remains visible as service status only;
- intentionally unused runtime paths are not listed as problems.

When the desktop loads readiness on a degraded or failed selected product path, the user should be able to expand a small diagnostic detail area near `Voice debug details` and see:

- selected family not ready;
- backend readiness degraded from preflight probe errors;
- resident audio degraded reasons where present;
- optional service outages only when clearly labeled optional;
- failure reasons that affect the current product path.

The detail list should not include:

- unused runtimes;
- intentionally skipped accelerators;
- fallback paths that are expected and working;
- optional Redis/SearXNG absence as if it blocks voice/chat;
- anything on a fully ready system.

---

## Acceptance Criteria

Slice Z.2 is complete when:

- `/readiness` remains the backend source of truth for product readiness.
- The desktop readiness panel remains compact and does not gain a permanent visible degraded list.
- `desktop/src/components/degraded-list.js` renders meaningful degraded/failure rows only for selected-path blockers or meaningful degradation reasons.
- Degraded detail is placed in a collapsed diagnostic surface titled `Degraded list detail` near `Voice debug details`, not as always-visible left-pane content.
- `Voice debug details` and `Degraded list detail` titles use smaller text than primary panel headings.
- Optional Redis/SearXNG outages remain represented as optional service degradation, not as core product failure.
- Resident audio degraded reasons are surfaced only when present in the readiness payload.
- Existing `/readiness`, desktop readiness, and `validate_backend.py profile` wording no longer contradict each other on ready/degraded state.
- Tests cover compact ready rendering, degraded-detail rendering, and any changed API/profile wording.

## Stop Conditions

Stop and report before editing further if:

- the change requires runtime selection or readiness policy changes;
- the desktop cannot display required readiness facts because the API payload lacks necessary source data;
- optional service status would need to become a product blocker;
- selected-path readiness cannot be separated from non-selected runtime options using the current payload;
- a needed change falls outside the implementation scope above;
- tests show the existing readiness contract is inconsistent with this document.
