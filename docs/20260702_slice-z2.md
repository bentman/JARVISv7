# 20260702_slice-z2.md
## Slice Z.2 — Product Doctor Command

Status: planned. This document is an implementation contract. It is not completion evidence.

---

## Purpose

Slice Z.2 adds one operator-facing doctor command that answers a simple question:

```text
Can this repo checkout run the current JARVIS desktop/backend product preview, and what should be done next?
```

The command should report environment, provisioning, model artifacts, readiness, backend viability, and a next action in one concise console summary. It should also write a diagnostic report under `reports/diagnostics/`.

This is a diagnostic and operator-experience slice. It must not install packages, download models, start long-running services, launch the desktop, add a new validation framework, or change runtime behavior.

---

## Research Basis

Use the established `doctor` command pattern used by developer tools: a command that inspects installed tooling, reports PASS/DEGRADED/FAIL-style state, and gives troubleshooting direction. Flutter exposes `flutter doctor` to show information about installed tooling, and its troubleshooting docs pair visible errors with explanation and suggested resolution.

Apply that pattern without adding a new CLI framework or external dependency. The repo already uses `argparse`; keep it.

## Current Codebase Evidence

Relevant existing surfaces:

- `scripts/AGENTS.md`
  - Scripts are repository operators.
  - Scripts should call backend helpers when behavior belongs to the app.
  - Scripts should support shared flags where applicable and make side effects obvious.
  - Current primary scripts are `bootstrap.py`, `provision.py`, `ensure_models.py`, `run_backend.py`, `run_jarvis.py`, and `validate_backend.py`.

- `backend/app/services/startup_context.py`
  - Owns the centralized profile/extras/preflight/readiness startup facts from Slice Z.1.
  - Exposes `load_startup_context(...)`, `readiness_summary(...)`, and `selected_path_readiness_summary(...)`.

- `scripts/provision.py`
  - Owns dependency install/verify/lock/explain behavior.
  - `verify` reports missing packages/version drift and exits non-zero when required dependencies are absent or mismatched.

- `scripts/ensure_models.py`
  - Owns model/runtime artifact acquisition and verification.
  - `--verify-only` checks model artifacts without fetching.
  - It already reports selected LLM policy/model/profile metadata for the current host.

- `scripts/run_backend.py`
  - Owns backend API launcher behavior.
  - `--dry-run` loads startup context and reports the planned backend host/port without starting uvicorn.

- `backend/app/api/service_status.py`
  - Owns Redis and SearXNG reachability probes with short timeouts.
  - These services are additive and should not be treated as required for the core product preview unless existing readiness policy says so.

- `scripts/validate_backend.py`
  - Owns governance validation commands.
  - Doctor is not a replacement for closeout validation.

## Target Mechanism

Create one script:

```text
scripts/doctor.py
```

Primary command:

```powershell
backend\.venv\Scripts\python scripts\doctor.py
```

Supported flags:

```text
--verbose
--dry-run
--trace-to <path>
--profile
```

Recommended check categories:

```text
environment
provisioning
models
readiness
backend
services
```

Status vocabulary:

```text
PASS       required product-preview check is satisfied
DEGRADED   product preview can probably run, but a capability/service is missing or degraded
FAIL       required product-preview check is not satisfied
SKIP       intentionally not checked in dry-run or not applicable
```

Exit code contract:

```text
0 = PASS
1 = FAIL
2 = DEGRADED
3 = ENVIRONMENT_UNSATISFIED
```

Use `3` only when the doctor command cannot inspect the repo at all, such as missing Python/runtime prerequisites before normal checks can run.

---

## Expected Console Shape

Keep output concise and stable enough to test:

```text
JARVISv7 product doctor
status=PASS|DEGRADED|FAIL
report=reports\diagnostics\<timestamp>-doctor.txt
[PASS] environment: repo python and project files detected
[PASS] provisioning: required packages present
[PASS] models: selected model artifacts present
[PASS] readiness: ready; tokens=<n>
[PASS] backend: run_backend dry-run passed
[DEGRADED] services: redis unreachable; searxng reachable
next_action=<one short action>
```

`--verbose` may include subprocess commands, stdout/stderr excerpts, profile id, extras, model selection, and probe error keys.

`--profile` should include the existing host fingerprint and profile/readiness facts, not a new profile schema.

## Required Checks

### Environment

Check only local, side-effect-free facts:

- repo root resolved from the script path
- `pyproject.toml` exists
- `backend/` exists
- `scripts/` exists
- current Python executable is usable
- warn or degrade when not run from `backend/.venv/Scripts/python`

### Provisioning

Use the existing provisioning authority. Do not duplicate dependency resolution logic.

Preferred mechanism:

```powershell
backend\.venv\Scripts\python scripts\provision.py verify
```

Capture return code, stdout, and stderr for the report.

### Models

Use the existing model authority. Do not duplicate catalog verification logic.

Preferred mechanism:

```powershell
backend\.venv\Scripts\python scripts\ensure_models.py --verify-only
```

Capture return code, stdout, and stderr for the report.

### Readiness

Use `backend.app.services.startup_context.load_startup_context(...)` and the shared readiness summary helpers. Do not rebuild profile/extras/preflight/readiness locally.

Report:

- profile id where available
- arch
- selected readiness summary
- family readiness rows for `stt`, `tts`, `llm`, and `wake`
- preflight probe error keys when present

### Backend Viability

Use the existing backend launcher dry-run path. Do not bind a port or start uvicorn.

Preferred mechanism:

```powershell
backend\.venv\Scripts\python scripts\run_backend.py --dry-run
```

A dry-run failure is a doctor failure.

### Services

Use `backend.app.api.service_status.collect_service_statuses(...)` for Redis and SearXNG only.

Treat additive service failures as `DEGRADED`, not `FAIL`, unless existing readiness policy already treats the service as required for the selected product path.

## Next Action Rules

The doctor command must emit exactly one short next action.

Suggested priority:

1. Environment failure -> fix Python/venv/repo checkout.
2. Provisioning failure -> run `backend\.venv\Scripts\python scripts\provision.py install`.
3. Model failure -> run `backend\.venv\Scripts\python scripts\ensure_models.py`.
4. Readiness failure -> inspect the report path and preflight probe errors.
5. Backend dry-run failure -> run `backend\.venv\Scripts\python scripts\run_backend.py --dry-run --verbose`.
6. Service degradation only -> start optional services or continue without them.
7. All checks pass -> proceed to the repo-run launch path.

Do not emit multiple competing next actions.

## Behavior Contract

The doctor command is diagnostic only.

It must not:

- install Python packages
- run `provision.py install`
- fetch or download model artifacts
- run `ensure_models.py` without `--verify-only`
- start uvicorn
- launch the desktop
- start Docker containers
- run live runtime tests
- run broad unit/regression validation
- mutate `.env`, config, models, cache, data, or runtime artifacts

It may write a diagnostic report under `reports/diagnostics/`.

---

## Acceptance Criteria

Slice Z.2 is complete when:

- `backend\.venv\Scripts\python scripts\doctor.py` exists and runs from the repo root.
- The command prints one concise product doctor summary with overall status, report path, check rows, and one next action.
- The command writes a diagnostic report under `reports/diagnostics/`.
- Provisioning checks delegate to `scripts/provision.py verify`.
- Model checks delegate to `scripts/ensure_models.py --verify-only`.
- Readiness checks consume `backend.app.services.startup_context` rather than rebuilding startup facts.
- Backend viability uses `scripts/run_backend.py --dry-run` and does not start uvicorn.
- Optional Redis/SearXNG service failures produce `DEGRADED`, not `FAIL`, unless existing readiness policy requires them.
- Focused tests cover PASS, FAIL, DEGRADED, dry-run, and report-path behavior.

## Stop Conditions

Stop and report before editing further if:

- implementation requires changing provisioning, model verification, readiness, runtime selection, or backend launch behavior;
- the doctor command needs to install, fetch, start, or mutate anything to determine state;
- backend viability cannot be checked without starting uvicorn;
- service checks block for long periods or require Docker orchestration;
- a needed file is outside the implementation scope above.
