# Slice CC - Latency and Response-Path Cleanup
Status: planned. This document is an implementation contract for a coding agent. It is not completion evidence.

---

## Purpose

Reduce avoidable latency and preserve response fidelity in validated hot paths without changing product architecture. This slice covers:

1. preserving model response text before voice-only cleanup;
2. reusing HTTP clients in local LLM request and readiness paths;
3. bounding STT input payloads and avoiding repeated model-file probes after runtime readiness is known;
4. isolating streaming partial assistant response work as the final, separately validated phase.

Use the `ProjectVision.md` wording: `streaming partial assistant response`. Do not add invented runtime behavior, shadow voice paths, or simulated completion evidence.

## Target Observable Behavior

- Text turns and recorded artifacts retain code blocks, markdown, and structured model output that should be visible in chat and memory.
- Voice output still receives voice-safe text, but that voice-only text does not overwrite the stored model response.
- Local LLM request/probe paths avoid avoidable TCP connection churn.
- STT receives bounded utterance audio without terminal silence added by endpoint detection.
- STT runtime calls do not repeatedly scan model files after readiness has already been established by warmup or first successful load.
- Existing conversation states, tool behavior, memory write behavior, degraded-mode behavior, and barge-in behavior remain intact.
- TTS partial playback work is isolated at the end and must not disturb validated typed interaction.

## CC.1 - Response Text Preservation and Policy Reuse

### Goal

Preserve the model response text used by typed chat, artifacts, and episodic memory. Derive voice-safe text separately for voice output only. Avoid compiling the same personality policy twice during one reasoning turn.

### Scope

Modify:

- `backend/app/conversation/engine.py`
- `backend/app/cognition/prompt_assembler.py`
- `backend/tests/unit/conversation/test_engine.py`
- Existing prompt/LLM tests only if signature compatibility requires updates.

### Required behavior

- In `_run_reasoning_path`, keep a distinct stored response value after bounding and style guard application.
- Derive a separate voice-output text value only when `speak_response=True`.
- `TurnResult.response_text`, turn artifacts, working memory, and episodic memory write input must use the stored response value, not the voice-output value.
- Text turns must preserve fenced code blocks, inline code, markdown emphasis, lists, and other renderable formatting.
- Voice turns must pass voice-safe text to the TTS path while retaining the stored response value in `TurnResult.response_text` and artifacts.
- Add an optional compiled policy path to `assemble_prompt_envelope` without breaking existing callers. Preserve the current `assemble_prompt(...)` wrapper behavior.
- Compile the personality policy once per reasoning turn in `TurnEngine` and reuse it for prompt assembly and style guarding.

### Files not to modify

- `backend/app/memory/*.py`
- TTS runtime files
- LLM runtime files
- Desktop files

### Validation

Prefer existing tests:

1. Extend `backend/tests/unit/conversation/test_engine.py`.
2. Add or update assertions that a text turn returning fenced code and markdown preserves that formatting in `TurnResult.response_text` and the written turn artifact.
3. Add or update assertions that voice output receives voice-safe text while `TurnResult.response_text` keeps the stored response value.
4. Keep existing state-transition, tool, retrieval, episodic-write, degraded-TTS, and barge-in tests passing.

Inner-loop commands:

```text

```

### Acceptance

```text

Personality policy compilation is not duplicated inside one reasoning turn.
Existing conversation state, tool, retrieval, memory, degraded-TTS, and barge-in behavior remains intact.
```

## CC.3 - STT Hot-Path Cleanup

### Goal

Reduce avoidable STT overhead by removing terminal silence from completed utterance payloads and avoiding repeated model-file probes after runtime readiness is already known.

### Scope

Modify:

- `backend/app/services/utterance_segmenter.py`
- `backend/app/runtimes/stt/onnx_whisper_runtime.py`
- `backend/tests/unit/services/test_utterance_segmenter.py`
- `backend/tests/unit/runtimes/stt/test_stt_runtime.py`

### Required behavior: utterance payload bounds

- When endpoint reason is `silence`, remove exactly the terminal `silence_after_speech` samples from `UtteranceSegment.audio`.
- Preserve configured pre-roll.
- Preserve interior pauses that occur before later speech.
- Do not trim `max-duration`, `stream-ended`, `too-short`, or `no-speech` payloads unless an existing test proves that behavior is expected.
- Keep diagnostics internally consistent with the returned audio payload.
- Do not change VAD thresholds, no-speech timeout behavior, pre-roll behavior, or speech start debounce behavior.

### Required behavior: STT model-file probes

- Preserve fail-closed behavior before a runtime is loaded or verified.
- After successful warmup or first successful load, `transcribe()` must not repeat full model-file availability scans on every call.
- For `OnnxWhisperRuntime`, use loaded model state or an explicit readiness flag to avoid repeated required-file checks after load.
- For `QnnWhisperRuntime`, use loaded encoder/decoder session state or an explicit readiness flag to avoid repeated configured-file lookup after load.
- Preserve QNN provider guard behavior: CPU fallback must still fail closed.
- Preserve existing tokenizer, feature extractor, decode limit, empty transcript, and provider-selection behavior.

### Files not to modify

- Non-STT runtimes
- Audio capture services other than `utterance_segmenter.py`
- Memory, conversation, or desktop files

### Validation

Prefer existing tests:

1. Update `backend/tests/unit/services/test_utterance_segmenter.py` for new terminal-silence trimming expectations.
2. Ensure existing tests still cover no-speech timeout, too-short, max-duration, bounded pre-roll, quiet speech, noise rejection, spike debounce, interior hesitation, and stream-ended behavior.
3. Update `backend/tests/unit/runtimes/stt/test_stt_runtime.py` with focused assertions that repeated `transcribe()` calls do not repeat file probes after readiness/load.
4. Keep existing QNN decode loop and CPU-fallback guard tests passing.

Inner-loop commands:

```text
backend/.venv/Scripts/python -m pytest backend/tests/unit/services/test_utterance_segmenter.py
backend/.venv/Scripts/python -m pytest backend/tests/unit/runtimes/stt/test_stt_runtime.py
```

### Acceptance

```text
Silence-ended utterance payloads exclude terminal endpoint silence.
Interior pauses and configured pre-roll are preserved.
STT runtimes fail closed before readiness but avoid repeated file probes after readiness/load.
Existing STT provider, QNN, VAD, and utterance-state behavior remains intact.
```

## CC.4 - TTS Partial Playback Split

### Goal

Implement streaming partial assistant response only after CC.1 through CC.3 are complete and validated. This work must extend the existing TTS/playback path without creating a separate conversation path or breaking typed interaction.

### Split

CC.4A defines a safe runtime/playback contract with fallback behavior.

CC.4B wires that contract through `TurnEngine` while preserving state transitions, degradation, artifacts, and interruption behavior.

If CC.4 cannot be completed without broad concurrency or architecture changes, stop after CC.1 through CC.3 and leave CC.4 unimplemented for a follow-up slice. Do not partially wire an unvalidated voice path.

### Scope

Modify only after CC.1 through CC.3 pass:

- `backend/app/runtimes/tts/base.py`
- `backend/app/runtimes/tts/kokoro_onnx_runtime.py`
- `backend/app/runtimes/tts/playback.py`
- `backend/app/conversation/engine.py`
- `backend/tests/unit/conversation/test_engine.py`
- TTS/playback unit tests if an existing file exists; otherwise add the smallest focused unit test file following current test naming patterns.

### Required behavior: CC.4A runtime/playback contract

- Keep the existing blocking TTS method intact for backward compatibility.
- Add a default chunk iterator method to `TTSBase`, for example `iter_audio(self, text: str) -> Iterable[np.ndarray]`, that yields the existing full audio array.
- Override the chunk iterator in Kokoro only when the installed package exposes a verified chunk API. Do not assume undocumented package behavior.
- Add playback support for an iterable of audio chunks using `sounddevice.OutputStream` or the narrowest equivalent existing pattern.
- Preserve existing `play`, `start`, `stop`, `is_playing`, `last_output_device`, and bounded wait behavior for existing tests and callers.
- Keep sample-rate reporting correct for both full-array and chunked output paths.

### Required behavior: CC.4B conversation integration

- `TurnEngine` must still use the canonical turn engine for text and voice.
- `RESPONDING -> SPEAKING -> IDLE` must remain the normal voice path.
- TTS unavailable still returns a degraded, non-failed voice turn.
- TTS/playback exceptions still close the turn to `FAILED` with the correct failure phase.
- Barge-in must still stop playback, emit the interruption event, transition through `INTERRUPTED -> RECOVERING -> IDLE`, and write the event into the turn artifact.
- `tts_output_device` and phase duration fields must remain populated where existing tests expect them.
- Do not let partial playback change `TurnResult.response_text`; CC.1 owns stored response text behavior.

### Files not to modify

- STT runtimes
- LLM runtimes
- Search runtimes/tools
- Memory storage implementations
- Desktop UI implementation

### Validation

Prefer existing tests:

1. Extend `backend/tests/unit/conversation/test_engine.py` to cover chunked audio playback through existing fake TTS/playback patterns.
2. Preserve existing degraded-TTS, playback failure, speaking-state, output-device, and barge-in tests.
3. Add narrow playback tests for iterable chunk handling and stop behavior if no existing playback test file exists.
4. Run runtime voice tests only when the host has the required runtime/audio capabilities; otherwise report `SKIPPED` with the validator exit code and reason.

Inner-loop commands:

```text
backend/.venv/Scripts/python -m pytest backend/tests/unit/conversation/test_engine.py
```

### Acceptance

```text
Existing full-array TTS callers still pass.
Chunked TTS output can begin playback before the final chunk is available when the runtime supports chunks.
Fallback chunk iterator preserves current behavior when no verified chunk API exists.
Conversation state, degraded mode, interruption events, artifacts, output-device reporting, and typed turns remain intact.
```
