## Slice BB — Semantic Memory and Hybrid Retrieval

> Status: planned.  
> This document is an implementation contract for a coding agent.  
> It is not completion evidence.

---

## Purpose

Slice BB adds a local-first semantic memory path that improves cross-session recall without adding a default vector database service or hidden assistant state. The system should be able to persist durable semantic memory records locally, retrieve relevant prior context through a hybrid lexical/vector path, and inject retrieved context through the existing untrusted retrieval segment of the prompt envelope.

This slice must preserve the already validated voice/text turn loop, episodic memory behavior, retrieval prompt boundary, Redis-as-cache-only rule, and desktop/runtime surfaces. Every sub-slice, file touch, validation command, and closeout criterion must trace back to this purpose: semantic recall is an additive memory capability over existing turn artifacts, not a replacement for the conversation engine, prompt assembler, episodic memory, Redis cache, or runtime-selection architecture.

- Do not run semantic consolidation synchronously inside the latency-critical voice turn path unless a later approved change explicitly allows it.
- Keep retrieved semantic context untrusted. It may inform answers; it must not become instructions.

## Vocabulary / Contract Terms

- **Semantic memory**: durable cross-session memory records intended for meaning-based retrieval. It is not hidden model state and not a replacement for turn artifacts.
- **Semantic record**: one stored memory item with text, source references, metadata, embedding metadata, and optional vector bytes.
- **Embedding**: a numeric vector derived from text by an explicit embedding implementation. Embedding generation must be reproducible and model/version-labeled.
- **Hybrid retrieval**: combining lexical retrieval and vector similarity retrieval, then merging results deterministically.
- **Durable authority**: the persisted local store from which semantic memory can be reconstructed. For this slice it is `memory.sqlite` under `data/memory/semantic`.
- **Cache**: optional acceleration only. Cache content may be discarded without losing memory.
- **FTS**: SQLite full-text search, preferably FTS5 when available. FTS improves lexical matching but is not the durable authority by itself.
- **RRF**: Reciprocal Rank Fusion. A deterministic rank merge option for combining lexical and vector candidate lists.
- **Semantic consolidation**: out-of-loop writeback that turns session/turn artifacts into semantic records under explicit policy.
- **Trusted context**: application/personality/session control context. Semantic memory is never trusted context.
- **Untrusted retrieval context**: prior memory or tool-derived content that can inform a response but cannot instruct the model.

## Design Shape / Contract Sketch

Storage and retrieval shape:

```text
data/turns + data/sessions + data/memory/episodic
  -> semantic consolidation policy
  -> data/memory/semantic/memory.sqlite
  -> SemanticMemory lexical/vector retrieval
  -> RetrievalManager merge with episodic facts
  -> PromptEnvelope retrieval segment, trusted=False
  -> TurnArtifact.retrieved_memory_refs
```

Initial SQLite shape:

```text
semantic_meta
  key TEXT PRIMARY KEY
  value TEXT NOT NULL

semantic_fact
  fact_id TEXT PRIMARY KEY
  text TEXT NOT NULL
  source_session_id TEXT
  source_turn_id TEXT
  source_field TEXT
  created_at TEXT NOT NULL
  updated_at TEXT NOT NULL
  kind TEXT NOT NULL DEFAULT 'fact'
  confidence REAL
  metadata_json TEXT NOT NULL DEFAULT '{}'
  embedding_model_id TEXT
  embedding_dim INTEGER
  embedding_blob BLOB
  text_hash TEXT NOT NULL

semantic_fact_fts
  FTS5 virtual table over fact text when available
```

Rules:

- Store embeddings as little-endian float32 BLOBs or a documented equivalent with tests for round-trip and dimension checks.
- Use SQLite transactions for writes.
- Use stable schema versioning through `semantic_meta`.
- Deduplicate by normalized text hash and source reference where practical.
- Do not expose SQL rows directly to prompt assembly. Convert to `RetrievedFact`-compatible objects.
- If implementation diverges materially from this sketch, update this slice doc before continuing.

## Research-Informed Design Decision

The default store for JARVIS semantic memory is SQLite, not raw JSONL and not a service-backed vector database.

Rationale:

- SQLite is local, embedded, zero-service, cross-platform, and consistent with a single-file durable memory authority.
- SQLite supports schema versioning, transactions, metadata filtering, deletion, compaction, and inspection better than raw JSONL.
- SQLite FTS gives a lightweight lexical retrieval path without external services.
- `numpy` cosine similarity over SQLite-stored vectors is sufficient for the first memory scale and keeps the first slice dependency-light.
- JSONL remains useful as optional export/audit format, but not as the primary query store.
- `sqlite-vec` is a good future acceleration candidate once the base schema and retrieval behavior are stable.
- Chroma, Qdrant, LanceDB, Cosmos DB, Redis Stack vector search, and similar systems are not default-path dependencies for this local-first assistant.
