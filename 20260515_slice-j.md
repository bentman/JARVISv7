**Date:** 2026-05-15
**Status:** Proposed
**Depends on:** Slice I closed and verified (`SYSTEM_INVENTORY.md`)

---

## Purpose

Group I normalized the acceleration path and proved it end-to-end. The data is correct. The problem is that the desktop shell does not yet surface it in a way that gives a user or operator meaningful situational awareness.

The current shell is a diagnostics surface. It shows a `BOOTSTRAP` state in a header card, a readiness list rendered into a `<div>`, a `<pre>` block for voice debug output, and a `Hold to Talk` button. State transitions are visible only as a text string. Degraded conditions are a body attribute toggle. No subsystem indicator shows whether STT/TTS/LLM/Wake is individually ready, degraded, or operating on a fallback device. No named interaction state is visible during a turn. The operator cannot tell from the UI alone why a turn failed, what device is active, or whether the backend is using QNN, CUDA, or CPU for a given family.

Group J closes this gap by building the information-dense, named-state shell that the product requires. The backend already exposes everything needed — readiness per family, fallback reasons, turn state, wake status, degraded conditions. This slice consumes that data and surfaces it with appropriate fidelity.

This is also the prerequisite for Group K (UI controls and operator settings). Controls cannot be built on top of an opaque display surface. The readiness data must be visible and reliable before interactive controls that modify it are added.

## Sub-Slice J.1 — Per-Subsystem Readiness Panel + Named State Labels

### Why Here

The current readiness panel renders a flat `<ul>` of family items inside a `<div id="readiness-panel">`. State is a single text string in the header. Neither surface communicates what is usable, what is degraded, or why. Before interaction-quality improvements can land, the operator must be able to see subsystem health at a glance without reading the `<pre>` debug block.

This sub-slice introduces the two structural elements missing from the current shell: a sidebar readiness panel with per-subsystem status indicators, and named interaction state labels that match the full conversation state machine.

### Goal

The desktop shell sidebar shows a per-subsystem readiness indicator (green/amber/red dot + label) for each of STT, TTS, LLM, and Wake. Each indicator shows the active device and the readiness reason on hover or expansion. Named interaction states are visible during a turn: `Ready`, `Listening`, `Transcribing`, `Thinking`, `Speaking`, `Interrupted`, `Degraded`, `Failed` — not raw enum strings from the backend.

A degraded-conditions list is visible below the readiness indicators when any family is not in a `ready` state, listing each degraded condition with its reason string.

### Scope

**`desktop/src/index.html`** — restructure the sidebar `<aside>` section:

The current flat `<dl>` backend facts block is replaced with a structured sidebar containing three named sections:

## Sub-Slice J.2 — Wake Status + PTT Indicator

### Why Here

Wake status is currently displayed in two `<dd>` elements (`wake-status`, `wake-detail`) in the sidebar. It conveys provider name and availability but does not communicate monitoring state, last detection, or PTT-only fallback visually. The PTT button has no visual active state. A user cannot tell whether the system is passively listening via wake word or requires manual PTT, without reading the sidebar text carefully.

### Goal

The wake section of the sidebar shows an explicit `active / inactive / PTT-only` indicator with the provider name. The PTT button has a clear visual pressed/active state and an in-flight guard that prevents double-invocation. The wake monitoring state (`monitoring: true/false`) is reflected in the indicator.

### Scope

**`desktop/src/components/wake-indicator.js`** — `WakeIndicator` module:

- Accepts the `GET /status/wake` payload.
- Renders a wake status row showing:
  - Provider name and an active/inactive/PTT-only label
  - A dot indicator: active monitoring → `--color-ready`; PTT-only fallback → `--color-degraded`; error → `--color-failed`
  - `last_detected` timestamp when non-null, formatted as `Last: <relative time>` (e.g. `Last: 2 min ago`)
- Export: `renderWakeStatus(wakePayload, containerEl)`.

**`desktop/src/index.html`** — wake section in sidebar:

Replace the current `<dt>Wake</dt><dd id="wake-status">` / `<dt>Wake detail</dt><dd id="wake-detail">` pair with:

```html
<section class="wake-section">
  <h3>Wake</h3>
  <div id="wake-indicator"></div>
</section>
```

**`desktop/src/main.js`** — replace `refreshWakeStatus()` direct DOM manipulation with `WakeIndicator.renderWakeStatus(status, document.querySelector('#wake-indicator'))`.

**PTT button in-flight guard:**

The current `Hold to Talk` button has no visual active state beyond `pointerdown`/`pointerup` handler semantics. Replace with:

- `aria-pressed="false"` attribute; set to `"true"` during capture.
- `data-state="idle"` attribute; transitions to `"recording"` on `pointerdown` and back to `"idle"` on stop.
- Button label changes: `Hold to Talk` → `🎤 Release` during capture.
- In-flight guard: if `mediaRecorder` is already active when `pointerdown` fires, the new capture attempt is rejected silently (no error message; guard only prevents double-invocation).
- Visual: CSS rule `.ptt-button[data-state="recording"]` applies a pulse animation using `--color-ready` as the glow color.

## Sub-Slice J.3 — Visual Design System: Dark Theme + Interaction States

### Why Here

J.1 and J.2 introduce semantic structure and component architecture. J.3 applies a coherent visual language. Without this sub-slice, the shell has correct structure but lacks the visual fidelity appropriate for an always-on desktop assistant — the interaction states defined in J.1 and J.2 carry no weight without distinct visual differentiation.

This is not cosmetic polish. An ambient desktop assistant with opaque or undifferentiated state display fails as a product. The operator cannot tell at a glance whether the system is ready, speaking, or in a degraded condition.

### Goal

The desktop shell uses a coherent dark theme with cyan accent, distinct per-state colors for all named interaction states, and explicit visual differentiation between ready, degraded, and failed subsystem indicators. All colors and spacing are defined as CSS custom properties in `desktop/src/tokens.css`. No hard-coded hex values appear anywhere in `style.css` or component JS files.

## Sub-Slice J.4 — Conversation Log: Markdown Rendering + Turn Quality

### Why Here

The current conversation log renders plain text in `<p>` elements. Code blocks, bullet lists, and structured responses from the LLM are displayed as raw strings. For a desktop assistant used for research, code help, and structured analysis, unrendered markdown is a significant readability regression from what the product vision requires.

This sub-slice adds markdown rendering and improves conversation log turn quality — timestamps, role indicators, and turn separation — to match the information density expected of a persistent local assistant.

### Goal

LLM responses that include markdown (code blocks, headers, lists, bold/italic) render correctly in the conversation log. Each turn entry shows a timestamp, role label, and clear visual separation. Presence messages are visually distinct from user/assistant turns.

### Scope

**Markdown renderer:**

Use `marked.js` (available via CDN `https://cdn.jsdelivr.net/npm/marked/marked.min.js`). Import in `index.html` via `<script>` before `main.js`. No build step; no npm package required. `marked.parse(text)` returns HTML; inject via `element.innerHTML` for assistant messages only.

Security: all assistant response text comes from the local backend, not from untrusted external input. No user-controlled content is ever passed to `marked.parse` — only `response_text` from `POST /task/text` and `POST /task/voice` responses. This is not an XSS surface. Document this constraint in a comment in `main.js`.

**`appendMessage(role, text)` — refactor:**

```
user:      plaintext in <p> (sanitized; never marked.parse)
assistant: marked.parse(text) → innerHTML
presence:  plaintext in <p class="presence-text"> (italic, muted)
system:    plaintext in <p class="system-text"> (small, muted)
```

Each `<article class="message {role}">` entry:
- Timestamp in `<time class="stamp" datetime="{iso}">` with locale time display
- Role label in `<strong class="role">{role}</strong>` mapped to display names:
  - `user` → `You`
  - `assistant` → `JARVIS`
  - `presence` → (hidden; presence messages show no role label)
  - `system` → `System`
- Response body in appropriate element per role above

**Code block styling in `style.css`:**

Add token-driven rules for `pre`, `code`, `pre code` within `.message.assistant`:
- Background: `--color-bg-elevated`
- Border: `1px solid --color-border`
- Border-radius: `--radius-sm`
- Font: `--font-mono`, `--font-size-sm`
- Padding: `--space-sm`
- Overflow: `auto`

- `test_marked_js_script_present` — assert `marked.min.js` or `marked` CDN script is present in `index.html`
- `test_append_message_uses_html_for_assistant` — static test not possible; document that this is verified via desktop live test only
