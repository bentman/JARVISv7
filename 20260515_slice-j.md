# 20260515_slice-j.md
## Slice J — Runtime/Readiness UX + Degraded-State Surfacing

**Date:** 2026-05-15
**Status:** Proposed
**Depends on:** Slice I closed and verified (`SYSTEM_INVENTORY.md`)

---

## Purpose

Group I normalized the acceleration path and proved it end-to-end. The data is correct. The problem is that the desktop shell does not yet surface it in a way that gives a user or operator meaningful situational awareness.

The current shell is a diagnostics surface. It shows a `BOOTSTRAP` state string in a header card, a readiness list rendered into a `<div>`, a `<pre>` block for voice debug output, and a `Hold to Talk` button. State transitions are visible only as a raw text string. Degraded conditions are a body attribute toggle with no readable explanation. No subsystem indicator shows whether STT, TTS, LLM, or Wake is individually ready, degraded, or operating on a fallback device. No named interaction state is visible during a turn. The operator cannot tell from the UI alone why a turn failed, what device is active, or whether the backend is in a degraded but functional condition.

Group J closes this gap by building the information-dense, named-state shell that the product requires. The backend already exposes everything needed — readiness per family with device and reason strings, turn state, wake status, and degraded condition details. This slice consumes that existing data and surfaces it with appropriate fidelity.

This slice is also the prerequisite for Group K (UI controls and operator settings). Controls cannot be built on top of an opaque display surface. The readiness data must be visible and reliable before interactive controls that modify it are added.

## Sub-Slice J.1 — Per-Subsystem Readiness Panel + Degraded-Condition Display

### Why Here

The current readiness panel renders a flat list of family items into a single `<div>`. The operator cannot tell at a glance which specific subsystem is degraded, what device it is using, or why a fallback was taken. This sub-slice replaces that flat display with a structured per-family readiness presentation that exposes the device, runtime, and reason strings already present in the `GET /readiness` payload.

### Goal

The desktop shell sidebar shows a clear per-family readiness row for each of STT, TTS, LLM, and Wake. Each row shows the active runtime, device, and a ready or degraded indicator. When any family is not in a `ready` state, a separate degraded-conditions section lists each affected family with its reason string. No backend or schema changes are required.

### Scope

**`desktop/src/components/readiness-panel.js`** — new module `ReadinessPanel`:

- Accepts the `GET /readiness` payload.
- Renders one row per family showing: family name label, active `runtime/device` string, and a ready or degraded indicator driven by the `ready` field.
- The indicator distinguishes at minimum: `ready`, `degraded` (not ready but a runtime is selected), and `failed` (runtime unavailable). Drive these states from the `ready` flag and the presence of known failure substrings in the `reason` string (e.g. `MISSING`, `unavailable`). The exact substrings to match should be read from the live readiness payload at implementation time to ensure alignment.
- Tooltip or inline expansion on each row shows the full `reason` string.
- Export: `renderReadiness(readinessPayload, containerEl)`.

**`desktop/src/components/degraded-list.js`** — new module `DegradedList`:

- Accepts the `GET /readiness` payload.
- Collects all families where `ready: false` or where the `reason` string indicates a known fallback condition.
- Renders a named degraded-conditions section with one entry per affected family and its reason string.
- The section is hidden entirely when no degraded conditions are present.
- Export: `renderDegradedList(readinessPayload, containerEl)`.

**`desktop/src/index.html`** — restructure the sidebar `<aside>`:

- Replace `<div id="readiness-panel">` with a container for `ReadinessPanel` output.

**`desktop/src/main.js`** — update call sites:

- Import component modules at the top of `main.js`.
- The `body[data-degraded]` attribute toggle is preserved for backward compatibility with existing CSS rules.

## Sub-Slice J.3a — Wake/PTT Interaction Boundary Assessment

### Why Here

The current shell presents a single manual activation path implemented as Hold-to-Talk: capture is held for the duration of a pointer press. This is a temporary proof path, not the intended product behavior. The intended model is two distinct activation paths — wake word (passive default) and PTT (deliberate start/stop). Before J.3b replaces HTT with true PTT, J.3a must confirm that true PTT is achievable within the existing invoke surface without new Tauri commands, backend changes, or plugin additions. J.3a is inspection and boundary definition only. No production code changes.

### Goal

Documented assessment of whether true click-to-start / click-to-stop PTT is implementable using the existing `MediaRecorder` API in the WebView2 layer and the existing `submit_voice` Tauri command. If no blocker is found, J.3a closes `PROCEED` and J.3b begins. If a blocker is found, J.3a closes `BLOCKED` with the exact constraint named, and J.3b closes as `Deferred`.

### Scope

**Inspect the following — read-only, no code changes:**

- `desktop/src/main.js` — current `startVoiceCapture()`, `stopVoiceCapture()`, and `handleVoiceCaptureStop()` functions; confirm the full HTT flow and where `submit_voice` is invoked.
- `desktop/src-tauri/src/lib.rs` — confirm `submit_voice` accepts `audio_bytes: Vec<u8>` and calls `submit_voice_turn`; confirm no streaming audio command exists.
- `desktop/src-tauri/src/backend.rs` — confirm `submit_voice_turn` posts a complete WAV payload to `POST /task/voice`.

**Intended v7 PTT behavior to define:**

- **Idle state:** button communicates the user can start a voice turn; no capture in progress.
- **Start action:** user clicks (does not hold) the button; capture begins; button transitions to recording state.
- **Recording state:** button communicates recording is in progress and another click will stop and submit it.
- **Stop action:** user clicks the button a second time; capture stops; WAV is encoded; `submit_voice` is invoked; button transitions to processing state.
- **Processing state:** button is non-interactive until the turn result is returned; returns to idle after completion or failure.
- **Duplicate-press guard:** a second click in recording state stops and submits rather than starting a new capture; a click in processing state is ignored.

**Blocker assessment criteria:**

| # | Check | Expected finding |
|---|-------|------------------|
| B1 | `MediaRecorder.start()` and `.stop()` are discrete callable events in WebView2 | Yes — W3C standard; Chromium-based |
| B2 | `submit_voice` accepts a complete `audio_bytes` payload, not a stream | Yes — confirmed in `lib.rs` |
| B3 | No existing JS global state prevents click-start/click-stop handler replacement | Verify by reading `main.js` |
| B4 | `dataavailable` fires correctly on `.stop()` in WebView2 | Yes — Chromium-based; standard behavior |
| B5 | No backend route change required for click-start/click-stop vs. hold-to-talk | Yes — `POST /task/voice` receives the same WAV payload |

If any check is `No` or `Unknown`, J.3a closes as `BLOCKED` with the specific check named.

## Sub-Slice J.4 — v7-Native Visual Polish

**Depends on:** J.3b closed (as `Verified` or `Deferred`). J.4 must not begin until J.3b closes, so that visual polish is applied to the intended PTT interaction model and not to the HTT proof path.

### Why Here

J.1–3b introduce the structural components and semantic markup that the shell needs. J.4 applies a coherent visual language appropriate for a persistent, dark-themed desktop assistant. Without this, the shell has correct structure but undifferentiated visual weight — ready, degraded, and failed states look the same; interaction states carry no visual feedback; the conversation log has no visual hierarchy.

This sub-slice does not change behavior. It changes how existing behavior looks.

### Goal

The desktop shell presents a coherent dark theme with a small set of v7-native visual tokens. Subsystem status indicators are visually differentiated by readiness state. Named interaction states from J.2 have distinct visual emphasis. The PTT button recording state (`data-capture-state="recording"`) is visually obvious. The conversation log has clear turn-level visual hierarchy. All styling is expressed through CSS custom properties defined in one place, with no hard-coded values scattered across the stylesheet.

### Scope

**Visual token set — described as desired outcomes, not as CSS code:**

Introduce a small v7-native visual token set covering: background depth (a progression from base surface to elevated surface), text weight (primary, secondary, and muted), a single accent color used for active/ready states and brand identity, semantic state colors (one per named interaction state — ready, degraded/interrupted, failed, active-capture), border treatment (subtle separation without heavy lines), spacing rhythm, and typography scale. The token set should be minimal — covering only what J.1–J.4 require. Tokens are defined in one location and referenced everywhere else.

**Dark theme — described as desired visual qualities:**

The base surface should read as a deep, neutral dark — not pure black, but dark enough that accent colors read clearly against it. The accent color for ready/active states should be vivid and cool-toned, consistent with the product's technical character. Panel surfaces are slightly lighter than the base, creating depth without high contrast. Text uses a light neutral for primary content and a mid-tone for secondary information. Muted text is used for timestamps, system messages, and presence lines.

**Per-state visual treatment — described as desired outcomes:**

Each named interaction state from J.2 should have a distinct visual emphasis on the state label:

- `Ready` / `Starting` — neutral, unobtrusive; the system is waiting
- `Listening` — the accent color; the system is actively attending
- `Transcribing` — a cooler, more muted emphasis than listening; processing is underway
- `Thinking` — a calm, mid-intensity treatment; the LLM is working
- `Speaking` — a distinct, positive emphasis; the assistant is producing output
- `Interrupted` / `Recovering` — a warm amber or cautionary tone; the flow was broken
- `Degraded` — the same cautionary tone as interrupted; something is not optimal
- `Failed` — a clear error tone; action may be needed

These treatments are applied via CSS attribute selectors on `data-state` — no inline styles. The implementation must use the token set, not hard-coded values.

**Subsystem readiness indicators — described as desired outcomes:**

Each family row in the readiness panel should carry a small visual marker (a dot, icon, or color-coded element) that communicates ready/degraded/failed at a glance. `ready: true` uses the accent/ready color. `ready: false` with a functional fallback uses the cautionary color. `ready: false` with no usable runtime uses the error color. The marker must be distinguishable without relying on color alone — use shape or label in addition to color.

**HTT recording state — described as desired outcome:**

The PTT button in recording state (`data-capture-state="recording"`) should be visually distinct enough that a user glancing at the window can immediately tell the system is capturing audio. A subtle pulsing or glowing effect using the accent color is appropriate. The effect must be CSS-only — no JS animation loops.

**Conversation log visual hierarchy — described as desired outcome:**

User turns and assistant turns should be visually distinct from each other and from system/presence messages. Each turn entry carries a timestamp and role label that are readable but not dominant. Code content in assistant turns should be visually set apart from prose content (monospace font, distinct background treatment). Presence messages (listening/transcribing/reasoning acknowledgments) should read as ambient — present but lower visual weight than turn content.

**Implementation constraints:**

- All color values, spacing values, and other design decisions must be expressed as CSS custom properties defined in one location (a new file or a clearly delimited section at the top of `style.css`).
- No hard-coded hex, rgb, or hsl values may appear anywhere in `style.css` outside the token definition section.
- No inline styles in HTML or JS.
- No new external dependencies. No CDN resources.
- The implementation must not change any JS behavior — only CSS rules.

## Sub-Slice J.5 — Conversation Log Rendering (Deferred Pending Safety Review)

### Why Here

The current conversation log renders all content as plain text. Code blocks, lists, and structured responses from the LLM display as raw strings. For an assistant used for technical help, this is a meaningful readability gap.

### Deferral Condition

J.5 is **not ready for Agent execution without explicit approval of a rendering approach.** The following safety concerns must be resolved before implementation:

1. **No CDN.** External script delivery is not permitted in this shell per the out-of-scope constraints above. Any markdown renderer must be vendored locally into `desktop/src/vendor/` and committed to the repo.

2. **No unsanitized `innerHTML`.** All content entering `innerHTML` must pass through a sanitization step. The renderer must produce safe HTML for assistant messages only — user input, transcript text, and system messages must use `textContent` exclusively.

3. **Library selection.** A specific local renderer must be approved before J.5 begins. Candidate: `marked.js` vendored locally (single file, no build step). The vendored file must be read from `desktop/src/vendor/` — not fetched from network. The `tauri.conf.json` CSP policy must be confirmed to permit inline script evaluation if `marked` requires it, or a CSP-compatible alternative must be identified.

4. **Sanitizer.** `marked.js` alone does not sanitize output. A separate sanitizer (e.g. `DOMPurify`, also vendored) must be applied to the rendered HTML before injection into the DOM. A single-library approach that handles both rendering and sanitization would be preferred.

If all four conditions are resolved and approved by Architect before J.5 begins, the sub-slice proceeds with the following goal:

### Goal (if approved)

LLM responses that include markdown code blocks, lists, or emphasis render correctly in the conversation log. User input, transcript text, and system messages continue to use `textContent`. A sanitization step is applied to all rendered HTML before DOM injection. Role labels and timestamps remain visible on each turn entry.

### Scope (if approved)

- Vendor the approved library/libraries into `desktop/src/vendor/` and load via `<script src="./vendor/...">` in `index.html`
- Refactor `appendMessage(role, text)` so that `assistant` messages use `renderer.parse(text) → sanitize → innerHTML` and all other roles use `element.textContent = text`
