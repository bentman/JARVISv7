## Slice J тАФ Runtime/Readiness UX + Degraded-State Surfacing

**Date:** 2026-05-15
**Status:** Proposed
**Depends on:** Slice I closed and verified (`SYSTEM_INVENTORY.md`)

---

## Purpose

Group I normalized the acceleration path and proved it end-to-end. The data is correct. The problem is that the desktop shell does not yet surface it in a way that gives a user or operator meaningful situational awareness.

The current shell is a diagnostics surface. It shows a `BOOTSTRAP` state string in a header card, a readiness list rendered into a `<div>`, a `<pre>` block for voice debug output, and a `Hold to Talk` button. State transitions are visible only as a raw text string. Degraded conditions are a body attribute toggle with no readable explanation. No subsystem indicator shows whether STT, TTS, LLM, or Wake is individually ready, degraded, or operating on a fallback device. No named interaction state is visible during a turn. The operator cannot tell from the UI alone why a turn failed, what device is active, or whether the backend is in a degraded but functional condition.

Group J closes this gap by building the information-dense, named-state shell that the product requires. The backend already exposes everything needed тАФ readiness per family with device and reason strings, turn state, wake status, and degraded condition details. This slice consumes that existing data and surfaces it with appropriate fidelity.

This slice is also the prerequisite for Group K (UI controls and operator settings). Controls cannot be built on top of an opaque display surface. The readiness data must be visible and reliable before interactive controls that modify it are added.

## Sub-Slice J.1 тАФ Per-Subsystem Readiness Panel + Degraded-Condition Display

### Why Here

The current readiness panel renders a flat list of family items into a single `<div>`. The operator cannot tell at a glance which specific subsystem is degraded, what device it is using, or why a fallback was taken. This sub-slice replaces that flat display with a structured per-family readiness presentation that exposes the device, runtime, and reason strings already present in the `GET /readiness` payload.

### Goal

The desktop shell sidebar shows a clear per-family readiness row for each of STT, TTS, LLM, and Wake. Each row shows the active runtime, device, and a ready or degraded indicator. When any family is not in a `ready` state, a separate degraded-conditions section lists each affected family with its reason string. No backend or schema changes are required.

### Scope

**`desktop/src/components/readiness-panel.js`** тАФ new module `ReadinessPanel`:

- Accepts the `GET /readiness` payload.
- Renders one row per family showing: family name label, active `runtime/device` string, and a ready or degraded indicator driven by the `ready` field.
- The indicator distinguishes at minimum: `ready`, `degraded` (not ready but a runtime is selected), and `failed` (runtime unavailable). Drive these states from the `ready` flag and the presence of known failure substrings in the `reason` string (e.g. `MISSING`, `unavailable`). The exact substrings to match should be read from the live readiness payload at implementation time to ensure alignment.
- Tooltip or inline expansion on each row shows the full `reason` string.
- Export: `renderReadiness(readinessPayload, containerEl)`.

**`desktop/src/components/degraded-list.js`** тАФ new module `DegradedList`:

- Accepts the `GET /readiness` payload.
- Collects all families where `ready: false` or where the `reason` string indicates a known fallback condition.
- Renders a named degraded-conditions section with one entry per affected family and its reason string.
- The section is hidden entirely when no degraded conditions are present.
- Export: `renderDegradedList(readinessPayload, containerEl)`.

**`desktop/src/index.html`** тАФ restructure the sidebar `<aside>`:

- Replace `<div id="readiness-panel">` with a container for `ReadinessPanel` output.
- Add a degraded-conditions container for `DegradedList` output, positioned below the subsystems section.
- Preserve the existing session info, personality selector, and wake elements тАФ they are refactored in later sub-slices.
- The `<pre id="voice-detail">` debug block is wrapped in a `<details>` element with a short summary label. Collapsed by default; still accessible. This removes it from primary view without losing its content.

**`desktop/src/main.js`** тАФ update call sites:

- Replace the current `renderReadiness(readiness)` function calls with `ReadinessPanel.renderReadiness(readiness, containerEl)` and `DegradedList.renderDegradedList(readiness, degradedEl)`.
- Import component modules at the top of `main.js`.
- The `body[data-degraded]` attribute toggle is preserved for backward compatibility with existing CSS rules.

## Sub-Slice J.2 тАФ Named Interaction State Labels

### Why Here

The current shell displays raw state keys (`BOOTSTRAP`, `READY`, `FAILED`) directly as text. These are implementation identifiers, not user-facing labels. A user cannot tell from `REASONING` that the assistant is thinking, or from `BOOTSTRAP` that the app is starting up. This sub-slice maps existing backend and frontend state values to human-readable labels without changing the state machine itself.

### Goal

Every named state in the current shell is displayed using a clear, user-facing label. The state label element updates during a turn so the user can follow the turn progression without reading raw enum strings. No new states, no state-machine changes, no backend changes.

### Scope

- Defines a mapping from existing state string values to display labels. The mapping must be derived from the actual state strings enumerated in Gate J-2, not invented. At minimum, the following mappings are expected based on current repo behavior:

|---|---|
| `BOOTSTRAP` | Starting |
| `STARTING` | Starting |
| `READY` | Ready |
| `IDLE` | Ready |
| `LISTENING` | Listening |
| `TRANSCRIBING` | Transcribing |
| `REASONING` | Thinking |
| `ACTING` | Acting |
| `RESPONDING` | Responding |
| `SPEAKING` | Speaking |
| `INTERRUPTED` | Interrupted |
| `RECOVERING` | Recovering |
| `DEGRADED` | Degraded |

- Applies a `data-state` attribute to the label element so CSS rules in J.4 can apply distinct visual treatment per state.

Wake status is currently displayed in two `<dd>` elements in the sidebar. The content is informative but not immediately readable: a user must parse the text to understand whether the system is passively monitoring via wake word or requiring manual Hold-to-Talk. The HTT button has no visual active state, no accessible pressed state, and no guard against duplicate capture invocations.

This sub-slice improves both surfaces. No new Tauri commands, no backend changes, no global hotkeys, and no Tauri plugin additions.

### Goal

The wake section of the sidebar shows a clear provider-and-state summary: whether wake word monitoring is active, inactive, or HTT-only fallback. The HTT button has a visible recording state, an accessible pressed attribute, a label that changes during capture, and a guard that prevents a second capture from starting while one is already in progress.

- Shows the reason string when the state is `HTT-only` or when `last_error` is non-null.
- Does not render `last_detected` timestamp or detection count in J.3 тАФ those are operator-diagnostic data deferred to Group K's diagnostics panel.

## Sub-Slice J.4 тАФ v7-Native Visual Polish

### Why Here

J.1тАУJ.3 introduce the structural components and semantic markup that the shell needs. J.4 applies a coherent visual language appropriate for a persistent, dark-themed desktop assistant. Without this, the shell has correct structure but undifferentiated visual weight тАФ ready, degraded, and failed states look the same; interaction states carry no visual feedback; the conversation log has no visual hierarchy.

This sub-slice does not change behavior. It changes how existing behavior looks.

### Goal

The desktop shell presents a coherent dark theme with a small set of v7-native visual tokens. Subsystem status indicators are visually differentiated by readiness state. Named interaction states from J.2 have distinct visual emphasis appropriate to their meaning. The HTT button's recording state is visually obvious. The conversation log has clear turn-level visual hierarchy. All styling is expressed through CSS custom properties defined in one place, with no hard-coded values scattered across the stylesheet.

### Scope

**Visual token set тАФ described as desired outcomes, not as CSS code:**

Introduce a small v7-native visual token set covering: background depth (a progression from base surface to elevated surface), text weight (primary, secondary, and muted), a single accent color used for active/ready states and brand identity, semantic state colors (one per named interaction state тАФ ready, degraded/interrupted, failed, active-capture), border treatment (subtle separation without heavy lines), spacing rhythm, and typography scale. The token set should be minimal тАФ covering only what J.1тАУJ.4 require. Tokens are defined in one location and referenced everywhere else.

**Dark theme тАФ described as desired visual qualities:**

The base surface should read as a deep, neutral dark тАФ not pure black, but dark enough that accent colors read clearly against it. The accent color for ready/active states should be vivid and cool-toned, consistent with the product's technical character. Panel surfaces are slightly lighter than the base, creating depth without high contrast. Text uses a light neutral for primary content and a mid-tone for secondary information. Muted text is used for timestamps, system messages, and presence lines.

**Per-state visual treatment тАФ described as desired outcomes:**

Each named interaction state from J.2 should have a distinct visual emphasis on the state label:

- `Ready` / `Starting` тАФ neutral, unobtrusive; the system is waiting
- `Listening` тАФ the accent color; the system is actively attending
- `Transcribing` тАФ a cooler, more muted emphasis than listening; processing is underway
- `Thinking` тАФ a calm, mid-intensity treatment; the LLM is working
- `Speaking` тАФ a distinct, positive emphasis; the assistant is producing output
- `Interrupted` / `Recovering` тАФ a warm amber or cautionary tone; the flow was broken
- `Degraded` тАФ the same cautionary tone as interrupted; something is not optimal
- `Failed` тАФ a clear error tone; action may be needed

These treatments are applied via CSS attribute selectors on `data-state` тАФ no inline styles. The implementation must use the token set, not hard-coded values.

**Subsystem readiness indicators тАФ described as desired outcomes:**

Each family row in the readiness panel should carry a small visual marker (a dot, icon, or color-coded element) that communicates ready/degraded/failed at a glance. `ready: true` uses the accent/ready color. `ready: false` with a functional fallback uses the cautionary color. `ready: false` with no usable runtime uses the error color. The marker must be distinguishable without relying on color alone тАФ use shape or label in addition to color.

**HTT recording state тАФ described as desired outcome:**

The HTT button in recording state should be visually distinct enough that a user glancing at the window can immediately tell the system is capturing audio. A subtle pulsing or glowing effect using the accent color is appropriate. The effect must be CSS-only тАФ no JS animation loops.

**Conversation log visual hierarchy тАФ described as desired outcome:**

User turns and assistant turns should be visually distinct from each other and from system/presence messages. Each turn entry carries a timestamp and role label that are readable but not dominant. Code content in assistant turns should be visually set apart from prose content (monospace font, distinct background treatment). Presence messages (listening/transcribing/reasoning acknowledgments) should read as ambient тАФ present but lower visual weight than turn content.

**Implementation constraints:**

- All color values, spacing values, and other design decisions must be expressed as CSS custom properties defined in one location (a new file or a clearly delimited section at the top of `style.css`).
- No hard-coded hex, rgb, or hsl values may appear anywhere in `style.css` outside the token definition section.
- No inline styles in HTML or JS.
- No new external dependencies. No CDN resources.
- The implementation must not change any JS behavior тАФ only CSS rules.

## Sub-Slice J.5 тАФ Conversation Log Rendering (Deferred Pending Safety Review)

### Why Here

The current conversation log renders all content as plain text. Code blocks, lists, and structured responses from the LLM display as raw strings. For an assistant used for technical help, this is a meaningful readability gap.

### Deferral Condition

J.5 is **not ready for Agent execution without explicit approval of a rendering approach.** The following safety concerns must be resolved before implementation:

1. **No CDN.** External script delivery is not permitted in this shell per the out-of-scope constraints above. Any markdown renderer must be vendored locally into `desktop/src/vendor/` and committed to the repo.

2. **No unsanitized `innerHTML`.** All content entering `innerHTML` must pass through a sanitization step. The renderer must produce safe HTML for assistant messages only тАФ user input, transcript text, and system messages must use `textContent` exclusively.

3. **Library selection.** A specific local renderer must be approved before J.5 begins. Candidate: `marked.js` vendored locally (single file, no build step). The vendored file must be read from `desktop/src/vendor/` тАФ not fetched from network. The `tauri.conf.json` CSP policy must be confirmed to permit inline script evaluation if `marked` requires it, or a CSP-compatible alternative must be identified.

4. **Sanitizer.** `marked.js` alone does not sanitize output. A separate sanitizer (e.g. `DOMPurify`, also vendored) must be applied to the rendered HTML before injection into the DOM. A single-library approach that handles both rendering and sanitization would be preferred.
