# 20260505-slice_h.md
## Slice H — Voice Acceleration

**Date:** 2026-05-05
**Status:** Proposed
**Depends on:** Slice G closed and verified (`SYSTEM_INVENTORY.md`)

---

## Purpose

Groups A–G establish a fully functional voice-first assistant on CPU across both host classes. The voice path works. The problem this group solves is that it may work too slowly: CPU-only STT on x64 can be acceptable, but ARM64 QNN, x64 CUDA, and Windows DirectML all offer hardware-accelerated inference that could make real-time voice interaction materially faster and more responsive.

Group H proves which accelerated paths are real and activates only those that survive viability gating. Paths that cannot be proven on the current host close as `SKIP-no-host`, `SKIP-prereq-missing`, or `Deferred` — never as `BLOCKED`. CPU voice paths remain the fallback authority for every device that does not reach `PASS`.

## Close-State Vocabulary

Every sub-slice and every device row in H.8 must close using only the following states:

| State | Meaning |
|-------|---------|
| `PASS` | Accelerated path proven: package installed, EP registered, model exported/loaded, live inference correct on this host class |
| `SKIP-no-host` | Required hardware not present on validation host; path is sound but cannot be proven here |
| `SKIP-prereq-missing` | Hardware present but a required operator-installed prerequisite (QAIRT SDK, CUDA toolkit, driver) is absent; path is blocked by environment, not by code |
| `Deferred` | Path is not attempted in Slice H; deferred to a named later group |
| `Degraded-memory-constrained` | Hardware present and prerequisites met but available RAM/VRAM is below the threshold needed to run the accelerated model; CPU fallback is the correct outcome |

`BLOCKED-*` is not an acceptable closeout state. A sub-slice that cannot close must report a concrete stop condition and await direction.

### Why Here

H.3 owns runtime activation: acquire a proven precompiled QNN artifact via Qualcomm AI Hub Models, complete the `QnnWhisperRuntime` encode→decode transcription loop for that artifact, select `qnn` only when readiness and artifact proof are both present, prove a complete voice turn (audio in → transcript out) on ARM64, and preserve CPU fallback.

**H.3 must not begin until:**
1. H.2 closes as `PASS`, AND
2. The H.3 pre-qualification task (`2026-05-11-slice_h.3_pre-qual.md`) closes as **PROCEED** with all PQ-1 through PQ-11 criteria met and the artifact tensor contract documented in `CHANGE_LOG.md`.

If the pre-qual closes as **DEFER** for any named stop condition, H.3 closes as `Deferred` with the pre-qual stop condition recorded.

### Artifact Acquisition Method

**Qualcomm AI Hub Models (`qai_hub_models`) — precompiled_qnn_onnx path.**

The target artifact is `whisper_base_en` exported for `qualcomm-snapdragon-x-elite` as `precompiled_qnn_onnx`. This produces two ONNX files each containing an embedded QNN context binary compiled for the Snapdragon X Elite HTP. These files are loadable with `QNNExecutionProvider` via `onnxruntime-qnn` on Windows ARM64 with no QAIRT SDK compilation step at runtime.

Export is performed once using **x64 Python** (required by `qai_hub_models`; ARM64 Python is not supported for export). The exported artifacts are OS-agnostic and are deployed using the ARM64 `backend/.venv` for inference. Full acquisition procedure is defined in `2026-05-11-slice_h.3_pre-qual.md`.

Target artifact location after pre-qual: `models/stt/whisper-base-en-qnn-snapdragon-x-elite/` (exact subdirectory names per Step 4 inventory in pre-qual `CHANGE_LOG.md` entry).

The STT runtime selects `qnn` only when `ep:QNNExecutionProvider` and artifact catalog proof are both present. CPU fallback is preserved and not degraded.

## Sub-Slice H.7 — TTS Acceleration Viability / Device Slot Normalization

### Why Here

TTS (Kokoro ONNX) has only been proven on CPU. QNN, CUDA, and DirectML are all potential TTS acceleration paths, but each requires separate proof because Kokoro ONNX uses specific op sets and voices binary that may not be supported under every EP. H.7 investigates TTS acceleration viability and normalizes the TTS runtime's `device` slot to be consistent with STT device naming.


### Goal

For each candidate TTS acceleration device (CUDA, DirectML, QNN), determine whether Kokoro ONNX can be loaded and produce synthesis output. IMPORTANT: H.7 requires an explicit package/API viability proof that the Kokoro ONNX runtime (or its Python wrapper) can be instructed to use the target ONNX execution provider (for example by accepting `providers` or equivalent session configuration). Do not assume provider override is available. If provider override is provably unsupported by the Kokoro ONNX package and no upstream fix/export is available within scope, close non-CPU TTS device paths as `Deferred` or `SKIP-prereq-missing`. Normalize the TTS runtime device parameter naming to match STT conventions (`cpu`, `cuda`, `directml`, `qnn`).

### Scope

**TTS device slot normalization (read-verify + minimal adjustment if needed):**
- Confirm `backend/app/runtimes/tts/` runtime constructor accepts `device` parameter with values `cpu`, `cuda`, `directml`, `qnn`.
- If device naming is inconsistent with STT, correct it. This is a naming/contract normalization, not a functional change.
- CPU device path must remain untouched and verified green.

**TTS CUDA viability probe:**
- Verify the Kokoro ONNX loader/package exposes a way to select the ONNX `CUDAExecutionProvider` (for example by accepting `providers` into an ONNX session or via a documented API). If the package does not expose provider selection and no in-scope wrapper exists to enforce it, mark the CUDA path `SKIP-prereq-missing` or `Deferred` with the reason "provider-override-missing".
- If provider selection is available, attempt to load Kokoro ONNX model with `CUDAExecutionProvider` on x64 NVIDIA host.
- Produce a short synthesis output and confirm the audio array is non-empty and non-NaN.
- Close as `PASS` if successful, `SKIP-no-host` if CUDA host unavailable, `SKIP-prereq-missing` if Kokoro op set is incompatible.

**TTS DirectML viability probe:**
- Verify the Kokoro ONNX loader/package exposes a way to select the ONNX `DmlExecutionProvider`. If provider selection is unavailable, mark DirectML path as `SKIP-prereq-missing` or `Deferred` with reason "provider-override-missing".
- If provider selection is available, attempt to load Kokoro ONNX model with `DmlExecutionProvider` on DirectML-capable host.
- Same acceptance criteria as CUDA viability probe.
- Close as `PASS`, `SKIP-no-host`, `SKIP-prereq-missing`, or `Deferred` based on result.

**TTS QNN viability probe (ARM64 only):**

If provider selection is not possible, close TTS QNN as `Deferred`.
- Attempt load only if a quantized Kokoro ONNX is available from H.2 work or the model catalog.
- If no quantized Kokoro artifact exists, close TTS QNN as `Deferred`.

**New tests:**
- `backend/tests/runtime/voice/test_tts_live.py` — extend: `test_tts_cuda_live_synthesis_produces_audio` (`skip_unless_cuda`), `test_tts_directml_live_synthesis_produces_audio` (`skip_unless_directml`)
