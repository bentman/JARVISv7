# 20260505-slice_h.md
## Slice H — Voice Acceleration

**Date:** 2026-05-05
**Status:** Proposed
**Depends on:** Slice G closed and verified (`SYSTEM_INVENTORY.md`)

---

## Purpose

Groups A–G establish a fully functional voice-first assistant on CPU across both host classes. The voice path works. The problem this group solves is that it may work too slowly: CPU-only STT on x64 can be acceptable, but ARM64 QNN, x64 CUDA, and Windows DirectML all offer hardware-accelerated inference that could make real-time voice interaction materially faster and more responsive.

Group H proves which accelerated paths are real and activates only those that survive viability gating. Paths that cannot be proven on the current host close as `SKIP-no-host`, `SKIP-prereq-missing`, or `Deferred` — never as `BLOCKED`. CPU voice paths remain the fallback authority for every device that does not reach `PASS`.

## Close-State Vocabulary

Every sub-slice and every device row in H.8 must close using only the following states:

| State | Meaning |
|-------|---------|
| `PASS` | Accelerated path proven: package installed, EP registered, model exported/loaded, live inference correct on this host class |
| `SKIP-no-host` | Required hardware not present on validation host; path is sound but cannot be proven here |
| `SKIP-prereq-missing` | Hardware present but a required operator-installed prerequisite (QAIRT SDK, CUDA toolkit, driver) is absent; path is blocked by environment, not by code |
| `Deferred` | Path is not attempted in Slice H; deferred to a named later group |
| `Degraded-memory-constrained` | Hardware present and prerequisites met but available RAM/VRAM is below the threshold needed to run the accelerated model; CPU fallback is the correct outcome |

`BLOCKED-*` is not an acceptable closeout state. A sub-slice that cannot close must report a concrete stop condition and await direction.

## Sub-Slice H.2 — ARM64 QNN Prerequisite Gate

### Why Here

QNN STT activation (H.3) cannot proceed without proving that the prerequisite chain is complete. This gate is ARM64-specific and Qualcomm-specific. The gate also requires proving quantized Whisper model export on x64 first — export tooling is more stable on x64, and the exported artifacts are portable to ARM64 for inference.

This is a prerequisite proof gate, not a full activation sub-slice.

### Goal

Prove on x64 that a quantized Whisper ONNX model (int8 or INT8-QDQ) can be exported and loaded with `CPUExecutionProvider` as the export validation baseline. Separately prove on ARM64 that:
1. `onnxruntime-qnn` imports cleanly.
2. QAIRT backend DLLs are discoverable via `QAIRT_SDK_PATH`.
3. `QNNExecutionProvider` registers in the ONNX runtime session.
4. The quantized model can be opened in an ONNX session with `QNNExecutionProvider`.

The gate closes as `PASS` only when all four ARM64 proofs succeed. If any proof fails, the gate closes as `SKIP-prereq-missing` with the missing component named explicitly.

### Scope

**x64 quantized Whisper export proof:**
- Export or acquire a quantized (int8 / INT8-QDQ) Whisper ONNX encoder+decoder artifact.
- Validate load with `CPUExecutionProvider` on x64 using `onnxruntime`.
- This is export-tool validation only; the model may live in `models/stt/` under a new subdirectory (e.g., `whisper-small-onnx-int8/`) or alongside the existing float model.
- Export script or acquisition command must be documented in the CHANGE_LOG entry for H.2.

**ARM64 QNN prerequisite proof (in `backend/app/hardware/preflight.py`):**
- `dll:QnnHtp` token: confirm `QnnHtp.dll` is discoverable via `QAIRT_SDK_PATH` env var path on the ARM64 host.
- `ep:QNNExecutionProvider`: confirm `QNNExecutionProvider` appears in `onnxruntime.get_available_providers()` after DLL bootstrap on ARM64.
- ONNX session with `QNNExecutionProvider`: open the quantized Whisper model artifact in a session using `QNNExecutionProvider` on ARM64 without error.

**New unit tests (`backend/tests/unit/hardware/test_qnn_prerequisite.py`):**
- `test_qnn_dll_token_present_when_path_configured` — synthetic DLL path configured, token expected
- `test_qnn_ep_token_present_when_ep_available` — mocked `get_available_providers()` includes QNN, token expected
- `test_qnn_ep_token_missing_when_ep_absent` — mocked `get_available_providers()` excludes QNN, token expected missing

**Live gate test (`backend/tests/runtime/hardware/test_qnn_gate_live.py`):**
- `@pytest.mark.live @pytest.mark.arm64 @pytest.mark.qnn`
- `test_qnn_dll_discoverable_via_qairt_sdk_path` — `skip_unless_arm64`; `skip_unless_qnn`
- `test_qnn_ep_registers_in_onnxruntime` — `skip_unless_arm64`; `skip_unless_qnn`
- `test_quantized_whisper_loads_with_qnn_ep` — `skip_unless_arm64`; `skip_unless_qnn`; loads the artifact exported in x64 proof

### Files Likely Touched

### Files Not to Modify

- `backend/app/hardware/profiler.py` — profiler is read-only here; preflight extension is the correct location

### Risks / Stop Conditions

## Sub-Slice H.7 — TTS Acceleration Viability / Device Slot Normalization

### Why Here

TTS (Kokoro ONNX) has only been proven on CPU. QNN, CUDA, and DirectML are all potential TTS acceleration paths, but each requires separate proof because Kokoro ONNX uses specific op sets and voices binary that may not be supported under every EP. H.7 investigates TTS acceleration viability and normalizes the TTS runtime's `device` slot to be consistent with STT device naming.


### Goal

For each candidate TTS acceleration device (CUDA, DirectML, QNN), determine whether Kokoro ONNX can be loaded and produce synthesis output. IMPORTANT: H.7 requires an explicit package/API viability proof that the Kokoro ONNX runtime (or its Python wrapper) can be instructed to use the target ONNX execution provider (for example by accepting `providers` or equivalent session configuration). Do not assume provider override is available. If provider override is provably unsupported by the Kokoro ONNX package and no upstream fix/export is available within scope, close non-CPU TTS device paths as `Deferred` or `SKIP-prereq-missing`. Normalize the TTS runtime device parameter naming to match STT conventions (`cpu`, `cuda`, `directml`, `qnn`).

### Scope

**TTS device slot normalization (read-verify + minimal adjustment if needed):**
- Confirm `backend/app/runtimes/tts/` runtime constructor accepts `device` parameter with values `cpu`, `cuda`, `directml`, `qnn`.
- If device naming is inconsistent with STT, correct it. This is a naming/contract normalization, not a functional change.
- CPU device path must remain untouched and verified green.

**TTS CUDA viability probe:**
- Verify the Kokoro ONNX loader/package exposes a way to select the ONNX `CUDAExecutionProvider` (for example by accepting `providers` into an ONNX session or via a documented API). If the package does not expose provider selection and no in-scope wrapper exists to enforce it, mark the CUDA path `SKIP-prereq-missing` or `Deferred` with the reason "provider-override-missing".
- If provider selection is available, attempt to load Kokoro ONNX model with `CUDAExecutionProvider` on x64 NVIDIA host.
- Produce a short synthesis output and confirm the audio array is non-empty and non-NaN.
- Close as `PASS` if successful, `SKIP-no-host` if CUDA host unavailable, `SKIP-prereq-missing` if Kokoro op set is incompatible.

**TTS DirectML viability probe:**
- Verify the Kokoro ONNX loader/package exposes a way to select the ONNX `DmlExecutionProvider`. If provider selection is unavailable, mark DirectML path as `SKIP-prereq-missing` or `Deferred` with reason "provider-override-missing".
- If provider selection is available, attempt to load Kokoro ONNX model with `DmlExecutionProvider` on DirectML-capable host.
- Same acceptance criteria as CUDA viability probe.
- Close as `PASS`, `SKIP-no-host`, `SKIP-prereq-missing`, or `Deferred` based on result.

**TTS QNN viability probe (ARM64 only):**
- Kokoro ONNX is typically a float model; QNN commonly requires quantized inputs. The QNN TTS path is likely `Deferred` unless a quantized Kokoro export exists and is discoverable via the model catalog.
- Additionally, confirm the Kokoro ONNX loader/package can accept provider selection for QNN (for example by honoring `providers=['QNNExecutionProvider']`) or that the runtime's preflight/token evidence shows QNN EP registration usable by the package. If provider selection is not possible, close TTS QNN as `Deferred`.
- Attempt load only if a quantized Kokoro ONNX is available from H.2 work or the model catalog.
- If no quantized Kokoro artifact exists, close TTS QNN as `Deferred`.

**New tests:**
- `backend/tests/runtime/voice/test_tts_live.py` — extend: `test_tts_cuda_live_synthesis_produces_audio` (`skip_unless_cuda`), `test_tts_directml_live_synthesis_produces_audio` (`skip_unless_directml`)
- `backend/tests/unit/runtimes/tts/test_tts_runtime.py` — extend: `test_tts_device_slot_accepts_cuda_string`, `test_tts_device_slot_accepts_directml_string`, `test_tts_device_slot_accepts_qnn_string`

**`backend/app/hardware/readiness.py` — extend `derive_tts_device_readiness()`:**
- Add CUDA and DirectML candidates using the same proven-EP logic as STT.
- CPU remains fallback when no accelerated EP is proven.

### Files Likely Touched

- `backend/app/runtimes/tts/` (device slot normalization + accelerated branches where proven)
- `backend/app/hardware/readiness.py`
- `backend/tests/unit/runtimes/tts/test_tts_runtime.py`
- `backend/tests/runtime/voice/test_tts_live.py`

### Files Not to Modify

- STT runtime — no cross-family changes
- Kokoro model artifacts — read-only; no re-download or re-export unless explicitly scoped by a new `ensure_models.py` step

### Risks / Stop Conditions

- If Kokoro ONNX uses ops not supported by a target EP, that device path is `SKIP-prereq-missing`. Document the unsupported op and close that device path. Do not patch the model.
- TTS acceleration is lower priority than STT acceleration for voice-first responsiveness (STT latency is the dominant factor in perceived response time). If TTS acceleration proof is complex, consider `Deferred` for H.7 TTS CUDA/DirectML and close H.7 as a normalization-only sub-slice. This is preferable to delaying H.8.
- Do not add a new TTS runtime family. All acceleration is a device branch inside the existing Kokoro ONNX runtime.

### Highest Relevant Validation Surface

Live synthesis proof on accelerated host: `test_tts_cuda_live_synthesis_produces_audio` or `test_tts_directml_live_synthesis_produces_audio`. If no accelerated host available, unit device-slot normalization tests are sufficient for device-slot normalization closure. `validate_backend.py regression` ≥96.

## Slice-Level Closeout Criteria

Slice H is complete when all of the following are true:

1. H.0 state table populated for both host classes with no empty cells.
2. H.1 CPU STT baseline confirmed on both host classes post-G; regression ≥96 on both.

9. H.8 voice acceleration matrix is complete: every (family, device, host class) cell has a close state from the vocabulary. No `BLOCKED-*` cells.
10. H.8 live turn matrix: every `PASS`-rated device path has a live full-turn test result.
11. `validate_backend.py regression` ≥96 on both host classes at H.8 closeout.
12. `CHANGE_LOG.md` has one entry per sub-slice H.0–H.8 with host class(es) and live proof cited (or `SKIP-*` / `Deferred` reason if live proof was not run).
13. `SYSTEM_INVENTORY.md` updated at Slice H group closeout. State: `Verified`.
14. No Group I, L, M, or unrelated work introduced.
