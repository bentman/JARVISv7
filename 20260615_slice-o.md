# 2026-06-15 Slice O — Agent Framework

Status: planning document only. No runtime implementation is included in this slice document.

## Purpose

Define the Agent Framework as an optional, policy-gated orchestration layer over the existing JARVISv7 conversation stack. Agents must compose the current turn engine, tool registry, memory, prompt, artifact, and API surfaces; they must not replace or fork those systems.

This slice expands `# Group O — Agent Framework` in `slices.md` after inspecting the current repository structure and runtime surfaces.

## Repository Alignment

Current implemented foundations:

- Turn execution lives in `backend/app/conversation/engine.py`.
- Session lifecycle and current-session continuity live in `backend/app/conversation/session_manager.py`, `backend/app/conversation/continuity.py`, and `backend/app/conversation/continuity_policy.py`.
- Prompt assembly and rendering live in `backend/app/cognition/prompt_assembler.py`, `backend/app/cognition/prompt_envelope.py`, and `backend/app/cognition/prompt_renderer.py`.
- Deterministic tool execution lives in `backend/app/cognition/executor.py` and `backend/app/tools/registry.py`.
- Tool adapters live under `backend/app/tools/`.
- Turn artifacts already include `agent_trace` in `backend/app/artifacts/turn_artifact.py`.
- Session artifacts and timelines live in `backend/app/artifacts/session_artifact.py`, `backend/app/artifacts/session_timeline.py`, and `backend/app/artifacts/storage.py`.
- Working memory, episodic storage, and retrieval live in `backend/app/memory/`.
- The backend API already includes a disabled read-only agent status route at `backend/app/api/routes/agents.py` with schema `backend/app/api/schemas/agents.py`.
- `config/agents/`, `config/prompts/`, and `data/agents/` exist as placeholder directories with `.gitkeep`.
- `config/app/policies.yaml` exists and currently has LLM/cloud policy entries only.
- `pyproject.toml` already declares the pytest marker `agents`.

Current absent surfaces:

- There is no `backend/app/agents/` runtime package.
- There are no agent role contracts, agent message models, agent ledger, agent orchestrator, or agent runtime services.
- There is no agent policy key in `config/app/policies.yaml`.
- There are no agent tests under `backend/tests/unit/agents/` or `backend/tests/integration/agents/`.
- The existing `/agents/status` response is a stub and must remain truthful until Group O activates a real agent surface.

## Non-Negotiable Boundaries

- Non-agent turns must continue to work unchanged through `TurnEngine.run_text_turn()` and `TurnEngine.run_voice_turn()`.
- Agent behavior must be explicit opt-in through policy and request/runtime routing. No hidden autonomous agent path may be introduced.
- Agents consume existing tools through the deterministic `ToolExecutor` / `ToolRegistry` boundary. No model-side freeform tool execution or unregistered tool dispatch.
- Agents consume memory through existing working memory, continuity packets, episodic memory, and retrieval APIs. No semantic/vector memory in Group O.
- Agents record trace data through `TurnArtifact.agent_trace` and any agent ledger records. No untraceable agent state.
- Redis, if used later, may only accelerate lookup/caching. Durable coordination authority remains disk/SQLite-backed under `data/agents/`.
- No dependency changes are expected for the base agent framework. SQLite is available through the Python standard library if ledger persistence uses it.
- No changes to hardware detection, provisioning, STT/TTS runtime selection, wake, desktop UI redesign, telephony, streaming, model routing, or semantic memory.

**Roles.** Planner, Executor, Critic, Curator, Learner.

**Contract rules.**

- Messages are typed data, not rendered chat transcripts.
- Role definitions are config-loaded and schema-validated.
- Role prompts are role-scoped resources under `config/prompts/agents/`.
- Role contracts may reference the conversation stack but must not instantiate `TurnEngine`, STT, TTS, wake, or model routing directly.
- Role output must be serializable for `agent_trace` and ledger persistence.
- This sub-slice defines contracts only; it does not execute agents, call tools, alter prompts, or change turn behavior.

**Acceptance.**

- Unit tests validate message schema, role config loading, role identifiers, and serialization.
- Existing non-agent tests remain unchanged.
- `/agents/status` remains disabled/read-only unless O.3 changes the status truthfully.
- No agent execution behavior exists or is claimed.

## O.2 — Agent Ledger / Artifact Trail

**Goal.** Add durable agent ledger or artifact records for future agent events, plans, outcomes, and policy decisions.

**Code add.**

- `backend/app/agents/ledger.py`
- `backend/app/agents/models.py` if shared ledger records are large enough to justify a separate module
- `data/agents/`
- `backend/tests/unit/agents/test_ledger.py`

**Ledger rules.**

- Durable ledger state is local and deterministic, preferably SQLite under `data/agents/`.
- Ledger records include trace id, session id when available, turn id when available, role id, message type, payload, timestamp, and parent/correlation ids.
- Ledger writes are append-oriented. Corrections are new records, not mutation of prior records, unless a narrow status field is required and tested.
- Redis is not durable authority for Group O. If used later, it is cache-only and must fail closed to the durable ledger.
- Ledger APIs must support query by trace id and by session/turn id for artifact reconstruction.
- Ledger writes do not imply agent execution. No hidden or background behavior is introduced.

**Acceptance.**

- Unit tests prove write/query/reopen behavior.
- Unit tests prove trace reconstruction order.
- Unit tests prove future event/plan/outcome/policy-decision records are durable and artifact-compatible.
- No live turn path calls the ledger for hidden work.
