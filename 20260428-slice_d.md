## Why This Group Exists Here

Group C proved the canonical turn engine works through validated Slice C runtime and service surfaces. Voice and text share one engine, sessions produce artifacts, interruption is deterministically tested, and the proving host (`scripts/run_jarvis.py`) makes failures visible. What does not yet exist is a durable application surface — a native desktop host that a user can launch, interact with continuously, and rely on across turns.

Group D builds that surface exactly once. The shell consumes the backend through an explicit API contract established in D.1. The Tauri native desktop application lands in D.2. Continuous resident interaction arrives in D.3. Wake-word invocation integrates into the resident loop in D.4. Personality and presence polish close the group in D.5.

The ordering is deliberate:
- D.1 fixes the backend-to-shell contract before any Tauri code is written, so the shell cannot accidentally own orchestration logic.
- D.2 builds the shell once against that fixed contract, on both host classes simultaneously.
- D.3 and D.4 extend the resident path incrementally rather than redesigning it.
- D.5 polishes without touching cognition or adding agents or tools.

After Group D, the application is usable as a continuous voice assistant on both supported Windows host classes. Everything in Groups E–I is capability extension on top of a working durable product.

## Group Goal

A user on either Windows x64 or Windows ARM64 launches JARVISv7 as a native desktop application, speaks naturally across multiple turns without manual re-invocation, invokes the assistant by wake word, sees hardware and runtime selection clearly, and experiences a consistent personality across turns. The backend remains authoritative. The desktop shell is a thin adapter.

## Sub-Slice D.1 — Application Shell Contract

### Why Here

Before a single line of Tauri code is written, the backend must expose an explicit, typed HTTP contract. If the contract is defined after the shell is built, the shell will inevitably own logic that belongs in the backend.

### Goal

A headless HTTP client (curl, httpx, or a test fixture) drives the full application lifecycle — startup, session management, text and voice turns, diagnostics — without any shell involvement. The shell in D.2 consumes this contract and nothing else.

### Scope

**`scripts/run_backend.py`** — if not already present: starts the FastAPI application via Uvicorn on a configurable host/port. Accepts `--host`, `--port`, `--reload` (dev mode). Emits the host-fingerprint line as the first stdout line per AGENTS.md §7. Calls the profiler, preflight, and runtime selectors at startup and injects them into the FastAPI application state.

**`backend/app/api/dependencies.py`** — FastAPI dependency providers:
- `get_engine() -> TurnEngine` — returns the application-level `TurnEngine` instance.
- `get_session_manager() -> SessionManager` — returns the application-level `SessionManager`.
- `get_personality() -> PersonalityProfile` — returns the active personality.
- `get_preflight() -> PreflightResult` — returns the cached preflight result.
- `get_profile() -> HardwareProfile` — returns the hardware profile.

**`backend/app/api/schemas/`** — Pydantic request/response models:
- `common.py` — `ErrorResponse`, `HealthResponse`
- `readiness.py` — `ReadinessResponse`: per-family runtime + device + model + ready + reason; active personality profile id; active LLM runtime name; degraded mode indicators
- `session.py` — `CreateSessionRequest`, `CreateSessionResponse`, `CloseSessionRequest`, `CloseSessionResponse`
- `task.py` — `TextTurnRequest`, `TextTurnResponse` (transcript, response_text, final_state, failure_reason, turn_id, session_id)
- `voice.py` — `VoiceTurnResponse` (same fields as text plus tts_degraded, interrupted)

**`backend/app/api/routes/`** — one file per route group:
- `health.py` — `GET /health` → `HealthResponse`. Always returns 200 if backend is running.
- `readiness.py` — `GET /readiness` → `ReadinessResponse`. Emits the structured startup summary consumed by the shell.
- `session.py` — `POST /session/create` → `CreateSessionResponse`; `POST /session/close` → `CloseSessionResponse`.
- `task.py` — `POST /task/text` → `TextTurnResponse`. Accepts `TextTurnRequest`, runs `turn_service.run_text_turn`.
- `voice.py` — `POST /task/voice` with multipart audio upload → `VoiceTurnResponse`. Accepts raw WAV bytes, decodes to numpy, runs `turn_service.run_voice_turn`.
- `diagnostics.py` — `GET /diagnostics/profile` → hardware profile JSON; `GET /diagnostics/preflight` → preflight token list and probe errors.

- `agents.py` — `GET /agents/status` → `{enabled: false, reason: "Group I not implemented"}`. Required by `slices.md`; returns a consistent disabled response in D.1 so D.2 shell and later can call it without a 404.
- `status.py` — `GET /status/wake` → `{provider: str, available: bool, reason: str}`. Defining this in D.1 avoids D.4 introducing an ad hoc endpoint outside the declared contract.
- Voice route receives audio as bytes over HTTP, not a live microphone connection. Live microphone capture remains in `voice_service.capture_audio()` used by the resident loop (D.3). The shell POSTs captured audio to this endpoint.

### Files Not to Modify

- `backend/app/runtimes/**`, `backend/app/hardware/**`

### Key Design Decisions

- API schemas are declared once in `backend/app/api/schemas/` and treated as a compatibility boundary with the desktop shell from D.2 onward. Later slices may add optional response fields with defaults; they must not rename or remove existing fields.
- Session lifecycle is explicit: `POST /session/create` before any turns; `POST /session/close` after. The shell does not manage session state internally.
- Voice audio upload over HTTP is intentionally synchronous in D.1. Streaming WebSocket voice is future scope (not in any current group).
- Backend owns all orchestration; no orchestration logic in route handlers. Handlers call services and return responses.

## Sub-Slice D.2 — Durable Desktop Host

### Why Here

The shell lands once on the durable surface, consuming D.1's fixed HTTP contract. The proving host (`scripts/run_jarvis.py`) is explicitly demoted to developer/diagnostic scaffold at this slice's landing. `desktop/` is the durable surface from this point onward.

### Goal

A user on either Windows host class launches JARVISv7 as a native desktop application, sees hardware and runtime selection in the tray/UI, and completes a voice turn via PTT. Both host classes build natively without cross-compilation.

### Dev/Build Prerequisites by Host Architecture

These are documented in `README.md` and `desktop/README.md`. The slice doc records them here for implementation reference.

**Windows x64 developer host:**
- Rust stable toolchain via [rustup.rs](https://rustup.rs/)
- Node.js LTS (x64 build)
- MSVC v143 build tools — "Desktop development with C++" workload in Visual Studio 2022 or Build Tools
- WebView2 runtime (present on Windows 11; install Evergreen Bootstrapper on Windows 10)

**Windows ARM64 developer host:**
- Rust stable toolchain via rustup + `rustup target add aarch64-pc-windows-msvc`
- Node.js LTS (ARM64 build where available; x64 runs via emulation)
- **MSVC v143 — VS 2022 C++ ARM64 build tools**: in Visual Studio Installer, under the "Desktop development with C++" workload, select "MSVC v143 - VS 2022 C++ ARM64 build tools (Latest)". Per Tauri's Windows installer documentation this is required for ARM64 native builds.
- WebView2 runtime (present on Windows 11 ARM64 by default)

**Cross-compile x64 → ARM64** (documented escape hatch; not the primary dev path):
```
tauri build --target aarch64-pc-windows-msvc --bundles nsis
```
The NSIS installer runs via emulation on ARM; the bundled application binary is native ARM64.

### Scope

**`desktop/src-tauri/`** — Tauri 2.x Rust application:
- `main.rs` — application entry point; manages backend process lifecycle via `scripts/run_backend.py`; exposes window.
- `lib.rs` — Tauri app setup, command registration, system tray initialization.
- `backend.rs` — backend process lifecycle: spawn `run_backend.py` child process on app start; kill on app exit; expose port as Tauri state.
- `tray.rs` — system tray with named visible states: Starting, Ready, Listening, Transcribing, Thinking, Speaking, Interrupted, Degraded, Failed. Each state maps to a distinct icon or label.
- `Cargo.toml` — Tauri 2.x dependency; `tauri-build` build script.
- `tauri.conf.json` — Tauri configuration: app name, bundle identifier, window settings, tray behavior, allowed HTTP origins (localhost only for backend).
- `build.rs` — Tauri build script.

**`desktop/src/`** — web UI layer (vanilla HTML/CSS/JS or a minimal framework; keep thin):
- `index.html` — application window shell.
- `main.js` — fetches `/readiness` on load; populates hardware/runtime summary; drives PTT button; handles `/task/text` and `/task/voice` calls; updates state display.
- `style.css` — minimal styling.
- `components/` — at minimum:
  - `status/` — tray and in-window state display; maps to the 9 canonical `ConversationState` values visible to the user (IDLE, LISTENING, TRANSCRIBING, REASONING, RESPONDING, SPEAKING, INTERRUPTED, DEGRADED, FAILED).
  - `hardware-selection/` — read-only display of active family + device per STT/TTS/LLM/Wake, pulled from `/readiness` on startup.
  - `conversation/` — conversation history display; appends each turn's transcript and response.
  - `llm-selection/` — **read-only** display of the active LLM runtime, pulled from `/readiness`. No interactive switching in D.2; no backend policy endpoint for in-session switching is defined yet. If policy switching is added in D.5, this component is updated then.

**`desktop/README.md`** — extend with arch-split dev prerequisites per the Gate D-G1/D-G2 findings. This file already exists; extend it.

### Files Not to Modify

- `backend/app/**` — backend is consumed over HTTP, not modified
- `scripts/run_jarvis.py` — proving host remains available; it is not removed, just no longer the primary user-facing surface

### Key Design Decisions

- Backend process is a child of the Tauri application. Tauri's `Command` API spawns `run_backend.py` on app start and terminates it on app exit. The shell communicates with the backend exclusively over HTTP on localhost.
- Shell UI calls `/readiness` once at startup. Runtime/hardware selection information is read-only in the shell; changes require backend restart (acceptable for D.2 scope).
- PTT and audio ownership per Gate D-G3: Shell captures via `getUserMedia`, sends WAV bytes to `POST /task/voice`, plays synthesized audio via Web Audio API if returned, displays text if TTS degraded. This is the only D.2 audio path; no backend-side playback path runs concurrently.
- LLM selection display is read-only in D.2. No backend policy endpoint for in-session LLM switching is defined until D.5.
- Tray state is updated by observing turn `final_state` in `VoiceTurnResponse`. No polling endpoint introduced in D.2.

### Acceptance

- Native dev build succeeds on Windows x64: `cd desktop && pnpm tauri dev` produces a running window with no compile errors.
- Native dev build succeeds on Windows ARM64 using the ARM64 toolchain (D-G1 confirmed).
- Desktop live on both host classes: launch app → system tray visible → PTT → speak → transcript and response appear in conversation display → tray transitions to SPEAKING if TTS available → returns to IDLE.
- Hardware selection component displays active device per family from `/readiness`.

### Finish Line

A user on either Windows host class interacts with JARVISv7 through a native desktop application window. `README.md` documents arch-split build prerequisites. `desktop/README.md` is updated with the verified toolchain details. The proving host (`scripts/run_jarvis.py`) is explicitly documented as developer-only.

### Risks

- **PTT audio capture in Tauri WebView**: `navigator.mediaDevices.getUserMedia` requires the page to be served over HTTPS or localhost. Tauri's localhost dev server satisfies this. Confirm before implementation.
- **Audio playback**: backend returns synthesized audio bytes in the response; shell plays via Web Audio API. This is the chosen strategy per D-G3. Backend-side `playback.play()` is not called for D.2 shell turns.
- **Tray state polling latency**: without a push mechanism, tray state updates lag by one poll interval. Acceptable for D.2; noted for D.3/D.4 improvement.

## Sub-Slice D.3 — Resident Session Loop

### Why Here

D.2 proves the durable shell works for a single PTT-initiated turn. D.3 closes the gap to continuous interaction without the user needing to manually re-invoke each turn. This is where JARVISv7 begins to feel like a persistent assistant rather than a single-turn tool.

### Goal

Three or more consecutive voice turns complete in one session on both host classes without any manual re-invocation between turns. The desktop shell's tray and conversation display update continuously.

### Scope

**`backend/app/services/session_service.py`** — new service, created in D.3:
- `SessionService` manages a single active session across turns.
- `start_session() -> str` — creates a session via `SessionManager`; returns `session_id`.
- `end_session()` — closes the active session via `SessionManager`.
- `is_session_active() -> bool`
- Provides the lifecycle boundary for the resident loop: session is created once on app start (or first invocation), persists across turns, closes on app shutdown.
- Injected into the FastAPI app state and exposed via dependency in `api/dependencies.py`.

**Extend `backend/app/api/routes/session.py`**:
- `GET /session/status` → current session id, turn count, state.

**Desktop shell updates:**
- `main.js` — on app start, call `POST /session/create` once; on each PTT release, call `POST /task/voice` with audio; display response; loop without re-invoking create. On app close or explicit stop, call `POST /session/close`.
- Conversation display appends each turn sequentially in the same session view.
- Tray reflects `is_session_active` state: active session = distinct icon vs no-session idle.

**`backend/tests/unit/services/test_session_service.py`**:
- `test_start_session_creates_session_id`
- `test_end_session_marks_inactive`
- `test_is_session_active_reflects_state`

**`backend/tests/runtime/desktop/test_resident_loop_live.py`** — `@pytest.mark.live @pytest.mark.desktop`, `skipif SKIP_UNLESS_LIVE`:
- `test_three_consecutive_turns_in_resident_session_complete_without_error` — via HTTP to running backend; uses text turns (no live mic required for API-level proof); requires_ollama

### Files Not to Modify

- `TurnEngine`, `SessionManager`, `TurnArtifact` — consumed, not changed
- `scripts/run_jarvis.py` — proving host unchanged

### Key Design Decisions

- The resident session is started once and persists across turns. This is a different ownership model from D.2's single-PTT flow: in D.3, session creation moves from per-turn to per-session, and the shell no longer calls `POST /session/create` before each turn.
- `SessionService` owns the active session reference; `SessionManager` owns session-level artifact write. The service layer does not duplicate session logic.
- Working memory from C.3 flows naturally: `SessionManager.get_working_context()` is called per turn as before; continuity is a side effect of the resident session, not new infrastructure.

### Acceptance

- Desktop live on both host classes: three+ consecutive voice turns in one session, each turn's transcript and response visible in the conversation display, tray state transitions per turn without manual re-invocation.
- API-level resident loop proven: `test_resident_loop_live.py` passes with three text turns via HTTP on both host classes.
- Regression: 74+ tests green.

### Finish Line

JARVISv7 maintains a session across multiple turns without requiring the user to reinitiate. The desktop shell conversation view grows per turn in one session.

### Risks

- Audio device state across turns in the native shell: microphone may hold state between PTT presses. Document whether `sounddevice` or Web Audio API device handling requires explicit release/reacquire between turns. If so, handle it in the shell's PTT release handler.

## Sub-Slice D.4 — Wake Integration in Durable Host

### Why Here

PTT is sufficient for D.2/D.3 but is not the primary voice-first invocation model. Wake-word activation wires the B.4 wake runtime into the resident loop so the user can speak "hey jarvis" to invoke the assistant without touching hardware.

### Goal

"Hey jarvis" triggers LISTENING within the durable host on both host classes via openWakeWord. PTT remains available. Wake-unavailable degrades cleanly to PTT-only with a visible indicator.

### Scope

**`backend/app/services/session_service.py`** — extend with wake monitoring:
- `start_wake_monitoring(wake_runtime: WakeBase)` — begins a background thread or coroutine that continuously feeds microphone audio chunks to `wake_runtime.detect()`.
- `stop_wake_monitoring()` — stops the wake monitoring thread cleanly.
- On positive detection: signals the resident loop to transition to LISTENING; queues a voice capture + turn cycle.

**`backend/app/services/voice_service.py`** — extend:
- `stream_audio_chunks(chunk_duration_s: float) -> Iterator[np.ndarray]` — yields small audio chunks for wake monitoring. Used by `session_service.start_wake_monitoring()`.

**Backend API extension:**
- `GET /status/wake` → `{provider: str, available: bool, reason: str}`. Shell reads this to show/hide PTT-only degraded indicator.

**Desktop shell updates:**
- When `/status/wake` returns `available: false`, display a PTT-only indicator in the tray and conversation area.
- When wake is available, the PTT button remains as a manual override; the wake loop drives LISTENING automatically.

**`backend/tests/unit/services/test_session_service.py`** — extend:
- `test_wake_monitoring_start_and_stop_clean`
- `test_wake_detection_positive_triggers_listening_transition` (mock `WakeBase`)
- `test_wake_unavailable_degrades_to_ptt_only_without_error`

**`backend/tests/runtime/desktop/test_wake_integration_live.py`** — `@pytest.mark.live @pytest.mark.desktop @pytest.mark.wake`, `skipif SKIP_UNLESS_LIVE`:
- `test_wake_monitoring_starts_and_stops_cleanly_on_current_host`
- `test_hey_jarvis_fixture_triggers_detection_in_resident_loop` — uses `hey_jarvis_fixture.wav` fed as a simulated chunk; does not require open microphone

### Files Not to Modify

- `backend/app/runtimes/wake/**` — wake runtimes consumed, not changed
- `BargeInDetector` — barge-in remains mic-during-speech; wake-during-speech is documented as future polish (not in D.4 scope)
- `TurnEngine`, `SessionManager`, `TurnArtifact`

### Key Design Decisions

- Wake monitoring runs in a daemon thread managed by `SessionService`. It does not block the main backend event loop.
- Wake-word detection (openWakeWord) and barge-in detection (BargeInDetector) are separate responsibilities. Wake fires before a turn begins; barge-in fires during SPEAKING. They do not share infrastructure.
- Wake-during-SPEAKING (barge-by-wake) is explicitly out of scope for D.4. If wake fires while the assistant is speaking, the detection is ignored until the assistant returns to IDLE. Document this clearly in `session_service.py`.
- The thread owns audio chunk feeding; `voice_service.stream_audio_chunks()` is the only entry point for chunk streaming.

### Acceptance

- Desktop live on both host classes: speaking "hey jarvis" triggers LISTENING without PTT on both host classes where openWakeWord models are present and mic is available.
- Wake-unavailable closeout condition: if openWakeWord models are missing or the host mic is unavailable, `GET /status/wake` returns `available: false`; shell shows PTT-only indicator; no crash. PTT-only is an accepted D.4 closeout state only when live wake detection is genuinely not achievable on that host. If models are present and mic is available, live wake trigger is required.
- Regression: 74+ tests green.

### Finish Line

JARVISv7 listens for the wake phrase continuously in the resident loop. PTT remains available as a manual override. Wake-unavailable degrades cleanly.

### Risks

- **Microphone contention**: wake monitoring holds an audio stream open continuously. When a turn starts (LISTENING), the wake stream must yield the microphone without device conflict. Verify that `sounddevice` supports concurrent stream management or that a single stream is shared. Document the chosen strategy before implementation.
- **openWakeWord latency on ARM64**: `onnxruntime` on ARM64 may have higher inference latency per chunk. Verify that a 0.1–0.2 s chunk interval keeps detection responsive. If not, increase chunk size and document the sensitivity trade-off.

## Sub-Slice D.5 — Personality / Presence Polish

### Why Here

The application is fully functional after D.4. D.5 makes it feel intentionally designed: the assistant's personality is visibly selectable, responses reflect the chosen profile, and presence behaviors give the assistant a consistent character across turns.

### Goal

A user can select a personality profile in the shell; the next turn's response style reflects that profile. Acknowledgment phrases and response pacing vary per profile.

### Scope

**Backend — personality extension:**
- `backend/app/personality/adapter.py` — promote from stub to real implementation:
  - `apply_personality(prompt: str, profile: PersonalityProfile) -> str` — appends tone/brevity/formality guidance to the prompt based on the active profile's fields. Keeps guidance concise; does not add tool or memory instructions.
  - Uses `system_prompt_addendum` from the profile when non-empty.
- `config/personality/` — add at minimum one additional personality profile YAML alongside `default.yaml`:
  - `concise.yaml` — shorter, more direct response style.
  - `warm.yaml` — warmer, more conversational tone.

**Backend API extension:**
- `GET /personality/list` → list of available profile ids and display names from `config/personality/*.yaml`.
- `POST /personality/select` → `{profile_id: str}` — switches the active personality for subsequent turns; returns the new active profile summary.

**Desktop shell updates — `desktop/src/components/`:**
- `personality-selection/` — new component: dropdown or button group showing available profiles; calls `POST /personality/select` on change; shows current active profile.
- Presence behaviors in the conversation display: assistant acknowledgment phrases ("Understood.", "On it.", "Let me think.") appear as UI-side text injected at LISTENING/TRANSCRIBING/REASONING state transitions before the actual response arrives. These are UI-layer latency bridges, not LLM output.
- Response pacing: if the TTS sample rate is exposed in `VoiceTurnResponse`, the shell may use it as a visual cue; otherwise, presence is a fixed UI pattern.

**`backend/tests/unit/personality/test_adapter.py`** — extend or add:
- `test_adapter_appends_tone_guidance_to_prompt`
- `test_adapter_respects_brevity_field`
- `test_adapter_uses_system_prompt_addendum_when_set`
- `test_adapter_returns_prompt_unchanged_when_profile_has_no_addendum`

**`backend/tests/unit/api/test_routes.py`** — extend:
- `test_personality_list_returns_available_profiles`
- `test_personality_select_switches_active_profile` (mock)

### Files Not to Modify

- `backend/app/cognition/prompt_assembler.py` — personality is already injected as a parameter; adapter enriches the profile, does not change the assembler interface
- `backend/app/conversation/engine.py` — no changes; personality flows through existing injection
- `TurnArtifact` schema — `active_personality_profile_id` field already present

### Key Design Decisions

- Personality switch applies to the next turn, not the current one. The active profile is stored in `SessionService` state and used by `TurnEngine` on each `run_*_turn` call.
- `personality/adapter.py` returns the prompt enriched with tone/brevity/formality guidance, using the existing personality injection seam (`assemble_prompt` already accepts `PersonalityProfile`). It does not modify `prompt_assembler.py`. If the existing seam is insufficient to apply profile guidance without changing the assembler interface, stop and surface a narrow correction proposal before proceeding.
- Presence behaviors (acknowledgment phrases) are entirely shell-side UI text. They are not LLM output, not persisted to artifacts, and not part of the turn lifecycle. The shell displays them during state transitions as user-facing latency feedback.
- New personality YAML files are additive config. Adding a new profile requires only a new YAML file and no code change.

### Acceptance

- Backend unit: `test_adapter_*` suite green on both host classes.
- Desktop live on both host classes: selecting "concise" profile → next turn response is observably shorter than "default"; selecting "warm" profile → response is observably more conversational.
- Presence behaviors: acknowledgment text appears in the conversation display during LISTENING/TRANSCRIBING/REASONING states before the LLM response arrives.
- Regression: 74+ tests green.

### Finish Line

JARVISv7 presents a consistent, selectable personality across turns. Presence behaviors give the assistant character during inference latency. No cognition changes, no tools, no agents.

### Risks

- LLM response style variation per personality is non-deterministic. Acceptance requires an observable subjective difference in a live turn, not a unit-provable assertion. Document this explicitly in the D.5 CHANGE_LOG evidence: a live turn demonstrating profile-switched behavior is the evidence, not a unit assertion on LLM output.
