# 20260503-slice_f.md
## Slice F — Tool Foundation + ACTING Path

**Date:** 2026-05-03
**Status:** Proposed
**Depends on:** Slice E closed and verified (SYSTEM_INVENTORY.md)

---

## Purpose

Slice F converts ACTING from a stubbed state into a working turn-lifecycle stage and gives the turn engine its first set of real tools. After F.3, a spoken or typed request that requires a tool — looking up the time, reading a file, searching the web — executes deterministically, records the invocation and result in the turn artifact, and renders attribution in both the desktop shell and text surface.

The tool foundation is built here, before Group G memory retrieval and Group I agents, because:
- ACTING must be proven on the live path before any higher-level system builds on it.
- The tool registry must exist before G.2 retrieval injection or I.3 Planner/Executor can reference it.
- Internet search is already available via the E.5 runtime; connecting it to a tool adapter here is the lowest-cost, highest-value first tool.

## Out of Scope

- Group G: episodic memory write/retrieval (G.1–G.3) — not introduced here
- Group H: local llama.cpp (H.1), QNN inference (H.2), DirectML uplift (H.3)
- Group I: agent roles (Planner, Executor, Critic), agent ledger, agent messages — the tool registry is consumed by agents later; the registry itself is not an agent
- Write tools of any kind — filesystem writes, file creation, data modification — deferred to a future approved slice
- Tool-triggered model downloads or provisioning changes
- New runtime families or search providers beyond what Slice E established
- LLM prompt-routing logic that uses model-side tool-calling APIs (function calling, tool-use JSON schemas) — tool selection is done deterministically in `executor.py`, not by the LLM
- WebSockets, streaming, audio-path changes
- Production sandboxing enforcement — the filesystem sandbox path is configured and honored; OS-level sandboxing is not in scope

## Sub-Slice F.1 — Executor + ACTING State

### Why Here

`ConversationState.ACTING` is stubbed with `NotImplementedError`. The turn engine has no mechanism to execute a tool between REASONING and RESPONDING. F.1 replaces the stub with a real, deterministic executor that coordinates tool invocations within the turn lifecycle.

The executor is not an agent. It does not plan, evaluate, or learn. It runs one or more tool calls as directed, records the results, and returns control to the turn engine.

### Goal

A text or voice turn that requires tool use transitions through `ACTING`, executes a stub tool, returns a tool-grounded response, and records the invocation in the turn artifact. All of this happens without any change to non-tool turns.

### Scope

**`backend/app/cognition/executor.py`** — `ToolExecutor`:
- `execute(tool_name: str, tool_input: dict, registry: ToolRegistry) -> ToolResult`
- `ToolResult` dataclass: `tool_name: str`, `tool_input: dict`, `tool_output: str`, `error: str | None`, `success: bool`
- Calls `registry.invoke(tool_name, tool_input)` — fail-closed: if the tool raises or is missing, returns a `ToolResult` with `success=False` and `error=<reason>`. Never raises to the caller.
- One `ToolResult` per call. Sequential multi-tool execution is via multiple calls; no concurrency in F.1.

**`backend/app/conversation/engine.py`** — extend `_run_reasoning_path()`:
- `ACTING` exists as a valid state in the enum and transition guard but is not yet wired into the live turn lifecycle. The existing `enter_stub_state()` raises `NotImplementedError` for `ACTING`; F.1 replaces this by extending `_run_reasoning_path()`, not by modifying `enter_stub_state()`.
- Tool use is signalled by an explicit `tool_name: str | None` parameter added to `run_text_turn()` and `run_voice_turn()`. When `tool_name` is not `None`, the engine advances to `ACTING`, calls `executor.execute()`, injects the `ToolResult` into a follow-up prompt, re-enters REASONING for the final LLM response, then advances to RESPONDING. Record invocation in `TurnArtifact.tools_invoked` and `TurnArtifact.agent_trace`.
- LLM-driven tool selection is Group I scope. Do not implement keyword heuristics or model-side function-calling bindings in Slice F.
- If no tool use (`tool_name=None`): existing `REASONING → RESPONDING` path is unchanged.
- `TurnResult` extended with optional `tool_calls: list[dict]` and `tool_results: list[dict]` fields (both default to empty list).
- Non-tool turns must produce identical results to pre-F.1 baseline.

**Stop condition:** If populating `TurnArtifact.tools_invoked` or `TurnArtifact.agent_trace` requires changes to existing field types or names, stop and surface a narrow correction. The artifact schema is a fixed compatibility boundary from C.3.

**`backend/tests/unit/cognition/test_executor.py`**:
- `test_executor_calls_registry_invoke`
- `test_executor_returns_fail_closed_result_when_tool_missing`
- `test_executor_returns_fail_closed_result_when_tool_raises`
- `test_executor_result_has_success_false_on_error`

**`backend/tests/unit/conversation/test_engine.py`** — extend:
- `test_acting_state_entered_when_tool_requested`
- `test_acting_state_not_entered_when_no_tool_request`
- `test_tool_invocation_recorded_in_turn_artifact`
- `test_non_tool_turn_behavior_unchanged`

### Files Not to Modify

- `TurnArtifact` schema fields — extend `tools_invoked` population only; no renames or removals
- `ConversationState` enum and `VALID_TRANSITIONS` — read-verify only; no changes
- `backend/app/runtimes/**`, `backend/app/hardware/**`, `backend/app/services/**`
- `backend/app/tools/**` — not yet created; F.2 owns this

## Sub-Slice F.2 — Tool Registry + First Tool Set

### Why Here

ACTING is proven (F.1). F.2 registers the first real tools and proves each produces correct, traceable results in a live turn. The tool registry is the single discovery and dispatch surface; future tools — including Group I agent tool invocations — add themselves here without redesign.

### Goal

Time/date, read-only filesystem, and internet search each return correct results in a real turn. Every invocation is logged and traceable.

### Scope

**`backend/app/tools/registry.py`** — `ToolRegistry`:
- `register(tool: ToolBase) -> None` — registers a tool by name; raises if name is already registered.
- `invoke(tool_name: str, tool_input: dict) -> str` — dispatches to the named tool's `run()` method; raises `ToolNotFoundError` if the tool is not registered.
- `list_tools() -> list[str]` — returns registered tool names.
- Registry is instantiated once at application startup (via `api/dependencies.py`) and injected into `TurnEngine` / `ToolExecutor`.

**`backend/app/tools/registry.py`** — `ToolBase` ABC:
- `name(self) -> str` — unique tool identifier (e.g., `"time"`, `"filesystem.read"`, `"search"`).
- `description(self) -> str` — one-line human-readable description.
- `run(self, tool_input: dict) -> str` — executes the tool; returns a plain string result. Fail-closed: never raises; returns an error string on failure.

**`backend/app/tools/system/time_tool.py`** — `TimeTool`:
- `name()` → `"time"`
- `run({})` → ISO-8601 UTC datetime string.
- No external dependencies.

**`backend/app/tools/system/hardware_tool.py`** — `HardwareTool`:
- `name()` → `"hardware.info"`
- `run({})` → JSON string of selected fields from the application's `HardwareProfile` (arch, GPU vendor, NPU vendor, memory). Read-only.
- Reads from the cached profile in `api/dependencies.py`; does not call `run_profiler()` at tool-invocation time.

**`backend/app/tools/filesystem/read_tool.py`** — `FilesystemReadTool`:
- `name()` → `"filesystem.read"`
- `run({"path": "<relative_path>"})` → file contents as a string, or an error string if the path is outside the sandbox or the file does not exist.
- Sandbox path: `TOOL_FILESYSTEM_SANDBOX_PATH` from settings (default `data/`). Any path that does not resolve to a descendant of the sandbox root is refused with a clear error string — no exception propagated.
- Reads text files only (UTF-8). Binary files return an error string.

**`backend/app/tools/search/search_tool.py`** — `SearchTool`:
- `name()` → `"search"`
- `run({"query": "<query>", "max_results": 3})` → JSON string of `list[SearchResult]`, or `"[]"` if no provider is available.
- Calls `runtime, _trace = select_search_runtime(settings)` then `runtime.search(query, max_results=max_results)`. The `SelectionTrace` is available for logging but need not be surfaced in the tool result string.
- Does not call search providers directly; does not own provider selection.
- If the search runtime returns an empty list (all providers down or rate-limited), returns `"[]"` and logs the condition — does not raise.

## Sub-Slice F.3 — Tool Result Rendering in Desktop + Text Shells

### Why Here

Tools produce results. The shell must present them usefully without knowing tool internals. F.3 extends the API response schema with optional tool-result metadata and updates the desktop conversation component and text surface to render it.

The shell is a thin renderer. It displays what the backend sends. It does not execute tools, select tools, or parse tool output.

### Goal

A desktop or text-shell turn that invoked a tool shows: which tool was called, what the tool returned (summary), and the final LLM response grounded in the tool result. Attribution is visible and distinct from the response text.

### Scope

**`backend/app/api/schemas/task.py`** — extend `TextTurnResponse`:
- Add optional `tool_calls: list[ToolCallSummary] | None = None`
- `ToolCallSummary` dataclass (in `schemas/tools.py`): `tool_name: str`, `tool_input: dict`, `tool_output_summary: str`, `success: bool`

**`backend/app/api/schemas/voice.py`** — extend `VoiceTurnResponse` with the same optional `tool_calls` field.

**`backend/app/api/schemas/tools.py`** — new; defines `ToolCallSummary`.

**`backend/app/api/routes/task.py`** and `voice.py` — populate `tool_calls` from `TurnResult.tool_calls` when present; existing response shape unchanged when no tools were called.

**`desktop/src/components/conversation/`** — extend the conversation display to render tool attribution:
- If the response includes `tool_calls`, show a collapsible "Tool used: `<tool_name>`" line above the response text.
- Tool output summary displayed in a distinct style (e.g., monospace, muted color) to differentiate from the LLM response.
- No tool-specific rendering logic — the component renders the `tool_output_summary` string regardless of which tool produced it.

**Text surface** — `scripts/run_jarvis.py` `--verbose` mode already streams state transitions; extend to print tool attribution lines when `TurnResult.tool_calls` is non-empty. No structural change to the proving host.

**`backend/tests/unit/api/test_routes.py`** — extend:

## Slice-Level Closeout Criteria

Slice F is complete when all of the following are true:

1. F-G1 through F-G5 gates verified before the relevant sub-slice begins.
2. `ACTING` state is reached in a live turn; the state transition is recorded in `TurnArtifact`.
3. `ToolExecutor` is fail-closed: a missing or erroring tool never raises to the turn engine.
4. `ToolRegistry` accepts registration and dispatch; duplicate registration raises at startup, not at invocation time.
5. All four first tools are registered, callable, and unit-tested.
6. Time tool returns ISO-8601 string; filesystem tool refuses paths outside sandbox with a clear error string; search tool returns `[]` (not an exception) when providers are unavailable; hardware tool reads from cached profile.
7. At least one tool invocation is proven in a live turn on both host classes (`test_acting_live.py`).
8. `tools_invoked` in `TurnArtifact` is populated for tool turns; empty for non-tool turns.
9. Desktop conversation display shows tool attribution for tool turns.
10. Non-tool turns produce identical results to pre-F.1 baseline.
11. All unit suites green on both host classes.
12. `validate_backend.py regression` ≥95 on both host classes.
13. `CHANGE_LOG.md` has one entry per sub-slice F.1–F.3 with host class(es).
14. No Group G/H/I file introduced at Slice F closeout.
15. Slice A–E regression count has not decreased.
