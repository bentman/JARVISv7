## Slice K — UI Controls + Operator Settings

**Date:** 2026-05-26
**Status:** Proposed
**Depends on:** Slice J closed and verified (`SYSTEM_INVENTORY.md`)

---

## Purpose

Group J produced a shell that surfaces readiness data accurately, labels interaction states clearly, and communicates the wake/PTT activation paths. What the operator still cannot do from the UI is adjust anything. Every configurable setting requires editing files directly and restarting the backend manually.

Group K adds the interactive control surface. It is scoped tightly: settings visibility, read/write of an explicit allowlist of `.env` fields, restart-required UX, service health display, and lightweight appearance adjustments using the token set J.4 already established. Nothing in K re-engineers backend behavior, changes personality runtime logic, or introduces model management.

## Sub-Slice K.2a — Settings Inventory + Backend Read/Write Route

### Why Here

Before any settings UI is built, the backend must define exactly which fields are operator-configurable and provide a safe read/write surface. The server-side allowlist is the authoritative guard — the UI only reflects what the backend declares.

### Goal

`GET /config/operator` returns the current values of allowlisted `.env` fields. `POST /config/operator` writes only allowlisted fields to `.env` using a key-preserving strategy. Non-allowlisted keys are rejected with a named error.

### Scope

**Allowlist (from Gate K-1 enumeration of `.env.example` — these are the likely fields; Gate K-1 finalizes):**

| Field | Type | Label |
|-------|------|-------|
| `USE_OLLAMA` | bool | Use Ollama |
| `OLLAMA_BASE_URL` | text | Ollama endpoint |
| `OLLAMA_MODEL` | text | Ollama model |
| `OLLAMA_NUM_CTX` | int | Context window |
| `USE_LOCAL_MODEL` | bool | Use local model |
| `USE_SEARXNG` | bool | Use SearXNG |
| `SEARXNG_BASE_URL` | text | SearXNG URL |
| `USE_DDGS` | bool | Use DDGS |
| `USE_TAVILY` | bool | Use Tavily |
| `TAVILY_API_KEY` | password | Tavily API key |
| `REDIS_HOST` | text | Redis host |
| `REDIS_PORT` | int | Redis port |

All of these are present in `.env.example` under their existing names. Gate K-1 produces the final list.

**`backend/app/api/routes/config.py`** — new:

- `GET /config/operator` → `{ fields: [{ key, value, type, label, restart_required }] }`. Reads current environment values. Password-type fields return `"***"` if set, `""` if unset.
- `POST /config/operator` → accepts `{ fields: { key: value } }`. Key-preserving `.env` write (read existing lines, update matching keys, preserve unrelated lines, append new allowlisted keys if absent). Returns `{ written: [key, ...], rejected: [{ key, reason }] }`. Never writes a key outside the allowlist. Never creates or deletes `.env`; operates only on an existing file.

**`backend/app/api/schemas/config.py`** — new: Pydantic schemas for the above.

**`backend/tests/unit/api/test_routes.py`** — extend:
- `GET /config/operator` returns expected shape with all allowlisted keys
- `POST /config/operator` with a valid key returns `written: [key]`
- `POST /config/operator` with a non-allowlisted key returns `rejected: [{ key, reason }]`
- `POST /config/operator` does not modify lines outside the allowlist in the `.env` file

### Files Likely Touched

- `backend/app/api/routes/config.py` (new)
- `backend/app/api/schemas/config.py` (new)
- `backend/tests/unit/api/test_routes.py`

## Sub-Slice K.2b — Desktop Settings Panel

### Why Here

K.2a proved the backend surface. K.2b builds the settings panel component that calls it.

### Goal

- Dirty-state tracking: unsaved-change indicator when any field differs from loaded value.

- Cancel button: discards changes, prompts if dirty.
- Panel is rendered into a container element passed to `openSettings(containerEl)`. Closed via `closeSettings()`.
- Export: `openSettings(containerEl)`, `closeSettings()`.
- UI copy is sparse: field label, current value, input, "restart required." No explanatory paragraphs.

## Sub-Slice K.2c — Restart-Required UX

### Why Here

Saving settings without restarting leaves the backend running with the old values. The restart action must be explicit, unambiguous, and close the loop — the operator should end up back at a running, readiness-loaded state.

### Goal

After saving settings, the panel shows a persistent "restart required" state. A restart button triggers `stop_backend` → `start_backend` via existing Tauri commands in sequence. After restart, readiness is reloaded and the panel's loaded values refresh.

### Scope

**`desktop/src/components/settings-panel.js`** — extend:

- After a successful save, transition the panel to a `restart-required` state: hide the save/cancel buttons, show a restart button and a "Settings saved — restart required to apply" message (single line, not a paragraph).
- Restart button handler: call `stop_backend` → `start_backend` → reload readiness via `get_readiness` → refresh panel values via `GET /config/operator` → return panel to normal state.
- If restart fails, show the error inline (using `--color-failed`) and keep the restart button visible.
- The restart-required state persists until either a restart completes or the panel is closed (in which case, the header or sidebar should carry a persistent "restart required" indicator — a small badge or text adjacent to the settings trigger).

**`desktop/src/index.html`** — add a restart-required indicator element near the settings trigger (hidden by default; shown when `settings-panel.js` sets a restart-pending flag).

**`desktop/src/main.js`** — no structural changes; `settings-panel.js` owns the restart sequence.

## Sub-Slice K.4 — Appearance Adjustments

### Why Here

J.4 established a token-driven style system. Appearance adjustments are safe because all visual values are CSS custom properties — overriding a small set of tokens on `:root` changes density and accent without structural changes.

### Goal

The operator adjusts font size (two steps), panel density (two steps), and accent color (two presets) from a compact control in the sidebar. Changes apply immediately. Preferences persist between sessions. Semantic state tokens are never overridden.

### Scope

**Token overrides — only these tokens are modified at runtime:**

- Font size step: overrides `--text-sm`, `--text-md`, `--text-lg` (larger step increases each by ~15%)
- Density step: overrides `--space-2`, `--space-3`, `--space-4` (compact reduces each by ~25%)
- Accent preset A (default): `--color-accent: #00bfff` (current value — no change)
- Accent preset B (neutral): `--color-accent: #8b949e`

Semantic tokens (`--color-ready`, `--color-degraded`, `--color-failed`, `--color-capture`, `--color-speaking`, `--color-thinking`, `--color-transcribing`) are never overridden.

**`desktop/src/components/appearance-controls.js`** — new module:

- Renders a compact row in the sidebar: two font-size buttons (A / A+), two density buttons (compact / default), two accent buttons.
- On change: applies CSS custom property overrides via `document.documentElement.style.setProperty()`.
- Persists to `localStorage` as `jarvisv7_appearance: { fontSize, density, accent }`. Fallback to Tauri file write if Gate K-4 confirms `localStorage` unavailable.
- Export: `initAppearanceControls(containerEl)`, `applyStored()`.

**`desktop/src/index.html`** — appearance controls container at bottom of sidebar.

**`desktop/src/main.js`**:
