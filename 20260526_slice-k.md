# 20260526_slice-k.md
## Slice K — UI Controls + Operator Settings

**Date:** 2026-05-26
**Status:** Proposed
**Depends on:** Slice J closed and verified (`SYSTEM_INVENTORY.md`)

---

## Purpose

Group J established an information-dense, diagnostics-accurate shell. The readiness data is correct and visible. Named interaction states are labeled. Wake and PTT paths are clearly communicated. The visual language is token-driven.

What the shell still lacks is operator agency. A user cannot adjust the application's configurable behavior from the UI — not device selection, not personality management, not service toggles, not appearance preferences. Every configuration change currently requires editing files and restarting the backend manually. There is no way to understand from the UI which settings can be changed, what they affect, or when a restart is needed.

Group K closes this gap. It adds the interactive controls that a persistent local assistant needs day-to-day: the ability to switch personalities with context, view and modify operator-configurable settings, understand service state, and adjust the application's density and appearance. All controls are built on the stable readiness data J established. No control silently mutates runtime state — every change that requires a restart says so, and the UI makes the restart action explicit.

Group K is also the prerequisite for any advanced operator workflows in later groups. Controls that do not exist cannot be misused; controls that exist but lack guardrails will be. This slice builds both.

## Sub-Slice K.2 — Operator Settings Panel

### Why Here

Every configuration change currently requires editing `.env` directly and restarting manually. For a persistent local assistant, this is unacceptable friction. K.2 surfaces the operator-configurable fields from Gate K-1 as a UI-accessible settings panel with explicit restart-required semantics.

### Goal

A settings panel accessible from the sidebar exposes the Gate K-1 enumerated `.env` fields. Saving writes only to `.env` via a server-side allowlist route. All changes show "restart required." A restart button triggers `stop_backend` → `start_backend` via existing Tauri commands.

### Scope

**`desktop/src/components/settings-panel.js`** — new module `SettingsPanel`:

- Renders the operator-configurable fields from Gate K-1 enumeration.
- Each field shows: current value, an editable input appropriate to the value type (text, toggle, select), and a "restart required" label.
- Dirty-state tracking: visual indicator when unsaved changes exist.
- On save: writes modified fields via `POST /config/operator`. Displays confirmation.
- On restart: calls `stop_backend` then `start_backend` via existing Tauri commands in sequence. Reloads readiness after restart.
- On cancel: discards changes, confirms if unsaved changes exist.
- Export: `openSettings(containerEl)`, `closeSettings()`.

**Likely operator-configurable fields (final list from Gate K-1; these are representative):**

- `OLLAMA_BASE_URL` — Ollama endpoint; text input
- `OLLAMA_MODEL` — active model name; text input
- `LOG_LEVEL` — select (DEBUG / INFO / WARNING / ERROR)
- `REDIS_URL` — Redis connection string; text input
- `SEARXNG_URL` — SearXNG instance URL; text input
- `TAVILY_API_KEY` — password input (masked)
- `ENABLE_CLOUD_ESCALATION` — boolean toggle
- `ENABLE_REDIS` — boolean toggle (bootstrap; restart-required)
- `ENABLE_SEARXNG` — boolean toggle (bootstrap; restart-required)
- Any additional fields confirmed in Gate K-1

**`backend/app/api/routes/config.py`** — new:

- `GET /config/operator` — returns current values of operator-configurable fields from the running environment. Response: `{ fields: [ { key, value, restart_required, description } ] }`. Does not expose non-operator-configurable fields.
- `POST /config/operator` — accepts `{ fields: { key: value } }`. Writes only explicitly operator-configurable fields to `.env` using a key-preserving write strategy (read existing, update keys, write back — never truncate). Rejects any key not in the allowlist with an explicit error. Returns `{ written: [key, ...], rejected: [{ key, reason }] }`.

**`backend/app/api/schemas/config.py`** — new: request/response schemas for the above routes.

**`desktop/src/index.html`** — settings trigger:

- Add a settings trigger element in the sidebar (button or icon). Opens the settings panel.

**`desktop/src/main.js`** — import and wire `SettingsPanel`.

**`fetch()` assessment:** If Tauri 2.x WebView2 supports `fetch()` to the local backend (127.0.0.1:8765), the settings panel calls the REST routes directly — no new Tauri commands. If `fetch()` is blocked by CSP or allowlist, add `get_operator_config` and `set_operator_config` Tauri commands that proxy the REST routes. Assess at Gate K-3 before implementation.

**`backend/tests/unit/desktop/test_desktop_static_contract.py`** — extend:
- Assert `settings-panel.js` exports `openSettings` and `closeSettings`
- Assert a settings trigger element is present in `index.html`

**`backend/tests/unit/api/test_routes.py`** — extend:
- Assert `GET /config/operator` returns expected field list shape
- Assert `POST /config/operator` rejects a non-operator-configurable key
- Assert `POST /config/operator` accepts and writes a valid key

## Sub-Slice K.4 — Appearance Controls

### Why Here

The token-driven visual system from J.4 makes appearance customization safe: overriding a small set of CSS custom properties on `:root` changes the visual density and accent without touching structural CSS or semantic state colors. K.4 exposes three adjustable dimensions — font size, panel density, and accent color preset — with preferences persisted between sessions.

### Goal

The operator adjusts font size, panel density, and accent color preset from a control surface in the sidebar. Preferences apply immediately without a backend restart and are restored on next app open. Semantic state colors are never affected by accent changes.

### Scope

**`desktop/src/components/appearance-controls.js`** — new module `AppearanceControls`:

- Renders a compact appearance control section in the sidebar.
- **Font size:** three steps — compact, default, comfortable. Applies by setting `--font-size-base` and `--font-size-sm` overrides on `:root`.
- **Panel density:** two steps — compact, default. Applies `--space-sm` and `--space-md` overrides on `:root`.
- **Accent color:** three presets — default cyan, neutral white, warm amber. Each preset overrides only `--color-accent`, `--color-accent-dim`, and `--color-accent-glow`. Semantic state tokens (`--color-degraded`, `--color-failed`, `--color-speaking`, etc.) are never overridden by presets.
- Preferences persisted to `localStorage` keyed by `jarvisv7_appearance` (fallback to Tauri file write if Gate K-4 confirms `localStorage` unavailable).
- Export: `initAppearanceControls(containerEl)`, `applyStored()`.

**`desktop/src/index.html`** — appearance control container at bottom of sidebar.

**`desktop/src/main.js`** — extend:

- Call `AppearanceControls.applyStored()` before `startDesktop()` so preferences apply before readiness loads.
- Call `AppearanceControls.initAppearanceControls(appearanceContainerEl)` after startup completes.

**No backend changes.** Appearance preferences are client-side only.

**`backend/tests/unit/desktop/test_desktop_static_contract.py`** — extend:
- Assert appearance control container present in `index.html`
- Assert `appearance-controls.js` exports `initAppearanceControls` and `applyStored`
