The next logical inventory item should **not** be Agent Framework yet.

The next slice should be:

**Conversation Continuity and Session Memory Boundary**

This is the missing bridge between the newly verified realtime session boundary and the existing episodic memory foundation.

## Why this is next

`ProjectVision.md` defines JARVIS as a real-time conversation engine where the user can speak naturally, interrupt, continue, and rely on preserved turn continuity. The root loop explicitly includes interruption and natural continuation, not just one-shot voice turns.  

The vision also defines a session as a bounded conversational context that may contain one or more turns and exists specifically for continuity, memory policy, and interruption recovery.  The current inventory now has a verified realtime event boundary, but its entry says committed turn execution still belongs to `TurnEngine`; it does not claim durable continuity policy, event persistence, follow-up context assembly, or interruption recovery context. 

The memory foundation exists, but it is still episodic retrieval plus recency/keyword lookup. Inventory explicitly says semantic/vector memory is not implemented, durable memory authority is disk-backed episodic entries, and retrieval is not tool execution. 

So the gap is now between:

`Realtime session events`
and
`Memory-backed conversational continuity`

That bridge should come before agents, because agents need stable session context and trace state. It should also come before streaming response work, because streaming without coherent session continuity only improves latency, not natural conversation.

## What the next inventory capability should eventually be

When implemented and validated, the future inventory item should read roughly:

**Capability: Conversation continuity and session memory boundary**

State: Verified

Meaning: JARVIS can maintain explicit in-session conversational context across multiple turns, preserve interruption/recovery context, assemble a bounded continuity packet for the next committed turn, and persist enough session timeline data to reconstruct what happened without hiding memory inside the model.

This is not semantic memory. It is not agent orchestration. It is the durable continuity layer that makes natural follow-up possible.

## Why not agents next

`slices.md` currently places Agent Framework after Group M, but that sequencing is now suspect given the product direction. Agents are supposed to compose the conversation engine, tool registry, memory, and LLM runtime; they do not replace the turn engine. 

If agents are added before continuity/session memory is clarified, they will be tempted to invent their own context, history, and recovery model. That would violate the explicit cognition framework: no hidden state inside the model, memory externalized, plans/outcomes inspectable, and important transitions reconstructable from artifacts. 

The agent layer should consume a stable continuity substrate. It should not define one.

## Why not semantic/vector memory next

Semantic memory is important, but it is not the smallest next boundary. The current problem is not “better retrieval quality” yet. It is that the system needs an explicit, bounded, current-session continuity model that knows:

what just happened,
what the user interrupted,
what the assistant was saying,
what the next turn is likely referring to,
what can be safely carried forward,
what must be persisted as an artifact,
what may later become episodic or semantic memory.

Semantic/vector memory should later consume curated session material. It should not be the first mechanism to create conversational continuity.

## Recommended next slice

Title:

`Group N — Conversation Continuity and Session Memory Boundary`

This would likely require renumbering the current Agent Framework group to a later group. That is appropriate. The cleanup already removed brittle “specific future slice” language; the architecture should now follow boundary readiness, not old labels.

### Purpose

Create the durable continuity layer that connects realtime session events, committed turn artifacts, bounded working memory, episodic memory, interruption recovery, and prompt assembly.

Turn artifacts with session id, transcript, prompt, memory refs, tools, response, interruption events, final state, and failure reason required by vision.

## Proposed slice structure

### N.0 — Continuity Boundary Census

Confirm exact current ownership of:

`SessionManager`
`SessionService`
`RealtimeConversationSession`
`RealtimeEventLedger`
`TurnArtifact`
`SessionArtifact`
`episodic.py`
`retrieval.py`
`write_policy.py`
`prompt_assembler.py`

Output should state which component owns:

live event ordering,
committed turn result,
working memory,
episodic write/read,
retrieved context,
session closeout,
future semantic write eligibility.

No code changes needed here unless the repo requires a planning artifact.

### N.1 — Session Timeline Artifact

Add a durable session-level timeline or extend the existing session artifact boundary so realtime events can be persisted by reference without bloating turn artifacts.

This should not replace `TurnArtifact`. It should record session-level event facts:

session started,
invocation source,
user turn committed,
turn id linked,
assistant response started,
speech started only if real,
interruption detected,
recovery/idle/failure.

This closes the prior secondary gap: Group M has an in-memory ledger, but not yet durable/user-visible continuity evidence.

### N.2 — Continuity Packet

Add a bounded `ContinuityPacket` built from the active session timeline, recent turn artifacts, working memory, and relevant episodic retrieval.

It should be explicit and small:

last user request,
last assistant response summary,
open topic,
pending correction/interruption context,
active tool/result context if any,
recent retrieved memory refs,
safe carry-forward facts,
excluded/expired context.

This packet becomes an input to prompt assembly as trusted application/session context, separate from untrusted retrieval/tool/user content.

### N.3 — Follow-up and Interruption Recovery Policy

Add deterministic policy for whether the next turn is considered a continuation.

Inputs:

time since last turn,
same active session,
interruption occurred,
failure occurred,
user source wake/PTT/text,
last final state,
explicit reset/stop phrase later,
active response was interrupted.

Outputs:

continue current session,
start new session,
recover interrupted response,
ignore stale context,
summarize and close.

This must not be LLM-decided. The LLM can consume the result, but not own the policy.

### N.4 — Prompt Assembly Integration

Wire the continuity packet into the existing personality prompt envelope without crossing authority boundaries.

Order should be approximately:

application rules,
personality style,
trusted session continuity packet,
working memory,
retrieved episodic context,
tool context,
user request,
output contract.

This keeps continuity explicit and prevents it from becoming hidden model state.

### N.5 — Session Closeout and Memory Writeback Hook

At session closeout, decide what is eligible for episodic writeback or future semantic curation.

Initial scope should be conservative:

record session summary metadata,
link turn ids,
persist interruption/recovery events,
do not add semantic/vector embeddings,
do not make autonomous memory decisions.

This gives semantic memory a future feed without implementing semantic memory yet.

### N.6 — Validation and Inventory Closeout

Only after x64 and ARM64 validation should inventory gain the capability.

Acceptance should prove:

A second turn in the same session receives bounded continuity context.

A new session does not accidentally inherit stale context.

An interrupted response produces recovery context.

The continuity packet is visible in artifacts or diagnostic evidence.

Episodic retrieval still works unchanged.

Prompt envelope authority ordering remains correct.

No agents, semantic vectors, streaming transport, model routing, or desktop redesign are introduced.

## Why this helps real-time conversation

From the prior production-bot research, real voice agents need more than STT → LLM → TTS. They need live session mechanics, turn-taking, interruption, state, fulfillment, handoff, and traceability.

Group M added the live session event spine.

The next needed capability is the **stateful conversational memory spine**.

Only after that does it make sense to add:

streaming STT,
semantic endpointing,
streaming assistant response,
tool progress phrases,
workflow/page state,
agents,
semantic memory.

Otherwise those systems will each invent their own continuity model.

## Decision

Insert a new group before Agent Framework:

**Group N — Conversation Continuity and Session Memory Boundary**

Move Agent Framework later.

This is the next inventory-worthy capability because it turns the verified realtime session boundary and existing episodic memory substrate into an explicit continuity layer. It directly advances natural conversation, prepares durable memory extension, and protects the architecture from bolt-on agents or bolt-on semantic memory.
