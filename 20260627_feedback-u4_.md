# Slice U4 Feedback: QNN EPContext Session Compatibility

## Summary

Slice U4 should not wire `whisper-qualcomm-qnn` yet.

The U4 handoff package passed artifact-shape inspection, but the required QNN-only session-load proof failed before any model staging, catalog wiring, runtime selection, or governance logging was performed.

The failure is not caused by the proposed side-by-side folder layout. The same hard session-load failure also occurs with the existing catalog QNN Whisper artifact under the current split QNN runtime stack.

## Host Evidence

- Host class: Windows ARM64 / arm64
- Python environment: `backend\.venv`
- Provisioned QNN packages:
  - `onnxruntime==1.27.0`
  - `onnxruntime-qnn==2.3.0`
- `scripts\validate_backend.py profile` reported:
  - `qnn:provider_library_registered`
  - `ep:QNNExecutionProvider`
  - `dll:QnnHtp`

## Failure

Command shape:

```powershell
backend\.venv\Scripts\python -c "... create_qnn_session(path, disable_cpu_fallback=True) ..."
```

Encoder session-load failure:

```text
RuntimeError: QNN session creation failed for model='cache\slice-u4\artifact-inspect\encoder\extracted\job_jgooj3vkg_optimized_onnx\model.onnx'
...
[ONNXRuntimeError] : 1 : FAIL : This session contains graph nodes that are assigned to the default CPU EP, but fallback to CPU EP has been explicitly disabled by the user.
```

With CPU fallback enabled, session creation still failed:

```text
[ONNXRuntimeError] : 9 : NOT_IMPLEMENTED : Failed to find kernel for com.microsoft.EPContext(1) (node:'QNNContext' ep:'CPUExecutionProvider'). Kernel not found
```

Diagnostic checks that did not change the failure:

- Changing current working directory to the ONNX sidecar directory
- Using `QnnCpu.dll`, `QnnGpu.dll`, or `QnnHtp.dll`
- Adding common QNN provider/session options, including explicit context-disable and HTP target/performance options
- Adding an explicit `com.microsoft` opset import in a cache-only diagnostic copy

## Assessment

The current readiness gate proves QNN provider registration, not QNN Whisper EPContext model loadability.

Provider registration is healthy, but session proof is not. `QNNExecutionProvider` is exposed by ONNX Runtime, yet ONNX Runtime assigns the `EPContext` node to CPU. CPU fallback is correctly disabled for the proof, so the failure is valid and should block U4 wiring.

Do not accept fallback-enabled success for this slice. In this case fallback-enabled creation does not succeed anyway, and the error confirms CPU has no kernel for `EPContext`.

## Proposed Correction

Do not stage or wire `whisper-qualcomm-qnn` yet.

Add a narrow ARM64/QNN live compatibility gate before U4 model selection changes:

- Location: `backend\tests\runtime\hardware\test_qnn_gate_live.py`
- Scope:
  - Load the selected QNN Whisper encoder ONNX with `disable_cpu_fallback=True`
  - Load the selected QNN Whisper decoder ONNX with `disable_cpu_fallback=True`
  - Require `QNNExecutionProvider` to be primary for both sessions
  - Fail with an explicit artifact/session-load reason when `EPContext` is assigned to CPU

Keep provider-registration tests, but do not treat provider registration alone as proof that STT QNN artifacts are executable.

If runtime diagnostics need clearer failure wording, make the smallest change in:

```text
backend\app\hardware\qnn_provider.py
```

That change should only classify/report EPContext assignment failure. It should not relax CPU fallback, add alternate provider behavior, or change AMD64 behavior.

## Boundary

This correction should not:

- Pin, downgrade, force reinstall, or uninstall `onnxruntime` / `onnxruntime-qnn` inside U4
- Use `--no-deps`
- Create a new Python venv
- Create new Qualcomm Workbench jobs
- Change AMD64 selection, provisioning, or runtime behavior
- Stage `whisper-qualcomm-qnn` into `models\stt`
- Change `whisper-base-en-qnn-snapdragon-x-elite`
- Update `CHANGE_LOG.md` or `SYSTEM_INVENTORY.md` without explicit User consent
