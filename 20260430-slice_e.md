## Slice E — Local Service Substrate

**Date:** 2026-04-30
**Status:** Proposed
**Depends on:** Slice D closed and verified (SYSTEM_INVENTORY.md 2026-04-30)

---

## Purpose

Group E stands up the local service substrate — Redis and SearXNG — before any slice that consumes cache or search. If the infrastructure boundary is defined after the consumers are built, each consumer draws its own connection logic and the substrate becomes implicit. Group E prevents that by making Redis and SearXNG first-class, fail-closed, locally-owned services before Group F tools or Group G memory retrieval use them.

This is infrastructure work, not application work. The backend continues to run without these services present. Subsystems that need them report unavailable and degrade cleanly.

## Pre-Implementation Gates

**Gate E-G1 — Docker available on both host classes.**
Confirm Docker Desktop (or equivalent) is installed and `docker compose` command is available on Windows x64 and Windows ARM64. If Docker is absent on either host, E.1 closes with a `SKIP-no-docker` state for that host; backend fail-closed behavior covers that case. Document host-specific Docker status in CHANGE_LOG before E.2 begins.

**Gate E-G2 — SearXNG engine configuration minimum.**
SearXNG requires a `settings.yml` config file to enable at least one search engine. Confirm the minimum config is documented in `config/cache/searxng/settings.yml` before E.3 validation. Without at least one engine enabled, SearXNG returns no results even when healthy.

**Gate E-G3 — `httpx` and `redis-py` availability.**
`httpx` is already in base dependencies (used by LLM runtimes). `redis-py` (`redis>=5.0`) is not yet declared. E.2 requires it. Confirm whether it can be added to base `[project].dependencies` or whether it needs a new extra. Recommendation: base, since Redis is used as coordination substrate that any provisioned host may need. This requires a `pyproject.toml` edit and re-provisioning; resolve before E.2 begins.

## Sub-Slice E.1 — Redis + SearXNG via docker-compose

### Why Here

`docker-compose.yml` already declares both services. E.1 validates that the declared stack runs, both services reach healthy state, and they are reachable from the backend host. It also adds the missing service-discovery config files that `slices.md` requires.

### Goal

`docker compose up` starts Redis and SearXNG; both report healthy; `redis-cli ping` returns `PONG`; SearXNG `/healthz` returns 200.

### Scope

- **`docker-compose.yml`** — already present. Extend only if a concrete gap is found (e.g., missing SearXNG config volume, missing network, wrong port binding). Do not rewrite working config.
- **`config/cache/redis.yaml`** — new. Documents Redis connection defaults consumed by the E.2 cache manager:
  ```yaml
  host: 127.0.0.1
  port: 6379
  db: 0
  max_connections: 10
  socket_timeout: 2.0
  socket_connect_timeout: 2.0
  ```
- **`config/cache/searxng/settings.yml`** — new. Minimum SearXNG config enabling at least one search engine (e.g., DuckDuckGo HTML or Brave). Required for SearXNG to return any results. Must use a format compatible with the `searxng/searxng:latest` image.
- **`config/models/search.yaml`** — new. Documents search provider defaults consumed by the E.3 runtime:
  ```yaml
  primary: searxng
  fallback: ddgs
  tertiary: tavily
  searxng_base_url: http://127.0.0.1:8080
  ddgs_timeout_s: 10
  tavily_api_key_env: TAVILY_API_KEY
  max_results: 5
  ```
- **Operator prerequisite**: Docker Desktop or equivalent documented in `config/hardware/notes.md`. `TAVILY_API_KEY` documented in `.env.example` as optional (Tavily is tertiary / policy-gated).
- **Fail-closed**: the backend must run correctly when neither service is present. No application code changes are required to enforce this — it is a property of E.2/E.3 implementations which do not yet exist. E.1 only validates the compose stack.

### Files Not to Modify

- `backend/app/**` — no backend wiring in E.1
- `pyproject.toml` — no dependency changes in E.1

### Validation Commands

```powershell
docker compose up -d
docker compose ps
docker exec jarvisv7-redis redis-cli ping
```

```powershell
backend\.venv\Scripts\python scripts\validate_backend.py regression
```

### Acceptance

```
PASS docker compose up -d: both services start without error
PASS docker compose ps: redis health=healthy, searxng health=healthy
PASS redis-cli ping: PONG
PASS GET http://127.0.0.1:8080/healthz: 200
PASS regression: validate_backend.py regression green on both host classes (count ≥ 95)
SKIP-no-docker: acceptable per-host closeout state if Docker is absent; document in CHANGE_LOG
```

### Finish Line

Both services are confirmed reachable and healthy. Config files are in place for E.2 and E.3. The compose stack can be started and stopped cleanly without affecting the backend regression suite.

### Risks

- SearXNG image pulls require internet access; cached images work offline.
- `searxng/searxng:latest` image tag may change behavior between pulls. If a specific image digest is required for reproducibility, pin it.
- ARM64: SearXNG image publishes multi-arch; Redis 7-alpine publishes multi-arch. Both should work natively on ARM64 but confirm before closing E.1 on that host.

## Sub-Slice E.2 — Cache Layer Wired to Redis

### Why Here

Once Redis is reachable (E.1), the cache layer can be wired. Defining it here — before any consumer (G.3 retrieval caching, F tools, etc.) — ensures a single cache authority exists rather than each consumer inventing its own Redis connection.

### Goal

The cache manager is callable and degrades explicitly (returns cache-miss, not exception) when Redis is unavailable.

### Scope

**`backend/app/cache/__init__.py`** — empty.

**`backend/app/cache/redis_client.py`** — `RedisClient`:
- `get_client() -> redis.Redis | None`: returns a connected client using `config/cache/redis.yaml` values, or `None` with a logged warning if Redis is unreachable. Never raises to caller.
- Connection is lazy (created on first call); module-level singleton pattern.

**`backend/app/cache/keys.py`** — cache key namespacing:
- `make_key(namespace: str, *parts: str) -> str`: joins namespace and parts with `:` separator.
- Canonical namespaces documented as module-level constants: `NS_RETRIEVAL = "retrieval"`, `NS_SESSION = "session"`, `NS_SEARCH = "search"`.

**`backend/app/cache/policies.py`** — `CachePolicy` dataclass:
- `ttl_seconds: int` — per-namespace TTL.
- `DEFAULT_RETRIEVAL_TTL = 300`, `DEFAULT_SESSION_TTL = 3600`, `DEFAULT_SEARCH_TTL = 60`.
- Policies are read from `config/cache/redis.yaml` if present; defaults used when absent.

**`backend/app/cache/manager.py`** — `CacheManager`:
- `get(key: str) -> str | None`: returns cached value or `None` (cache-miss or Redis down).
- `set(key: str, value: str, ttl: int) -> bool`: stores value; returns `False` without raising if Redis down.
- `delete(key: str) -> bool`: deletes key; returns `False` without raising if Redis down.
- `is_available() -> bool`: returns True only if the Redis client is connected and responsive.
- All methods fail-closed: `None`/`False` on any Redis error, never raises.
- `CacheManager` is injected into FastAPI app state in `run_backend.py` startup; exposed via `api/dependencies.py`.

**`pyproject.toml`** — add `redis>=5.0` to base `[project].dependencies` (after E-G3 decision).

**`backend/tests/unit/cache/__init__.py`** — empty.

**`backend/tests/unit/cache/test_cache_manager.py`**:
- `test_get_returns_none_when_redis_unavailable` (mock client)
- `test_set_returns_false_when_redis_unavailable` (mock client)
- `test_get_returns_value_when_redis_available` (mock client)
- `test_set_stores_value_with_ttl` (mock client)
- `test_is_available_false_when_client_none`
- `test_make_key_joins_parts_with_colon`
- `test_cache_policy_defaults_used_when_config_absent`

**`backend/tests/runtime/services/__init__.py`** — empty.

**`backend/tests/runtime/services/test_cache_live.py`** — `@pytest.mark.live @pytest.mark.requires_docker`, `skipif SKIP_UNLESS_LIVE`:
- `test_cache_roundtrip_against_running_redis`
- `test_cache_returns_none_when_key_missing`
- `test_cache_is_available_true_when_redis_running`
- `test_cache_is_available_false_when_redis_stopped` (requires stopping Redis during test; skip if stop not possible in test context)

### Files Not to Modify

- `backend/app/api/routes/**` — cache manager is injected via dependencies; routes do not change in E.2
- Any Slice A/B/C/D implementation files
- `docker-compose.yml` — E.1 owns this

### Validation Commands

```powershell

backend\.venv\Scripts\python -m pytest backend\tests\unit\cache -q
backend\.venv\Scripts\python scripts\validate_backend.py unit

```

(Live test requires Docker running; run separately with `JARVISV7_LIVE_TESTS=1`.)

### Acceptance


### Finish Line

`CacheManager` is injectable, fail-closed, and proven against a live Redis on at least one host class. Every future consumer (G.3, F tools) calls it; none open their own Redis connections.

### Risks

- `redis-py` `redis.Redis` connection-pool behavior differs from `redis.StrictRedis`; use `redis.Redis` directly with `decode_responses=True` and confirm on ARM64.
- Lazy singleton pattern means the first caller pays the connection cost. If this creates startup latency, initialize eagerly in `run_backend.py` startup sequence.

## Sub-Slice E.3 — Internet Search Runtime

### Why Here

SearXNG is running (E.1) and the cache layer exists (E.2). The search runtime wires to both: it queries SearXNG (or its fallbacks), and results may be cached via the E.2 cache manager in G.3. Defining the search runtime boundary here — before Group F tools — ensures the tool adapter in F.2 has one search interface to consume rather than inventing its own.

### Goal

A known search query returns results via SearXNG when healthy; falls back to DDGS when SearXNG is stopped; Tavily path is present but policy-gated.

### Scope

**`backend/app/runtimes/internetsearch/__init__.py`** — empty.

**`backend/app/runtimes/internetsearch/base.py`** — `SearchBase` ABC:
- `search(query: str, max_results: int = 5) -> list[SearchResult]`
- `is_available(self) -> bool`
- `SearchResult` dataclass: `title: str`, `url: str`, `snippet: str`, `source: str`

**`backend/app/runtimes/internetsearch/searxng_runtime.py`** — `SearXNGRuntime`:
- Uses `httpx` to call `GET http://127.0.0.1:8080/search?q=...&format=json` (base URL from `config/models/search.yaml`).
- `is_available()`: `GET /healthz`; returns `False` with logged reason if unreachable.
- Returns `[]` on error without raising.

**`backend/app/runtimes/internetsearch/ddgs_runtime.py`** — `DDGSRuntime`:
- Uses `duckduckgo_search` package (`pip install duckduckgo-search`). Requires addition to base or a new extra — see gate E-G3 note below.
- `is_available()`: always returns `True` (network-dependent; fail-closed on exception).
- Falls back on SearXNG outage; not a primary path.

**`backend/app/runtimes/internetsearch/tavily_runtime.py`** — `TavilyRuntime`:
- Uses `httpx` to call Tavily API. Requires `TAVILY_API_KEY` env var.
- `is_available()`: returns `False` if `TAVILY_API_KEY` not set or policy `cloud_enabled: false`.
- Policy-gated via `config/app/policies.yaml`; never primary.

**Search selector** — `select_search_runtime(policy, cache_manager) -> SearchBase`:
- Lives in `backend/app/routing/runtime_selector.py` as `select_search_runtime()` (extends existing selector file).
- Prefers SearXNG → DDGS → Tavily per `config/models/search.yaml` and policy.
- Returns a `NullSearchRuntime` that returns `[]` and logs a reason if all providers fail availability checks.

**`config/models/search.yaml`** — populated in E.1; consumed here.

**`backend/tests/unit/runtimes/internetsearch/__init__.py`** — empty.

**`backend/tests/unit/runtimes/internetsearch/test_search_runtime.py`**:
- `test_searxng_is_available_false_when_unreachable` (mock httpx)
- `test_searxng_returns_results_on_valid_response` (mock httpx)
- `test_ddgs_returns_empty_list_on_exception_without_raising`
- `test_tavily_is_available_false_when_no_api_key`
- `test_tavily_is_available_false_when_policy_gated`
- `test_selector_prefers_searxng_when_available`
- `test_selector_falls_back_to_ddgs_when_searxng_down`
- `test_null_runtime_returns_empty_list_without_raising`

**`backend/tests/runtime/services/test_search_live.py`** — `@pytest.mark.live @pytest.mark.requires_docker`, `skipif SKIP_UNLESS_LIVE`:
- `test_searxng_returns_results_for_known_query`
- `test_ddgs_returns_results_for_known_query`
- `test_selector_falls_back_to_ddgs_with_searxng_stopped`

### DDGS dependency note

`duckduckgo-search` is not currently in `pyproject.toml`. It is a lightweight package with no ML dependencies. Add to base `[project].dependencies` alongside `redis>=5.0` (both resolved at the E-G3 gate). If adding to base is rejected, a `search` extra is an alternative — but base is preferred since search fallback is part of normal degraded operation.

### Files Not to Modify

- `backend/app/runtimes/llm/**`, `backend/app/runtimes/stt/**`, `backend/app/runtimes/tts/**`, `backend/app/runtimes/wake/**` — no changes to existing runtime families
- `docker-compose.yml` — E.1 owns this

### Validation Commands

```powershell
backend\.venv\Scripts\python -m compileall backend\app\runtimes\internetsearch backend\app\routing\runtime_selector.py
backend\.venv\Scripts\python -m pytest backend\tests\unit\runtimes\internetsearch -q
backend\.venv\Scripts\python scripts\validate_backend.py unit
backend\.venv\Scripts\python scripts\validate_backend.py regression
```

(Live test requires Docker + SearXNG running; run separately with `JARVISV7_LIVE_TESTS=1`.)

### Acceptance

```
PASS compile: runtimes/internetsearch, runtime_selector.py extension
PASS unit: search runtime unit suite — N passed on both host classes
PASS live searxng: known query returns results via SearXNG (JARVISV7_LIVE_TESTS=1, Docker running) OR SKIP-no-docker
PASS live fallback: DDGS returns results when SearXNG stopped OR SKIP-no-docker
PASS regression: validate_backend.py regression green on both host classes (count ≥ 95)
```

### Finish Line

One search interface exists. F.2 tool adapter consumes it without knowing which provider is active. G.3 retrieval caching can optionally accelerate it via the E.2 cache manager. Tavily is present and policy-gated for when local search is insufficient.

### Risks

- SearXNG JSON search API requires `format=json` parameter; the default HTML format is not machine-readable. Confirm the installed image supports JSON format.
- DDGS rate-limiting: `duckduckgo-search` may be rate-limited or blocked by DuckDuckGo. Document this as a known limitation; it is a fallback, not a primary path.
- SearXNG `settings.yml` engine configuration: if no engines are enabled or all engines are blocked by rate-limits, SearXNG returns an empty result set rather than an error. The search runtime should treat an empty result set as a soft failure and log the condition.

## Slice-Level Closeout Criteria

Slice E is complete when all of the following are true:

1. E-G1 through E-G3 gates resolved before implementation begins.
2. `docker compose up` starts both services; both reach healthy state on at least one host class (or SKIP-no-docker documented per host).
3. `config/cache/redis.yaml`, `config/cache/searxng/settings.yml`, and `config/models/search.yaml` exist and are non-empty.
4. `CacheManager` is injectable, fail-closed (returns `None`/`False` not exceptions), and proven against live Redis.
5. `SearXNGRuntime`, `DDGSRuntime`, and `TavilyRuntime` are importable, unit-tested, and fail-closed.
6. Search selector returns `SearXNGRuntime` when healthy; falls back to `DDGSRuntime` when SearXNG down.
7. All unit suites green on both host classes.
8. `validate_backend.py regression` green on both host classes (count ≥ 95).
9. `CHANGE_LOG.md` has one entry per sub-slice E.1–E.3 with host class(es).
10. No Group F/G/H/I file introduced at Slice E closeout.
11. Slice A+B+C+D regression count has not decreased.
